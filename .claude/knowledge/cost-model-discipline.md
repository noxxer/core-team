# Cost-aware Model Discipline

Дисциплина выбора модели (Opus / Sonnet / Haiku) и параллелизма subagent-ов.
Цель — не платить за Opus там, где Sonnet справится, и не получать Sonnet-результат там, где нужен Opus.

## Принцип

> Default — Sonnet. Opus — только при явном обосновании. Haiku — для quick-pass.

При конфликте оптимизации и надёжности **побеждает надёжность**: security/data-migration/strategy всегда получают более умную модель + отдельный review.

## Таблица моделей

| Модель | Когда использовать | Не использовать |
|--------|-------------------|-----------------|
| **Opus** | Архитектура (architect), стратегия (CTO), планирование сложных пайплайнов, security audit с высокими ставками, FPF NQD на критическом решении | Реализация по готовому ARCH, рутинный code review, заполнение шаблонов, форматирование |
| **Sonnet** | Default. Имплементация (dev), code review (test), facilitator-оркестрация, designer/keeper/customer, обычное планирование | Тривиальные задачи, где Haiku справится |
| **Haiku** | Quick-pass проверки, batch-обработка типовых файлов, lookup в knowledge/, генерация шаблонов из формы | Архитектурные решения, security review, креативные задачи |

## Параллелизм: ≤7 subagent-ов в фазе

Опираемся на LIFT-COT capacity-bounds (когнитивный лимит координации):

| Фаза | Макс. параллельных subagent-ов | Источник лимита |
|------|-------------------------------|-----------------|
| Planning (3-стадийный вопрос, NQD) | **6** | Когнитивный лимит на удержание альтернатив |
| Validation (ревью, тестирование) | **7** | Magic number 7±2 для перекрёстной проверки |
| Integration (имплементация) | **4** | Лимит на согласование интерфейсов между потоками |

Если задача требует больше — режь на последовательные фазы. Не запускай 12 архитекторов параллельно "на всякий случай".

## Pilot-then-batch (при ≥3 параллельных одного типа)

Когда в фазе планируется **≥3 параллельных subagent-а одного типа** (например, 4 architect для разных модулей или 5 evidence-check на разных артефактах) — обязательна **пилот-фаза**:

1. **Phase N.1 — Pilot:** запустить **1 subagent-а** с полным контекстом
2. **Калибровка** — facilitator/orchestrator анализирует output пилота, корректирует промпт остальных
3. **Phase N.2 — Batch:** запустить **N-1** оставшихся параллельно после калибровки

**Reason:** параллельные runs одного типа имеют общий риск (плохой промпт, недосказанная конвенция). Пилот — дешёвый detector. Без пилота 4-5 неудачных runs стоят дороже чем 1 пилот + 4-5 успешных. Проверенное правило на практике.

**Когда пилот НЕ нужен:**
- 2 параллельных subagent-а одного типа (overhead не оправдан)
- 3+ subagent-ов **разных типов** в одной фазе (общего промпт-риска нет)

Автодетекция и автоматическое разделение фазы — в `skills/planner/SKILL.md` (execution mode).

## Когда повышать модель (escalate)

Признаки, что нужен Opus вместо Sonnet:
1. **Высокая цена ошибки** — данные миграции, security, financial logic, архитектура с долгим жизненным циклом
2. **Неструктурированная задача** — нет чёткого ARCH/чеклиста, нужно проектировать решение
3. **Множественные альтернативы с trade-offs** — NQD требует ≥3 альтернатив с обоснованием
4. **Cross-domain reasoning** — нужно связать стэк/бизнес/legal/UX в одном решении
5. **Семантическая сложность** — bounded contexts с разными invariants, FPF A.1.1

## Когда понижать модель (downgrade)

Признаки, что Haiku справится с тем, что обычно делает Sonnet:
1. **Жёстко структурированная задача** — заполнение готового шаблона, форматирование
2. **Lookup в knowledge** — найти секцию, скопировать нужный фрагмент
3. **Batch-обработка** — одинаковая операция над N файлами
4. **Форматные правки** — переименование, перенос строк, индентация

## Орчестрация: правило стоимости

Перед запуском пайплайна с дорогими моделями (Opus в нескольких subagent-ах) — оценить:
- Tokens estimate (по объёму input)
- Wall-clock estimate (по числу фаз)
- Relative $-cost vs naive single-agent

Если cost > 5x от naive solution без явной выгоды — пересмотри плана. Может, достаточно одного Sonnet-subagent с расширенным контекстом.

## Explicit model selection (обязательно)

Модель **каждого** dispatch subagent-а указывается **явно** — через frontmatter роли или параметр вызова. Запрещено молча наследовать модель родителя: унаследованный Opus в реализации/ревью даёт 3-5x перерасход без выгоды (повторяющийся перерасход). Перед `Agent(...)` сверься с таблицей default-моделей выше; отклонение от default — обоснуй одной строкой.

## Anti-patterns

- **Молчаливое наследование Opus** — subagent без явной модели наследует родителя → кратный перерасход
- **Opus "на всякий случай"** — каждый Opus-выбор должен быть обоснован
- **Все subagent-ы на Opus** — кратно дорого, редко оправдано
- **Один Opus-architect делает 12 фич сразу** — режь на последовательные фазы
- **Haiku на security review** — никогда; security всегда минимум sonnet, в идеале opus
- **Параллельный запуск без оценки cost** — кратные счета без выгоды
