# Core Protocols — Протоколы принятия решений

> Основано на McCarthy Core Protocols, Holacracy 5.0, Six Thinking Hats (de Bono).

---

## Протокол вопросов (3 стадии) — ОБЯЗАТЕЛЬНЫЙ

**Цель:** предотвратить потерю tensions и сократить обращения к основателю.

### Процесс

1. **TaskCreate** — ЛЮБАЯ tension, вопрос, проблема → сначала зафиксировать как задачу. Без фиксации — обсуждение запрещено
2. **Роли обсуждают** — Facilitator маршрутизирует tension к ролям через commands или Agent Teams. Роли анализируют, предлагают варианты
3. **Только потом к Founder** — если роли не могут разрешить tension самостоятельно → `AskUserQuestion` с вариантами и описанием последствий

**Нарушение:** задавать вопрос Founder без стадий 1-2 = процессная ошибка.

### AskUserQuestion — формат с последствиями (обязательно)

Каждая опция в `AskUserQuestion` **обязана** содержать:

1. **Краткое название** (1-3 слова)
2. **Что произойдёт при выборе** — последствие в терминах продукта / пользователя / системы
3. **Что НЕ произойдёт** — явное исключение, чтобы Founder понял scope

**Антипаттерны:**
- Open-ended «Что думаешь?» / «Как тебе?»
- Варианты без последствий (просто названия)
- Опции без явных trade-off-ов между ними

**Группировать 2-4 вопроса за один вызов** AskUserQuestion. Не перегружать Founder одиночными вопросами.

**Reason:** Founder должен видеть **информированный выбор**, а не угадывать что выйдет из каждой опции. Это compact-safe (вопросы переживают сжатие контекста) и сокращает повторные вопросы.

## Inventory-and-Proposal Protocol

**Цель:** предотвратить дорогие ошибки в крупных или необратимых операциях.

### Когда применять (триггеры)

Subagent **обязан** показать inventory-and-proposal до исполнения, если выполняется хотя бы одно:

- Работа с **внешней системой** (загрузка/выгрузка, API mass-write, миграция данных)
- **Batch ≥10 объектов** (массовое создание/обновление/удаление артефактов или записей)
- **Необратимая операция** (удаление, миграция схемы БД, публикация контента, изменение production)
- **Долгая локальная обработка** (≥10 минут на CPU/IO — транскрипции, конвертации, build)
- Работа с **более чем одной bounded context** одновременно

### Процесс

1. **Inventory** — что обнаружено: структура входа, найденные сущности, их количество и типы
2. **Proposal** — таблица «что → куда → обоснование» (или эквивалентный формат для домена)
3. **Спорные / неоднозначные пункты** — явный отдельный блок
4. **Открытые вопросы** для Founder через `AskUserQuestion` (с последствиями)
5. **Ожидание явного «ок»** от Founder — только после этого исполнение

### Что это НЕ значит

- НЕ требуется для S-операций (≤9 объектов, обратимые, локальные)
- НЕ требуется внутри уже одобренного pipeline (например, на каждой итерации loop-а)
- НЕ заменяет TaskCreate / AskUserQuestion — это **дополнение** на стадии "до делать"

## Express-Parallel Protocol (фоновые агенты без коллизий)

**Цель:** распараллелить работу нескольких subagent-ов без конфликтов записи.

**Ключевое правило:** фоновые агенты запускаются только в **непересекающихся зонах владения файлами на уровне файловой системы** (код vs артефакты vs спецификация). Зоны disjoint → нет гонок записи by design (FPF A.1.1 на уровне файлов).

### Процесс
1. Facilitator/orchestrator декомпозирует задачу на потоки и назначает каждому **непересекающийся набор путей** (например: поток A пишет `src/`, поток B — `project/artifacts/`, поток C — `docs/`)
2. Проверка disjoint: пересечение путей пусто. Есть пересечение → не параллелить, резать иначе
3. Запуск ≤7 агентов (см. `cost-model-discipline.md`)
4. После каждого артефакт-производящего агента — `ls` + `git status` для верификации, что файлы реально записаны туда, куда заявлено (не доверять отчёту агента о записи)

**Reason:** на практике даёт 0 коллизий и кратно меньше wall-clock против последовательной работы. Без disjoint-зон параллельные агенты затирают записи друг друга.

## Detect → Fix → Guard → Document (конституционный)

**Цель:** не допустить повторения класса багов. «Правило без гарда — это просто мнение.»

Каждый обнаруженный **класс** ошибки (не единичный баг) проходит 4 шага:

1. **Detect** — зафиксировать класс: что именно сломалось и почему (корневая причина, не симптом)
2. **Fix** — починить конкретный случай
3. **Guard** — добавить **machine-verifiable инвариант-тест**, который падает на этом классе. Плюс sanity-check на синтетической регрессии (тест действительно ловит проблему)
4. **Document** — записать класс + гард в ADR / context.md роли, чтобы знание пережило сессию

**Триггер:** баг, который «уже встречался» или может встретиться снова. Единичная опечатка — не требует гарда.

**Reason:** проверено на практике — предотвращает повторение класса багов. Без гарда исправленный баг возвращается в следующей сессии.

## Clarity Protocol (перед решением)

**Цель:** убедиться, что все понимают одно и то же, прежде чем решать.

### Процесс
1. Формулировка: что именно решаем? (одно предложение)
2. Проверка 8 точек:
   - [ ] Путь пользователя описан конкретно
   - [ ] Допущения названы явно
   - [ ] Ресурсы оценены
   - [ ] Сроки определены
   - [ ] Границы scope зафиксированы
   - [ ] Legal/compliance учтено
   - [ ] Нет "серых зон" — неясных ответственностей
   - [ ] Критерии успеха измеримы
3. Если хоть одна точка не закрыта → уточнить ДО перехода к Decider

### Red Flags (немедленная остановка)
- "разберёмся потом"
- скрытые допущения ("это же понятно")
- отсутствие дедлайнов
- абстрактный scope ("сделать хорошо")
- "это же очевидно" (ложный консенсус)

## Decider Protocol (голосование)

**Цель:** быстрое и явное принятие решения.

### Процесс
1. **Clarity пройдена** (обязательно)
2. **Предложение** — формулируется одним предложением
3. **Голос основателя:**
   - **Да** — принимаю
   - **Нет** — не принимаю + обязательная альтернатива
   - **Поддержка-с-условием** — принимаю если [условие]
4. **Резолюция** — если "Нет":
   - Возражающий предлагает альтернативу
   - Clarity Protocol для альтернативы
   - Повторное голосование
5. **Запись:** `project/decisions/DEC-NNN_title.md`

### Формат решения (ADR)
```markdown
# DEC-NNN: Название решения

**Дата:** YYYY-MM-DD
**Статус:** Proposed | Accepted | Deprecated
**Контекст:** [Почему возник вопрос]

## Варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| A | ... | ... |
| B | ... | ... |

## Решение
[Что выбрали и почему]

## Последствия
[Что изменится, на что повлияет]
```

## Perfection Game (обратная связь)

**Цель:** конструктивная критика без негатива.

### Процесс
1. Оценка от 1 до 10 (текущее состояние)
2. "Что добавить, чтобы стало 10?" (конкретные предложения)

### Правила
- Нельзя ставить оценку без предложений по улучшению
- Предложения должны быть конкретными и выполнимыми
- Автор не обязан принимать все предложения

## Check-In (начало сессии)

**Цель:** синхронизация состояния перед работой.

### Процесс (1 минута)
1. Текущее состояние (одно предложение)
2. Фокус сессии (что хотим достичь)
3. Ограничения (время, ресурсы, блокеры)

## Pass (право пропуска)

**Цель:** не блокировать процесс, если нечего добавить.

### Правила
- Любой участник может сказать "Pass" в любой момент
- Pass не требует обоснования
- Pass не означает несогласие — только отсутствие input

## Six Thinking Hats (режимы мышления)

**Цель:** структурированный brainstorm без хаоса.

| Шляпа | Фокус | Роль в команде |
|-------|-------|----------------|
| Белая | Факты, данные | Analyst |
| Красная | Интуиция, чувства | Customer |
| Чёрная | Критика, риски | Guardian |
| Жёлтая | Оптимизм, возможности | Product / Growth |
| Зелёная | Креатив, альтернативы | Designer / Brand |
| Синяя | Процесс, управление | Facilitator |

### Применение
1. Facilitator объявляет шляпу
2. Все думают в режиме этой шляпы (5 минут)
3. Следующая шляпа
4. Синтез результатов

## Tension Processing (маршрутизация)

**Цель:** каждое "напряжение" находит адресата.

### Процесс
1. Кто-то ощущает tension (проблему, идею, вопрос)
2. Facilitator определяет домен → маршрутизирует к роли
3. Роль анализирует и возвращает рекомендацию
4. Если cross-domain → обе роли высказываются
5. Facilitator ведёт к решению через Decider

### Типы tensions
- **Операционная** — что-то не работает → к владельцу домена
- **Стратегическая** — куда двигаться → Product + Analyst
- **Рисковая** — что может пойти не так → Guardian
- **Ресурсная** — чего не хватает → CFO / CTO
