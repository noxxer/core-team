# DPF: architect

> Предметный учебник для AI-агента (Domain Principles Framework). Идёт **после FPF**:
> FPF даёт общие принципы работы, DPF — SoTA-ходы, термины, ошибки и границы этого домена.
> Паттерны — декларативный метод мышления, **не** инструкция на выполнение (E.8).

## 0. Scope & Bounded Context (FPF A.1.1)

- **В фокусе:** Ремесло software architect: определение bounded contexts, проектирование контрактов между ними, ADR как механизм фиксации решений, точки расширения, trade-off анализ, handoff «дизайн → разработка»
- **Вне фокуса:** Реализация кода (это dev), code review (это test), DevOps/инфраструктура, UI/UX дизайн, product management, бизнес-стратегия (это cto)
- **Перевод терминов:** «bounded context» = «ограниченный контекст» (термин DDD), в проекте может называться «домен» — но это разные вещи; «handoff» здесь = артефакт ARCH-NN.md для dev
- **Maturity:** draft — 11 каналов Collection Protocol пройдены; провенанс-пробелы закрыты (P-06 [vendor-claim], P-08 заменён на AWS primary, P-12 URL добавлен); добавлен P-13 Saga. Для refined нужна верификация паттернов на реальных ревью ARCH-NN.md файлов проекта

| Ограниченный контекст | Bounded Context | Явная граница, внутри которой модель предметной области и язык однозначны и согласованы (Evans, 2003) | Микросервис, модуль кода, таблица БД |
| Вездесущий язык | Ubiquitous Language | Общий язык команды и доменных экспертов, закреплённый в коде, документах и разговорах внутри одного bounded context | Жаргон, словарь проекта в целом |

| Контракт | Contract / API Contract | Явный, версионированный интерфейс между двумя системами или контекстами: формат данных, семантика, гарантии | Внутренняя реализация, приватный интерфейс |

| Точка расширения | Extension Point | Явно спроектированное место, где система может быть расширена без изменения ядра | Случайное место для hack, глобальная переменная |
| Антикоррупционный слой | Anti-Corruption Layer (ACL) | Транслирующий слой, изолирующий внутреннюю модель контекста от чужого языка внешней системы | Адаптер без трансляции семантики, HTTP-прокси |

| Передача работы | Handoff | Артефакт (ARCH-NN.md), передающий design-решения от architect к dev с достаточным контекстом для реализации без потерь | Устное объяснение, ссылка на ADR |

## 2. Паттерны (FPF E.8)

> Каждый паттерн — повторяющаяся проблемная ситуация + проверенный ход. Минимум полей ниже обязателен.

### P-01: Определение границ bounded context через язык команды

- **Проблемная ситуация:** Домен большой, несколько команд или departmentов используют одни и те же слова («клиент», «заказ», «продукт») в разных смыслах — возникает семантический шум, модель разбухает
- **Силы:** Единая «God model» хрупка и масштабируется плохо; но дробить без смысла — создаёт лишние переводы и overhead
- **Ход (решение):** Граница bounded context совпадает с границей ubiquitous language: там, где один и тот же термин меняет смысл — граница. Вторичные сигналы: разные команды, разные бизнес-правила, разные технические представления. Зафиксировать границы в context map
- **Последствия:** Каждый контекст получает чёткую зону ответственности, модели внутри них стабильны; между контекстами появляются явные интеграционные точки
- **Trade-offs:** Больше границ = больше контрактов и переводов; меньше границ = больший риск семантического загрязнения
- **Пример:** В e-commerce «Customer» в Order Management хранит адреса доставки; в CRM «Customer» хранит историю контактов. Одно слово, две модели, два контекста — правильное решение
- **Пограничные случаи:** Когда граница слишком тонкая (контексты меняются вместе при каждом изменении) — пора объединять. Когда команды неоднородны — граница ≠ Team Topologies boundary, нужна явная договорённость
- **Антипаттерны:** Создавать контекст под каждый микросервис без анализа языка; делать «Universal Customer» на всю систему; смешивать bounded context с модулями кода
- **Связи:** стыкуется с [P-02] (контракты), [P-03] (ubiquitous language), [P-10] (team alignment)
- **Когда НЕ применять:** Маленькие системы с одной командой и однозначным языком — overhead от разбивки не окупится
- **SoTA-источник:** Fowler, M. BoundedContext (martinfowler.com, 2014); Evans E. DDD (2003); Vernon V. Implementing DDD (2013)
- **Проверка качества (E.21):** В каждом bounded context есть glossary с определениями терминов; один и тот же термин не имеет двух разных определений внутри одного контекста

### P-02: Контракт между контекстами — явный, версионированный, минимальный

- **Проблемная ситуация:** Два bounded context'а обмениваются данными; один из них меняется — другой ломается. Либо команды боятся менять что-либо («всё завязано»)
- **Силы:** Любая публичная зависимость ограничивает свободу эволюции; implicit API = source of truth по факту, но не по договорённости
- **Ход (решение):** Каждый контракт: (a) явный (OpenAPI / Protobuf / AsyncAPI / event schema), (b) минимальный (только то, что потребителю нужно сейчас), (c) версионированный. Upstream публикует Open-Host Service с Published Language; если нельзя — downstream строит ACL. Контракт версионируется семантически (major/minor)
- **Последствия:** Команды могут менять внутреннюю реализацию без страха сломать потребителей; breaking changes явны и договорены
- **Trade-offs:** Поддержка нескольких версий контракта требует ресурсов; schema-first подход увеличивает latency на начальном этапе
- **Пример:** Netflix перешёл на Federated GraphQL Gateway, где каждый Domain Graph Service экспонирует свой граф через стабильный контракт; внутренности меняются свободно (Netflix Engineering Blog, 2023)
- **Пограничные случаи:** Event-driven интеграция: контракт = event schema; нужен event schema registry (Confluent, AWS Glue). При synchronous вызовах — consumer-driven contract testing (Pact)
- **Антипаттерны:** Shared database между контекстами (God Schema); передача внутренних entity напрямую через границу; неявный контракт «мы договорились на словах»
- **Связи:** стыкуется с [P-01] (границы), [P-07] (ACL), [P-08] (API-first); противоречит «shared database» подходу
- **Когда НЕ применять:** Внутри одного bounded context — там контракты избыточны, лучше прямые вызовы
- **SoTA-источник:** Software Architecture Guild. Integration of Bounded Contexts (2024); Richards M. Software Architecture: The Hard Parts (2022), ch.4
- **Проверка качества (E.21):** Есть machine-readable schema для каждой интеграционной точки; consumer-driven contract tests проходят в CI

### P-03: Ubiquitous Language как дисциплина проектирования

- **Проблемная ситуация:** Команда и доменные эксперты говорят по-разному об одном и том же; код называет вещи иначе, чем бизнес; новый разработчик тратит дни на разбор «что значит status=3»
- **Силы:** Неформальный язык постепенно дрейфует; без явной дисциплины код и домен расходятся за месяцы
- **Ход (решение):** Завести и поддерживать glossary.md в каждом bounded context. Все: модели данных, endpoint names, event names, UI labels, тесты — используют термины из glossary. Architect модерирует добавление новых терминов и выявляет конфликты
- **Последствия:** Снижается когнитивная нагрузка; новые участники быстрее входят в контекст; обнаруживаются семантические противоречия на этапе дизайна, не после релиза
- **Trade-offs:** Требует явных усилий на поддержание; может вызывать споры при именовании
- **Пример:** В проекте Core Team — `glossary.md` в каждом bounded context проекта (архитектурное решение самого фреймворка)
- **Пограничные случаи:** Когда две команды не могут договориться о термине — это сигнал о границе bounded context, а не проблема коммуникации
- **Антипаттерны:** DTO с именами типа `UserData`, `Info`, `Payload`; status codes вместо named states; glossary существует но не обновляется
- **Связи:** стыкуется с [P-01] (границы через язык), [P-12] (handoff — язык должен быть в ARCH)
- **Когда НЕ применять:** Прототипы и throwaway-код — там overhead не оправдан
- **SoTA-источник:** Evans E. DDD (2003), ch.2; Fowler M. UbiquitousLanguage (martinfowler.com, 2006)
- **Проверка качества (E.21):** Glossary существует и обновлялся в последние 30 дней; в коде нет терминов, отсутствующих в glossary (можно проверить grep-ом)

### P-04: Architecture Decision Record (ADR) — фиксация значимых решений

- **Проблемная ситуация:** Команда принимает важные архитектурные решения, но через 3 месяца никто не помнит, почему было выбрано именно это решение, а не другое; новые участники повторно открывают закрытые вопросы
- **Силы:** Решения принимаются в условиях неопределённости; контекст, который был у команды, теряется; документация быстро устаревает если нет структуры
- **Ход (решение):** Каждое значимое решение (влияет на несколько команд, сложно откатить, касается архитектурных характеристик) фиксируется в ADR-файле: Context → Decision → Consequences → Alternatives considered. ADR immutable после принятия; superseded ADR создаётся новым файлом со ссылкой на предыдущий. Хранить рядом с кодом в VCS
- **Последствия:** История решений сохраняется; новые участники понимают «почему так»; команда меньше revisits закрытые вопросы
- **Trade-offs:** Требует дисциплины на поддержание; риск «ADR-театра» — формально создаются, но не читаются
- **Пример:** AWS рекомендует ADR как стандартную практику для cloud-архитектур; GitHub (github.com/adr) поддерживает экосистему шаблонов. Microsoft Azure Well-Architected Framework требует ADR для ключевых решений (Microsoft Learn, 2024)
- **Пограничные случаи:** Не каждое техническое решение достойно ADR — только те, что значимы архитектурно (влияют на характеристики системы: масштабируемость, безопасность, maintainability). Оперативные решения — в комментариях PR
- **Антипаттерны:** ADR без Alternatives considered (нарушает NQD); ADR для каждого PR (инфляция); редактирование принятых ADR вместо создания нового; ADR хранится в Confluence, а не в VCS
- **Связи:** стыкуется с [P-11] (trade-off analysis — это содержание ADR); противоречит устным договорённостям
- **Когда НЕ применять:** Прототипы и эксперименты; решения, легко обратимые за часы
- **SoTA-источник:** Nygard M. Documenting Architecture Decisions (cognitect.com, 2011); adr.github.io (2024); AWS Architecture Blog (2024); Microsoft Azure WAF (2024)
- **Проверка качества (E.21):** Для каждого значимого architectural choice последнего квартала существует ADR-файл в VCS; ADR содержит ≥2 рассмотренных альтернативы

### P-05: Точки расширения — явное проектирование для изменений

- **Проблемная ситуация:** Система постоянно меняется: новые провайдеры платежей, новые каналы доставки уведомлений, новые источники данных. Каждое добавление требует правки ядра
- **Силы:** Закрытое ядро = стабильность; открытое ядро = гибкость. Невозможно предугадать все точки изменений заранее
- **Ход (решение):** Architect явно идентифицирует и документирует extension points: абстракции (интерфейсы, plugin contracts), где новая реализация добавляется без изменения ядра. Используется Open-Closed Principle на системном уровне. Точки расширения фиксируются в ARCH как часть handoff
- **Последствия:** Dev может добавить новый платёжный провайдер не трогая routing-логику; architect контролирует где «швы» системы
- **Trade-offs:** Преждевременная абстракция — YAGNI-риск; если extension point не используется — лишняя сложность
- **Пример:** Stripe использует plugin-архитектуру для payment methods: новые методы регистрируются через общий интерфейс без изменения обработчика транзакций
- **Пограничные случаи:** Extension point vs стратегия: первый для plug-in-like добавления, второй — для выбора алгоритма в runtime. Не путать
- **Антипаттерны:** Добавление switch/if-else в ядре для каждого нового типа; «расширяемость» через копипаст; extension point без документации — никто не знает как расширять
- **Связи:** стыкуется с [P-02] (контракт = API extension point), [P-12] (handoff должен включать extension points)
- **Когда НЕ применять:** Единственная реализация, которая точно не изменится; прототипы
- **SoTA-источник:** Martin R.C. Clean Architecture (2018), Open-Closed Principle; Richards M., Ford N. Fundamentals of SA, 2nd ed. (2024), ch.8
- **Проверка качества (E.21):** В ARCH-файле перечислены extension points с описанием контракта расширения; dev может добавить новую реализацию без изменения файлов ядра

### P-06: Strangler Fig — инкрементальная замена legacy без остановки

- **Проблемная ситуация:** Существующая система должна быть заменена или переписана, но её нельзя остановить; «большой bang-перепись» рискованна и исторически проваливается
- **Силы:** Большая перезапись = высокий риск (темпераментная команда ещё не понимает домен до конца, нет знаний о граничных случаях); постепенная замена требует поддержки двух систем параллельно
- **Ход (решение):** Создать facade/gateway поверх старой системы; постепенно перехватывать функциональность, направляя трафик в новые компоненты. Старая система «задыхается» по мере роста новой. Граница перехода — bounded context. Каждый шаг — независимо тестируем и откатываем
- **Последствия:** Бизнес-операции не прерываются; каждый шаг миграции — low-risk; команда изучает домен инкрементально
- **Trade-offs:** Длиннее общая миграция; overhead на поддержку facade и маршрутизации; технический долг «двойного учёта»
- **Пример:** Анализ 28 stalled strangler-fig проектов (2022-2025) показал: проекты проваливаются в первые 90 дней без automated testing и clear ownership (vFunction, 2024) [vendor-claim: данные из маркетингового отчёта вендора без независимой верификации; основной вывод согласуется с Fowler/Fowler, но цифра «28» не верифицирована внешним источником]
- **Пограничные случаи:** Если legacy имеет shared database — нужен Database Strangler (параллельные writes в обе таблицы) до полного переключения. Event-driven Strangler: Kafka как шина между old/new
- **Антипаттерны:** Начать Strangler без определения scope и exit criteria; не тестировать facade на обратную совместимость; удалить legacy до полной верификации нового
- **Связи:** стыкуется с [P-01] (мигрировать по bounded context), [P-02] (facade = контракт), [P-07] (ACL для изоляции от legacy); противоречит «big bang rewrite»
- **Когда НЕ применять:** Маленькие системы (проще переписать за спринт); системы без трафика (нет смысла в facade)
- **SoTA-источник:** Fowler M. StranglerFigApplication (martinfowler.com, 2004); Microsoft Azure Architecture Center (2024); AWS Migration Blog (2024)
- **Проверка качества (E.21):** Определены metric-based exit criteria для каждого шага; старая система видит снижение трафика в мониторинге пропорционально прогрессу миграции

### P-07: Anti-Corruption Layer — защита внутренней модели от внешнего языка

- **Проблемная ситуация:** Внешняя система (legacy, third-party API, другой bounded context) использует чужой язык и модель; прямая интеграция загрязняет внутреннюю модель чужими именами и концепциями
- **Силы:** Прямой mapping = быстро и просто; но чужая модель просачивается внутрь, нарушая ubiquitous language
- **Ход (решение):** Создать явный слой-переводчик между своей и чужой моделью. ACL преобразует не только формат данных, но и семантику: `external.status=2` → `internal.OrderStatus.CONFIRMED`. ACL живёт на границе bounded context
- **Последствия:** Внутренняя модель остаётся чистой; команда может менять внешнего провайдера без изменения ядра; явная точка для логирования и мониторинга интеграции
- **Trade-offs:** Дополнительный слой = дополнительная complexity; нужно поддерживать при изменении внешнего API
- **Пример:** При интеграции с Stripe ACL переводит `payment_intent.status="requires_action"` в домен-специфичный `PaymentStatus.AWAITING_3DS`
- **Пограничные случаи:** Двунаправленный ACL (и входящие, и исходящие данные); ACL с state (если внешний API stateful — может потребоваться persistence в ACL)
- **Антипаттерны:** ACL без трансляции семантики (просто меняет имена полей); «Conformist» там, где модель важна для core domain; ACL превращается в God-translator со всей бизнес-логикой
- **Связи:** стыкуется с [P-02] (контракт upstream), [P-06] (strangler: ACL для изоляции от legacy)
- **Когда НЕ применять:** Интеграция с partnership-контекстом, где общая модель согласована (Shared Kernel); внешний API уже говорит на вашем языке
- **SoTA-источник:** Evans E. DDD (2003), ch.14; Software Architecture Guild (2024); Vernon V. Implementing DDD (2013)
- **Проверка качества (E.21):** Внутри bounded context нет ни одного типа/поля, названного по чужому API; ACL — единственное место, где упоминается `external.*`

### P-08: Contract-First (API-First) проектирование

- **Проблемная ситуация:** Команда начинает кодить API «как удобно», потребители разбираются по факту; изменения API ломают клиентов неожиданно
- **Силы:** Код пишется быстро «по наитию»; но без upfront контракта параллельная разработка producer/consumer невозможна; дизайн API определяет UX для dev-потребителей
- **Ход (решение):** Architect проектирует и согласует API schema (OpenAPI / AsyncAPI / Protobuf) до написания кода. Consumers дают feedback на schema — изменения дешевле, чем после реализации. Контракт версионируется и хранится в VCS. Потребители используют schema для генерации клиентов и мок-серверов
- **Последствия:** Параллельная разработка producer и consumer; consumers не блокированы; breaking changes обнаруживаются до deploy
- **Trade-offs:** Требует upfront design time; schema может стать outdated если её не синхронизировать с кодом (нужен contract test)
- **Пример:** AWS Prescriptive Guidance формулирует обязательное правило: «Design your API before you write any code» и предписывает schema-first подход для всех интеграций (AWS Prescriptive Guidance, 2024). Внутренний «Bezos API Mandate» (2002) закрепил API-first как организационный стандарт Amazon [hypothesis: оригинальный внутренний меморандум недоступен публично; источник — пересказ бывших сотрудников, наиболее известный — Steve Yegge, 2011]
- **Пограничные случаи:** GraphQL: schema-first; Event-driven: event schema в реестре (Confluent Schema Registry); gRPC: .proto как канонический контракт
- **Антипаттерны:** «Code-first» — генерировать OpenAPI из кода (теряется намеренность дизайна); недокументированные breaking changes; контракт только в голове architect'а
- **Связи:** стыкуется с [P-02] (контракт), [P-12] (handoff — контракт входит в ARCH); противоречит «build-and-see» подходу
- **Когда НЕ применять:** Внутренние функции в пределах одного bounded context; throwaway-прототипы
- **SoTA-источник:** AWS Prescriptive Guidance. API Design Best Practices (docs.aws.amazon.com/prescriptive-guidance/latest/design-api-first, 2024); InfoQ Trends Report 2025
- **Проверка качества (E.21):** Для каждого публичного API есть версионированная schema в VCS, созданная до первого PR с реализацией; consumer-driven contract tests присутствуют в CI

### P-09: Fitness Functions — автоматизированная охрана архитектурных характеристик

- **Проблемная ситуация:** Архитектурные характеристики (производительность, цикличность зависимостей, security) деградируют постепенно — «один PR ничего не сломал». Через год система не соответствует изначальным требованиям
- **Силы:** Нефункциональные требования сложнее верифицировать, чем функциональные; ручная проверка нескалируема; команда не замечает постепенный drift
- **Ход (решение):** Для каждой значимой архитектурной характеристики создать fitness function — автоматизированную проверку в CI/CD: нет циклических зависимостей между модулями (ArchUnit), p95 latency < 200ms (load test в pipeline), нет прямых cross-context DB вызовов (grep/AST-анализ). Блокировать merge при нарушении
- **Последствия:** Архитектурные характеристики верифицируемы объективно; drift обнаруживается при первом нарушении; команда получает fast feedback
- **Trade-offs:** Нужно инвестировать в написание fitness functions; неправильные пороги создают ложные блокировки
- **Пример:** Ford N. et al. вводят fitness functions как ключевую практику evolutionary architecture (Building Evolutionary Architectures, 2017, 2nd ed. 2023). vFunction использует их для architectural observability baselining (2024)
- **Пограничные случаи:** Fitness functions для distributed systems сложнее — latency зависит от сети; нужны базовые линии (baseline), а не абсолютные пороги
- **Антипаттерны:** «Measure everything» — 50 fitness functions с разным приоритетом = шум; fitness function проходит, но в prod нет мониторинга; fitness functions только в QA-окружении (нужны в CI на каждый PR)
- **Связи:** стыкуется с [P-04] (ADR фиксирует target характеристики → fitness function их охраняет); противоречит ручному architectural review без автоматизации
- **Когда НЕ применять:** Прототипы; системы без CI/CD; командам < 3 человека (overhead > польза)
- **SoTA-источник:** Ford N., Parsons R., Kua P. Building Evolutionary Architectures, 2nd ed. O'Reilly (2023); Ford N. et al. Software Architecture Metrics. O'Reilly (2022); code4it.dev (2024)
- **Проверка качества (E.21):** Для каждой зафиксированной в ADR архитектурной характеристики существует ≥1 автоматизированная проверка в CI; нет ни одной «принятой на веру» характеристики без проверки

### P-10: Socio-Technical Alignment — архитектура следует структуре команд

- **Проблемная ситуация:** Система спроектирована «логически правильно», но ownership размыт — один сервис меняют три команды; или наоборот, команда отвечает за части трёх разных bounded contexts
- **Силы:** Закон Конвея: «организации проектируют системы, которые копируют их коммуникационные структуры»; bounded contexts, которые не совпадают с team boundaries, создают permanent coordination overhead
- **Ход (решение):** Architect совместно с CTO и engineering leadership выравнивает границы bounded contexts с team boundaries (Team Topologies: stream-aligned teams). Cognitive load команды — ограничитель: один bounded context не должен превышать когнитивную ёмкость одной команды. При несоответствии — либо менять архитектуру, либо структуру команд
- **Последствия:** Команды работают автономно; меньше cross-team coordination; меньше конфликтов ownership
- **Trade-offs:** Перестройка команд дорога организационно; бывают legacy-ограничения на структуру; архитектура не всегда следует org-chart
- **Пример:** Amazon «two-pizza teams»: команда, которую нельзя накормить двумя пиццами — слишком большая, нужно делить ownership (Amazon Engineering, 2002-2024)
- **Пограничные случаи:** Enabling teams (по Team Topologies) пересекают несколько контекстов — это нормально, они не-owners; Platform teams — владеют платформой, не business-доменами
- **Антипаттерны:** Distributed monolith: микросервисы отдельно, команды вместе; component teams (front/back/data) вместо stream-aligned teams; «архитектурный комитет» — узкое горлышко при любом решении
- **Связи:** стыкуется с [P-01] (размер bounded context = cognitive load команды); противоречит component-team организации
- **Когда НЕ применять:** Очень маленькие организации (< 10 инженеров), где все context'ы на одной команде — overhead от выравнивания избыточен
- **SoTA-источник:** Skelton M., Pais M. Team Topologies, 2nd ed. IT Revolution (2025); Fowler M. ConwaysLaw (martinfowler.com, 2014); InfoQ Trends 2025 (Socio-Technical Architecture)
- **Проверка качества (E.21):** Для каждого bounded context явно определена one-owner team; в git blame для каждого core-файла ≥80% коммитов от одной команды

### P-11: Структурированный Trade-off анализ (NQD: ≥3 альтернативы)

- **Проблемная ситуация:** Architect предлагает «лучшее решение» без явного сравнения альтернатив; команда не понимает, от чего отказались; через год другой architect предлагает то же самое
- **Силы:** Cognitive biases (confirmation bias, anchoring); давление «быстро принять решение»; сложно сравнивать несопоставимые характеристики
- **Ход (решение):** Для каждого значимого решения рассмотреть ≥3 альтернативы (NQD). Для каждой — явный список: какие architectural characteristics улучшаются, какие ухудшаются. Использовать Architecture Characteristics Worksheet (Richards/Ford): scalability, reliability, maintainability, deployability, security, performance, cost. Решение = выбор trade-offs, не поиск «лучшего»
- **Последствия:** Явная документация «что потеряли» при выборе; команда понимает ограничения; упрощает ревизию решения при изменении контекста
- **Trade-offs:** Требует больше времени на анализ; может создать analysis paralysis при одинаково сильных альтернативах
- **Пример:** Ford N., Richards M. «Everything in software architecture is a trade-off. If an architect thinks they have discovered something that isn't a trade-off, more likely they just haven't identified the trade-off yet» (SA: The Hard Parts, 2022)
- **Пограничные случаи:** Reversible vs irreversible решения: для reversible достаточно одной альтернативы; ≥3 — для irreversible и высокой цены ошибки
- **Антипаттерны:** Выбор «популярного» решения без анализа (Netflix делает — и мы сделаем); один автор и один reviewer ADR без diverse perspectives; «решение» = инструмент (Kafka, MongoDB) без анализа архитектурных характеристик
- **Связи:** стыкуется с [P-04] (ADR — место записи trade-off анализа); противоречит «best practice» мышлению без контекста
- **Когда НЕ применять:** Тактические операционные решения (имя переменной, выбор библиотеки-утилиты)
- **SoTA-источник:** Richards M., Ford N. Software Architecture: The Hard Parts (2022), ch.1; Ford N., Richards M. Fundamentals of SA (2024), ch.2; FPF NQD принцип
- **Проверка качества (E.21):** В каждом ADR за последний квартал — ≥2 рассмотренные альтернативы с явным trade-off перечнем

### P-12: Handoff Protocol — передача дизайна в разработку без потерь

- **Проблемная ситуация:** Architect разработал дизайн; dev начинает реализацию и обнаруживает, что контракты не специфицированы, edge cases не описаны, почему именно это решение — непонятно. Architect недоступен — и dev принимает локальные решения, противоречащие дизайну
- **Силы:** Неявные знания architect'а не передаются в коде; ARCH-документ без контекста = «картинка без инструкции»; verbal handoff теряется
- **Ход (решение):** ARCH-NN.md как канонический handoff-артефакт, содержащий: (a) проблему и ограничения, (b) выбранное решение с обоснованием (ссылка на ADR), (c) контракты (API schemas), (d) extension points, (e) edge cases и invariants, (f) что NOT in scope для этой реализации, (g) DoD (Definition of Done) для dev. Architect остаётся доступным для Q&A во время реализации
- **Последствия:** Dev может начать работу без разъяснений; boundary cases обнаружены до кода; misalignment выявляется при чтении ARCH, а не при code review
- **Trade-offs:** ARCH-документ требует времени на написание; риск outdated ARCH при изменении решения
- **Пример:** В Core Team фреймворке: шаблон `.claude/templates/project/arch-template.md` — стандартизированный handoff от architect к dev
- **Пограничные случаи:** Agile-команды: ARCH может быть легковесным (один-два абзаца) для маленьких задач; для сложных features — полный формат
- **Антипаттерны:** ARCH — только диаграмма без текста; handoff «посмотри на ticket»; ARCH обновляется по ходу разработки dev-ом (становится post-hoc документацией)
- **Связи:** стыкуется со всеми P-01 — P-11 (ARCH собирает результаты всех паттернов)
- **Когда НЕ применять:** Trivial изменения в пределах одного файла/функции
- **SoTA-источник:** Winters T., Manshreck T., Wright H. Software Engineering at Google. O'Reilly, 2020, ch.10 «Documentation» (abseil.io/resources/swe-book/html/ch10.html) — design docs как инженерная норма Google; Core Team Framework arch-template.md (2024)
- **Проверка качества (E.21):** ARCH-файл содержит: решение с обоснованием, ≥1 API schema или sequence diagram, список того что NOT in scope; dev после прочтения ARCH не задаёт вопросы о контрактах

### P-13: Saga / Process Manager — координация cross-context workflows

- **Проблемная ситуация:** Бизнес-операция затрагивает несколько bounded contexts (например: «оформить заказ» = резервирование инвентаря + оплата + отправка уведомления). Распределённая транзакция невозможна без 2PC; но 2PC создаёт tight coupling и является точкой отказа
- **Силы:** Нужна eventual consistency при автономии контекстов; откат частично выполненной операции нетривиален; ни один из контекстов не должен знать о деталях других
- **Ход (решение):** Saga — последовательность локальных транзакций, каждая публикует событие, запускающее следующую. Два вида: (a) **Choreography** — события, нет центрального координатора, каждый сервис слушает и реагирует; (b) **Orchestration (Process Manager)** — отдельный Saga Orchestrator управляет последовательностью и вызывает участников явно. Для каждого шага предусмотреть compensating transaction (откат). Process Manager фиксирует состояние Saga в персистентном хранилище
- **Последствия:** Каждый bounded context остаётся автономным; откат реализован через компенсирующие транзакции; eventual consistency явна и документирована
- **Trade-offs:** Сложнее отлаживать чем 2PC; compensating transactions нужно проектировать заранее; видимость состояния хуже без центрального координатора (для Choreography)
- **Пример:** Richardson (microservices.io): «Order Service» — классический пример Saga с участниками Customer Service, Kitchen Service, Delivery Service. Каждый шаг компенсируется: создать заказ / отклонить заказ; заблокировать кредит / разблокировать кредит (Richardson, 2024)
- **Пограничные случаи:** Choreography проще при ≤3 участниках; Orchestration предпочтительна при ≥4 участниках и сложном порядке шагов. Idempotency ключей — обязательна для всех участников (at-least-once delivery). «Lost update» в Saga предотвращается через Semantic Lock countermeasure (Richardson, 2024)
- **Антипаттерны:** Реализовать Saga как синхронную цепочку HTTP-вызовов (не Saga); забыть compensating transactions — при сбое система в inconsistent state навсегда; Process Manager без персистентного состояния (рестарт = потеря прогресса)
- **Связи:** стыкуется с [P-01] (Saga живёт на границе bounded contexts), [P-02] (event schema = контракт между участниками Saga), [P-04] (ADR для выбора Choreography vs Orchestration); требует [P-07] (ACL) при разнородных моделях контекстов
- **Когда НЕ применять:** Операция затрагивает только один bounded context (используй локальную ACID-транзакцию); системы без требования eventual consistency; прототипы и MVP
- **SoTA-источник:** Richardson C. Saga pattern (microservices.io/patterns/data/saga.html, 2024); Hohpe G., Woolf B. Enterprise Integration Patterns — Process Manager (Addison-Wesley, 2004, ch.7); Ford N., Richards M. Software Architecture: The Hard Parts (2022), ch.12
- **Проверка качества (E.21):** Для каждого шага Saga определена compensating transaction; состояние Saga персистируется и наблюдаемо через monitoring; idempotency key задокументирован для каждого участника

## 3. Каталог антипаттернов (сквозной)

| Антипаттерн | Симптом | Почему вреден | Чем заменить (P-NN) |
|-------------|---------|---------------|---------------------|
| **Big Ball of Mud** | Нет явных границ, всё зависит от всего; «страшно трогать» | Любое изменение — high-risk; новые участники не понимают систему | P-01 (Bounded Contexts) |
| **Distributed Monolith** | Микросервисы, но каждое изменение требует синхронного деплоя нескольких; общая БД | Худшее из двух миров: сложность распределённой системы + coupling монолита | P-01, P-02 (явные контракты и границы) |
| **God Schema / Shared Database** | Все сервисы читают/пишут в одну БД; изменение схемы требует координации всех команд | Нет ownership, нет автономии команд; schema — скрытый контракт без версионирования | P-02 (явный контракт) |
| **Architecture Astronaut** | Дизайн ради дизайна; слои абстракций без реального использования; «нам это понадобится» | Complexity tax без benefit; код сложнее поддерживать | P-11 (trade-off анализ: YAGNI как характеристика) |
| **Resume-Driven Architecture** | Выбор технологий для CV, не для задачи («нам нужен Kafka» без анализа why) | Несоответствие технологии задаче; команда не знает инструмент достаточно | P-11 (trade-off через characteristics) |
| **ADR-театр** | ADR создаются формально, но никто не читает; alternatives section пустой | Нет передачи знаний; команда открывает закрытые вопросы повторно | P-04 (живой ADR с alternatives) |
| **Implicit Contract** | Команды «договорились» устно о формате данных; в коде нет schema | Первый же refactor или другая команда нарушает контракт незаметно | P-02, P-08 (contract-first) |
| **Premature Microservices** | Команда < 10 человек дробит на микросервисы с первого дня | Distributed systems complexity без benefit команды из одной пиццы | P-10 (team alignment), P-11 (trade-offs) |
| **Handoff-через-диаграмму** | ARCH = одна картинка без контекста решения | Dev принимает локальные решения, противоречащие intent архитектора | P-12 (полный handoff) |
| **Fitness Function Neglect** | Архитектурные характеристики проверяются вручную «иногда» | Постепенный drift; проблема обнаруживается в prod | P-09 (fitness functions в CI) |
| **2PC в микросервисах** | Команда использует распределённую 2PC-транзакцию для cross-context операции | Tight coupling, единая точка отказа, нет автономии контекстов | P-13 (Saga/Process Manager) |
| **Choreography без observability** | Saga через события без центрального трекинга состояния | Сбой в шаге N не виден без распределённой трассировки; compensating transactions срабатывают «вслепую» | P-13 (Orchestration + persistent state) |

## 4. Карта противоречий (между школами)

> Где практики/школы расходятся. Без этой карты DPF превращается в «книгу рецептов».

| Вопрос | Школа A | Школа B | Как выбирать в нашем контексте |
|--------|---------|---------|--------------------------------|
| **Microservices vs Modular Monolith** | Netflix/Amazon: каждая команда → отдельный сервис, масштабируется независимо | 37signals/Basecamp, DHH: modular monolith достаточен и дешевле; «microservices hype» | Команда < 10 человек → modular monolith; команда > 20 человек с разными ownership зонами → microservices (InfoQ 2025, Team Topologies 2025) |
| **Размер bounded context** | DDD-пуристы (Evans): контекст = бизнес-возможность, может быть крупным | Microservices-адепты: один контекст = один сервис, маленькие | По cognitive load команды. Если один сервис требует синхронизации ≥3 команд — слишком большой. Если изменение одной фичи требует ≥3 деплоев — слишком маленький (Richards/Ford 2022) |
| **Architecture Decision Making** | Architect-as-dictator: единый архитектор принимает все решения | Architect-as-facilitator: architect создаёт условия для решений командой | InfoQ Trends 2025: тренд к facilitator-роли; но для safety-critical систем centralized governance обязательна |
| **TOGAF vs легковесный подход** | Enterprise Architecture (TOGAF 10): полный lifecycle, артефакты, governance board | Agile Architecture (evolutionary arch): минимальные upfront решения, fitness functions охраняют | TOGAF — для enterprise с regulatory compliance; evolutionary arch — для продуктовых tech-компаний. Большинство — гибрид |
| **Синхронный vs асинхронный** | REST-пуристы: synchronous, простота отладки, clear request-response | Event-driven (Kleppmann): async = resilience, decoupling, scalability | Sync для: user-facing запросы с ожиданием ответа, транзакционные операции. Async для: интеграции между bounded contexts, high-throughput, eventual consistency допустима |
| **ADR granularity** | AWS: ADR для каждого архитектурно значимого решения, формально | Google: лёгкие design docs, не требуют heavy process | Для команд < 5 → лёгкие docs; для multi-team систем → формальные ADR с alternatives |

## 5. Guide применения — что / когда / как

| Сигнал ситуации | Применить паттерн(ы) | Не применять |
|-----------------|----------------------|--------------|
| Несколько команд используют разные определения одного термина | P-01 (определить bounded contexts), P-03 (ubiquitous language) | P-02 сразу (сначала границы, потом контракты) |
| Новая фича требует изменений в ≥3 сервисах одновременно | P-01 (пересмотреть границы), P-10 (проверить team alignment) | P-06 Strangler (это не legacy-проблема) |
| Команда боится трогать legacy систему | P-06 (Strangler Fig), P-07 (ACL для изоляции) | P-02 без P-07 (прямой контракт с legacy загрязнит модель) |
| Нужно интегрироваться с внешним API / третьей стороной | P-07 (ACL), P-02 (минимальный контракт), P-08 (contract-first) | P-10 (team alignment здесь не поможет) |
| Команда не помнит «почему так» спустя месяц | P-04 (ADR), P-11 (trade-off analysis) | P-09 (fitness functions не сохраняют решения) |
| Новая платёжная система / провайдер / канал | P-05 (extension points), P-07 (ACL) | P-01 (не нужен новый bounded context под каждого провайдера) |
| Нефункциональные характеристики деградируют | P-09 (fitness functions) | P-04 (ADR не автоматизирован) |
| Architect передаёт работу dev-команде | P-12 (handoff ARCH), P-02 (контракты в ARCH), P-05 (extension points в ARCH) | Verbal handoff |
| Стартап, команда 3-5 человек | P-03 (язык), P-04 (лёгкие ADR), P-11 (trade-offs) | P-09, P-10, P-06 (overhead не окупится) |
| Enterprise, regulatory compliance | P-04 (формальные ADR), P-09 (fitness functions для compliance), TOGAF-alignment | Evolutionary arch без governance |
| Операция пересекает ≥2 bounded contexts с необходимостью отката | P-13 (Saga: Choreography если ≤3 участника, Orchestration если ≥4) | 2PC-транзакция; синхронная цепочка HTTP-вызовов |
