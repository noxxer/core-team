# DPF: designer — UI/UX-дизайн

> Предметный учебник для AI-агента (Domain Principles Framework). Идёт **после FPF**:
> FPF даёт общие принципы работы, DPF — SoTA-ходы, термины, ошибки и границы этого домена.
> Паттерны — декларативный метод мышления, **не** инструкция на выполнение (E.8).

## 0. Scope & Bounded Context (FPF A.1.1)

- **В фокусе:** Проектирование UI-состояний экрана (loading / empty / error / success / partial), mobile-first стратегия приоритизации, паттерны взаимодействия (тач, жесты, навигация), базовая доступность (WCAG 2.2 AA). Ремесло дизайнера в продуктовой команде.
- **Вне фокуса:** Брендинг и визуальная идентичность, иллюстрация, моушн-дизайн как самоцель, исследования пользователей (UX research как отдельная дисциплина), системы дизайна уровня enterprise (Design Tokens deep-dive), бэкенд/фронтенд имплементация.
- **Перевод терминов:** «Состояние экрана» = UI State; «Частичная загрузка» = Partial / Skeleton State; «Пустое состояние» = Empty State; «Доступность» = Accessibility (a11y).
- **Maturity:** refined — провенанс-пробелы §6 закрыты (P-09/P-07/P-11), 9 из 11 каналов покрыты; живая верификация с дизайнером-практиком не пройдена (E.21 — явный открытый пробел, блокирует переход в reliable).

## 2. Паттерны (FPF E.8)

### P-01: Полная модель 5 состояний экрана

- **Проблемная ситуация:** Команды проектируют только «идеальный» сценарий (populated success state), упуская остальные состояния. При первом запуске, ошибке сети или пустом результате поиска пользователь видит пустой или сломанный экран.
- **Силы:** Дизайнеры работают с моками с тестовыми данными; разработчики реализуют только happy path; продукт принимает дизайн без проверки edge cases.
- **Ход (решение):** Для каждого экрана/компонента явно спроектировать все 5 состояний:
  1. **Loading** — индикатор ожидания (spinner или skeleton);
  2. **Empty** — отсутствие данных с объяснением и CTA;
  3. **Error** — описание проблемы + путь восстановления;
  4. **Partial** — частичные данные (skeleton + реальный контент вперемешку);
  5. **Success (populated)** — полный контент.
  В Figma — отдельные фреймы или компонентные варианты для каждого состояния.
- **Последствия:** Пользователи не теряются при пустом/сломанном экране; команда заранее договаривается об edge cases; снижается количество product-bugs на стадии QA.
- **Trade-offs:** Увеличивает объём дизайн-работы в 3-5 раз для каждого экрана; требует договорённостей с PM о копирайте для каждого состояния.
- **Пример:** Экран «Мои документы» в Google Drive имеет: loading (skeleton grid), empty (иллюстрация + «Создать первый документ»), partial (часть файлов загружена, остальные — skeleton), success (полный список), error (offline-баннер + retry).
- **Пограничные случаи:** Pagination: первая страница пустая или загружается — loading; конец данных при прокрутке — специальный «end of list» state; частичная ошибка (некоторые данные недоступны) — смешанный partial+error.
- **Антипаттерны:** «Дизайнить только happy path»; использовать один и тот же пустой экран для разных контекстов («No items» для нового пользователя и для пустого поиска); не проектировать состояние загрузки вообще.
- **Связи:** стыкуется с [P-02] (loading детально), [P-03] (empty детально), [P-04] (error детально), [P-05] (success детально), [P-06] (partial/skeleton детально).
- **Когда НЕ применять:** Простые статичные страницы без асинхронных данных (полностью server-rendered); однократные splash screens.
- **SoTA-источник:** Scott Hurff «Designing Products People Love» (2015), popularized by «The UI Stack» model; подтверждается Trendyol Tech Engineering Blog (Medium, 2024).
- **Проверка качества (E.21):** В Figma для каждого экрана есть 5 именованных вариантов компонента или 5 фреймов в разделе States; дизайн-ревью чеклист включает проверку всех 5 состояний.

### P-02: Дифференциация индикаторов загрузки по длительности ожидания

- **Проблемная ситуация:** Дизайнер ставит один spinner на все случаи жизни — от 200ms до 30 секунд. Пользователь не понимает, сколько ещё ждать; короткие операции создают «мигание» spinner.
- **Силы:** Длительность ответа сервера непредсказуема; дизайнер не всегда знает реальные p95 latency; spinner — самый простой элемент.
- **Ход (решение):** Выбирать индикатор в зависимости от ожидаемого времени:
  - **<1 сек** — никакого индикатора (воспринимается как мгновенное);
  - **1–3 сек** — skeleton screen или looping spinner;
  - **3–10 сек** — determinate progress bar (показывает остаток);
  - **>10 сек** — progress bar + процент + статус-текст («Обрабатываем файл…»);
  - **Background task** — collapsed background indicator, пользователь не заблокирован.
- **Последствия:** Пользователь понимает масштаб ожидания; снижается bounce rate при длинных операциях; skeleton уменьшает воспринимаемое время загрузки.
- **Trade-offs:** Требует знания реальных latency (p50/p95 из аналитики); для skeleton нужен точный макет будущего контента.
- **Пример:** YouTube: skeleton grid для ленты рекомендаций (2–4 сек), progress bar с % при загрузке видео (>10 сек). Stripe Dashboard: spinner только для submit платёжной формы (<3 сек).
- **Пограничные случаи:** Spinner без timeout — пользователь не знает, завис ли процесс. Всегда добавляй fallback timeout с error state. Загрузка по частям (streaming) — используй Partial/Skeleton (P-06).
- **Антипаттерны:** Несколько spinners одновременно на странице; spinner для мгновенных операций (<1 сек); progress bar без реального прогресса (фейковый «анимированный» прогресс до 90% и стоп).
- **Связи:** стыкуется с [P-01] (общая модель), [P-06] (skeleton как partial state); противоречит подходу «один spinner для всего».
- **Когда НЕ применять:** Optimistic UI (P-07) — там загрузка не блокирует; операции с мгновенным локальным ответом.
- **SoTA-источник:** Smart Interface Design Patterns / Vitaly Friedman (smart-interface-design-patterns.com, 2024); NN/g Skeleton Screens article (2017, актуально 2024).
- **Проверка качества (E.21):** В спецификации для каждого действия указан тип индикатора + пороговые значения времени; нет одиночного universal spinner для всех состояний.

### P-03: Empty State как онбординговый инструмент

- **Проблемная ситуация:** Новый пользователь открывает пустой экран и не понимает, что делать дальше. Пустой список без объяснений = потеря пользователя на первом шаге.
- **Силы:** PM хочет убрать «лишний» экран; разработчик возвращает пустой массив; дизайнер не думает о first-run experience.
- **Ход (решение):** Различать 3 типа empty state и проектировать каждый отдельно:
  1. **First-run** (новый пользователь): объяснение ценности + призыв к первому действию + (опционально) иллюстрация/анимация;
  2. **User-cleared** (удалил всё): позитивное подтверждение («Папка пуста») + возможность добавить;
  3. **No results** (поиск без совпадений): что искали + предложение уточнить запрос или сбросить фильтры.
  Копирайт: конкретный («Создайте первый проект»), не абстрактный («Нет данных»). Обязателен единственный чёткий CTA.
- **Последствия:** Снижается churn на онбординге; пользователь понимает, как начать работу; empty state становится touchpoint ценности продукта.
- **Trade-offs:** Требует отдельного копирайта для каждого контекста; иллюстрация/анимация = дополнительный ресурс.
- **Пример:** Slack при первом входе в пустой канал: иллюстрация + «Это начало канала #general» + «Пригласить участников». Dropbox Paper при пустом списке документов: «У вас пока нет документов. Создайте первый.» с большой кнопкой.
- **Пограничные случаи:** Empty state после фильтрации: не показывать онбординговое сообщение — пользователь не новый; Partial empty (часть секций пустая): использовать inline empty state внутри секции, не full-screen.
- **Антипаттерны:** Полностью пустой белый экран; одинаковое сообщение для всех типов empty state; много CTAs (выбор парализует); иллюстрация без actionable текста; задержка показа empty state (сначала spinner, потом пустота без transition).
- **Связи:** стыкуется с [P-01], [P-05] (success feedback при создании первого элемента).
- **Когда НЕ применять:** Empty state для системных секций, которые пользователь не может заполнить сам (например, «Нет уведомлений» — достаточно текста без CTA).
- **SoTA-источник:** Eleken Blog «Empty state UX examples and design rules that actually work» (2024); NN/g «Empty States» research.
- **Проверка качества (E.21):** Для каждого list/grid экрана в Figma есть отдельный empty state фрейм с контекстно-специфичным копирайтом; CTA ведёт к реальному сценарию.

### P-04: Конструктивные сообщения об ошибках

- **Проблемная ситуация:** Пользователь видит «Ошибка 500» или «Что-то пошло не так» без понимания причины и пути решения. Приходится идти в support.
- **Силы:** Разработчики пишут ошибки для логов, не для людей; PM не приоритизирует error copy; дизайнер не всегда знает все классы ошибок.
- **Ход (решение):** Три типа ошибок — три подхода:
  1. **Validation errors** (неверный ввод): inline, рядом с полем, мгновенно при потере фокуса (или at submit). Текст: что не так + как исправить («Введите email в формате name@example.com»);
  2. **System errors** (сервер недоступен): не блокирующий баннер или full-page с объяснением + retry action. Текст без технического жаргона;
  3. **Conflict errors** (несовместимые действия): модальный диалог с объяснением конфликта + вариантами разрешения.
  Правила копирайта: без обвинений пользователя; конкретный путь выхода; не «Попробуйте позже» без конкретики.
- **Последствия:** Снижается количество обращений в support; пользователь самостоятельно восстанавливается после ошибки; NPS растёт (ошибки неизбежны, важно как система на них реагирует).
- **Trade-offs:** Требует классификации всех ошибок бэкенда и написания копирайта; inline validation требует более сложной фронтенд-логики.
- **Пример:** Stripe: при неверной карте — inline текст под полем «Номер карты недействителен. Проверьте 16 цифр». Github: при 404 — полноэкранная страница с поиском «Эту страницу нельзя найти. Попробуйте поиск по репозиторию».
- **Пограничные случаи:** Сетевая ошибка без интернета: не показывать «сервер недоступен» — показать «Нет подключения, проверьте интернет». Частичная ошибка (один из N запросов упал): показывать partial+error state (P-06), не блокировать весь экран.
- **Антипаттерны:** Цвет как единственный индикатор ошибки (нарушение WCAG 1.4.1); сообщение в модале для inline-ошибки; stack trace в пользовательском интерфейсе; «Invalid input» без уточнения; накопление всех ошибок и показ в конце формы.
- **Связи:** стыкуется с [P-01] (error как одно из 5 состояний), [P-08] (доступность: цвет + иконка); противоречит паттерну «один модал для всех ошибок».
- **Когда НЕ применять:** Toast для критических/необратимых ошибок (пользователь может пропустить); не использовать full-page error для валидации полей.
- **SoTA-источник:** Tubik Studio «UX Design for Error States: The Art of Failing Well» (2024); NN/g «Error Message Guidelines»; Nielsen's Heuristic #9 «Help users recognize, diagnose, and recover from errors».
- **Проверка качества (E.21):** В дизайн-спеке все error states имеют категорию (validation/system/conflict); копирайт согласован с PM и написан без технического жаргона; тест на colorblind (Figma sim).

### P-05: Feedback успешного действия — минимально достаточный сигнал

- **Проблемная ситуация:** Пользователь нажал кнопку — ничего не изменилось визуально. Не ясно, сработало ли действие. Либо наоборот: полноэкранная «Success!» страница за каждым кликом = перегрузка.
- **Силы:** Разработчик не проектирует feedback по умолчанию; дизайнер копирует паттерн «Success page» из одного контекста во все; пользователь кликает повторно (дублирование).
- **Ход (решение):** Выбирать тип feedback по критичности и видимости действия:
  - **Тривиальное действие** (лайк, буксет, toggle): microinteraction на месте действия (анимация кнопки, смена иконки);
  - **Завершённая операция** (сохранено, отправлено): toast notification (auto-dismiss 3-5 сек) + undo action если возможно;
  - **Значимое действие** (оплата, удаление аккаунта): inline confirmation в контексте ИЛИ dedicated success screen с summary и next steps;
  - **Async операция** (файл обработан): notification в notification center или email.
- **Последствия:** Пользователь уверен, что действие выполнено; снижается количество повторных кликов; toast + undo снижают anxiety при деструктивных действиях.
- **Trade-offs:** Toast плохо работает для screen readers (требует aria-live); auto-dismiss — нарушение WCAG 2.2.2 если нет возможности pause; dedicated success page избыточна для частых действий.
- **Пример:** Gmail: «Письмо отправлено. Отменить» — toast с undo на 5 сек. Figma: при публикации библиотеки — toast «Library updated». Stripe: при успешном платеже — dedicated success screen с ID транзакции и ссылкой на receipt.
- **Пограничные случаи:** Массовые операции (удалить 50 элементов): один toast с итогом («50 файлов удалено. Отменить»), не 50 отдельных. Операции в фоне (export большого файла): progress + notification по готовности.
- **Антипаттерны:** Никакого feedback после submit; «Success» модал для тривиального действия; toast без aria-live region (невидим для screen readers); auto-dismiss toast для критической информации; дублирование feedback (toast + модал + баннер одновременно).
- **Связи:** стыкуется с [P-07] (optimistic UI — feedback до подтверждения сервера), [P-08] (доступность aria-live).
- **Когда НЕ применять:** Dedicated success page для действий, происходящих несколько раз в минуту; toast для ошибок (путает семантику).
- **SoTA-источник:** NN/g «Indicators, Validations, and Notifications: Pick the Correct Communication Option» (2015, актуально); Pencil & Paper «Success Message UX Examples & Best Practices» (2024).
- **Проверка качества (E.21):** Каждое CTA в дизайне имеет соответствующий success feedback фрейм; toast имеет aria-live в спеке; деструктивные действия имеют undo или confirmation.

### P-06: Skeleton Screen как Partial State

- **Проблемная ситуация:** Контент загружается блоком. Пользователь видит пустой экран до полной загрузки, не понимает, что происходит. Spinner не даёт структурного контекста.
- **Силы:** Сервер отдаёт данные частями (pagination, lazy loading, streaming); контент разной высоты — без заглушек layout прыгает при загрузке; spinner не информирует о структуре будущего контента.
- **Ход (решение):** Использовать skeleton screens (серые прямоугольники, повторяющие будущую структуру) для контентных блоков с временем загрузки 1–10 сек:
  - Skeleton отражает реальную структуру (высота, количество блоков, соотношение текст/изображение);
  - Анимация: shimmer (gradient sweep) или pulse указывает на «живость» процесса;
  - При постепенной загрузке: реальный контент вытесняет skeleton по мере прихода данных.
  Не скелетонить: кнопки, навигацию, статичные элементы (они не грузятся).
- **Последствия:** Пользователь строит mental model будущей страницы; perceived load time снижается; layout не прыгает при появлении контента (CLS = 0).
- **Trade-offs:** Skeleton должен точно отражать структуру — при изменении layout skeleton устаревает; animated skeleton потребляет больше CPU на слабых устройствах.
- **Пример:** LinkedIn лента: skeleton карточки с плейсхолдерами аватара, текста и изображения. YouTube рекомендации: skeleton grid 3×2 с плейсхолдерами превью и заголовков.
- **Пограничные случаи:** Контент с неизвестной высотой (комментарии разной длины): skeleton фиксированной высоты, затем layout expansion при загрузке — неизбежно. Отсутствие данных после загрузки skeleton: плавный переход в Empty State (P-03).
- **Антипаттерны:** Frame-display skeleton (только header/footer, без контента); skeleton для коротких операций (<1 сек — воспринимается как мигание); skeleton, который не совпадает со структурой реального контента; shimmer без реального прогресса (пользователь теряет доверие).
- **Связи:** стыкуется с [P-01] (partial = одно из 5 состояний), [P-02] (skeleton для 1-10 сек загрузки); противоречит «один spinner для всего» [P-02 антипаттерн].
- **Когда НЕ применять:** Операции <1 сек; формы (не контентные блоки); одиночные элементы с spinner достаточен.
- **SoTA-источник:** NN/g «Skeleton Screens» (2017, актуально 2024); Smart Interface Design Patterns — Loading (2024).
- **Проверка качества (E.21):** Skeleton фреймы в Figma точно воспроизводят структуру populated state; shimmer анимация в прототипе; отсутствие skeleton для элементов <1 сек загрузки.

### P-07: Optimistic UI — действие без ожидания

- **Проблемная ситуация:** Пользователь ставит лайк, отмечает задачу выполненной, переключает настройку — и ждёт 1-2 сек. 200-500ms latency разрушает ощущение мгновенного отклика.
- **Силы:** Сетевые запросы всегда имеют latency; паттерн «показать loading → дождаться → обновить» блокирует частые низкорисковые действия; пользователь воспринимает продукт как медленный.
- **Ход (решение):** Для частых, низкорисковых, обратимых действий — применять Optimistic UI:
  1. Немедленно обновить UI (как будто запрос успешен);
  2. Отправить запрос в фоне;
  3. При ошибке — откатить изменение + показать error toast с retry;
  4. При успехе — ничего не делать (UI уже обновлён).
  Критерии применимости: действие обратимо, последствия ошибки минимальны (не финансовые транзакции, не удаление без подтверждения).
- **Последствия:** Perceived performance = мгновенный отклик; пользователь чувствует «живой» интерфейс; снижается perceived latency.
- **Trade-offs:** При ошибке сервера — неожиданный откат UX; требует продуманного rollback + error state; нельзя применять к необратимым действиям.
- **Пример:** Twitter/X: лайк мгновенный (красное сердечко), запрос в фоне. Todoist: задача помечается выполненной сразу, исчезает из списка. При ошибке синхронизации — баннер «Не удалось сохранить, проверьте соединение».
- **Пограничные случаи:** Конфликт данных (два пользователя редактируют одновременно): Optimistic UI не подходит без механизма merge/conflict resolution. Финансовые операции: НИКОГДА — только после подтверждения сервера.
- **Антипаттерны:** Optimistic UI для деструктивных/необратимых действий (удаление, платёж); Optimistic UI без rollback механизма; отсутствие error state при откате.
- **Связи:** стыкуется с [P-05] (минимальный success feedback); противоречит паттерну «загрузчик для каждого действия» [P-02].
- **Когда НЕ применять:** Финансовые транзакции; удаление без корзины; действия с внешними сторонними сервисами; ситуации с нестабильным соединением или систематически высокой вероятностью серверной ошибки [hypothesis: часто упоминаемый порог «>5%» — empirical rule of thumb, непубликованный стандарт; конкретное значение определяется analytics каждого продукта].
- **SoTA-источник:** Simon Hearne «Optimistic UI Patterns for Improved Perceived Performance» (simonhearne.com, 2021); широко применяется Twitter, Todoist, Notion.
- **Проверка качества (E.21):** В спеке указаны все случаи применения Optimistic UI с rollback сценарием; нет Optimistic UI для финансовых/деструктивных операций.

### P-08: Доступность как проектное требование (WCAG 2.2 AA)

- **Проблемная ситуация:** Доступность добавляется «в конце» как checklist, а не как часть дизайн-решений. Цветовые контрасты, навигация с клавиатуры, альт-тексты не учтены — переработка в конце дорогая.
- **Силы:** WCAG воспринимается как «для инвалидов»; дизайнеры работают в идеальных условиях (мышь, высокий контраст, быстрый интернет); accessibility = compliance obligation, а не product quality.
- **Ход (решение):** Встраивать a11y в дизайн-решения с самого начала:
  - **Контраст:** минимум 4.5:1 для обычного текста, 3:1 для крупного (>24px) — WCAG 1.4.3;
  - **Не только цвет:** любой статус (error/success/warning) сопровождается иконкой ИЛИ текстом — WCAG 1.4.1;
  - **Touch targets:** минимум 44×44px (Apple HIG) / 48×48dp (Material Design 3) — WCAG 2.5.5;
  - **Focus states:** видимый focus ring для всех интерактивных элементов (WCAG 2.4.7, усилен в 2.4.11 WCAG 2.2);
  - **Motion:** уважать prefers-reduced-motion; анимации — skip/pause опция (WCAG 2.3.3).
- **Последствия:** Продукт доступен для людей с нарушениями зрения, моторики, когнитивными особенностями; снижается юридический риск (EU Accessibility Act с 2025, EAA); улучшается UX для всех (временные ограничения: рука в гипсе, яркое солнце).
- **Trade-offs:** Усложняет дизайн тёмных/кастомных тем; focus ring «портит» минималистичный дизайн; alt-text для сложных инфографик требует копирайта.
- **Пример:** Material Design 3 встроил WCAG 2.2 compliance в design tokens (color roles с достаточным контрастом по умолчанию). Apple HIG требует minimum touch target 44×44pt.
- **Пограничные случаи:** Custom иконки без label: требуют aria-label или visually-hidden текст. Skeleton screens: нужен aria-label «Загрузка...» для screen readers. Toast auto-dismiss: нарушает WCAG 2.2.2 если нет pause/stop.
- **Антипаттерны:** «Accessibility аудит после релиза»; цвет как единственный носитель смысла; placeholder вместо label для полей (исчезает при вводе); touch targets <44px; отключённые focus styles (outline: none без альтернативы).
- **Связи:** стыкуется с [P-04] (error: не только цвет), [P-05] (toast: aria-live), [P-09] (mobile-first: touch targets).
- **Когда НЕ применять:** AAA критерии избыточны для большинства продуктов; требования к motion (2.3.3) могут конфликтовать с brand animation — принимать осознанно.
- **SoTA-источник:** W3C WCAG 2.2 (октябрь 2023, ISO/IEC 40500:2025); Material Design 3 Accessibility Guidelines (Google, 2023); Apple HIG Accessibility (2024).
- **Проверка качества (E.21):** Figma Contrast Checker plugin показывает все цветовые пары 4.5:1+; нет элементов, где смысл передаётся только цветом; touch targets в mobile layout ≥44px.

### P-09: Mobile-first как стратегия приоритизации контента

- **Проблемная ситуация:** Команда проектирует для десктопа, затем «сжимает» интерфейс для мобильного. Мобильная версия получает все элементы десктопа, только мельче — перегруженная и неудобная.
- **Силы:** Дизайнеры работают на больших экранах; клиент/PM привык видеть «полный» интерфейс; мобильный трафик составляет около 60-65% глобально (StatCounter GlobalStats, gs.statcounter.com, 2024), но продукт исторически начинался с десктопа.
- **Ход (решение):** Начинать дизайн с экрана 390px (iPhone 14). Для мобильного: только критически важные функции на первом экране; прогрессивное добавление возможностей для tablet/desktop.
  Тактики:
  - **Контент-приоритизация**: What, So What, Now What — что главное, почему важно, что делать;
  - **Bottom navigation** для primary actions (thumb-zone: нижняя треть экрана);
  - **Touch targets**: ≥44×44px, spacing ≥8px между элементами;
  - **Type**: минимум 16px body text на мобильном, 1.5× line-height;
  - **Изображения**: responsive images, lazy loading, WebP/AVIF форматы.
- **Последствия:** Мобильные пользователи получают нативный опыт; команда вынуждена расставить приоритеты функционала; desktop становится enhanced версией, не базовой.
- **Trade-offs:** Сложнее убедить стейкхолдеров на ранних макетах (маленький экран = меньше «wow»); некоторые сложные B2B-инструменты реально не работают на мобильном.
- **Пример:** ~49% пользователей держат телефон одной рукой, ~75% касаний — большой палец (Steven Hoober, "How Do Users Really Hold Mobile Devices?", UXmatters, 2013 — частая ошибка приписывать эти цифры Wroblewski). Airbnb: кнопка поиска — центр нижней части экрана (большой палец).
- **Пограничные случаи:** B2B SaaS с таблицами данных: mobile-first не обязан означать «всё в таблице на мобильном» — можно упрощать до ключевых column + detail view. Высокоинтерактивные инструменты (CAD, spreadsheet): mobile-first — companion app, а не полная замена.
- **Антипаттерны:** «Mobile = уменьшенный desktop»; скрытые hamburger-меню для всех primary actions; формы с маленькими полями; использование hover-only interactions; автоплей media без user gesture.
- **Связи:** стыкуется с [P-08] (touch targets — пересечение mobile и a11y), [P-10] (design systems — responsive tokens).
- **Когда НЕ применять:** Внутренние инструменты, используемые исключительно на рабочих станциях; специализированные инструменты (code editors, video production).
- **SoTA-источник:** Luke Wroblewski «Mobile First» (A Book Apart, 2011); Figma Resource Library — Mobile-First Design (2025); Webflow Blog «Mobile-first design: 5 best practices» (2024).
- **Проверка качества (E.21):** Первый фрейм в Figma — 390px; bottom navigation для primary actions; нет интерактивных элементов <44px.

### P-10: Атомарный дизайн и дизайн-токены как основа масштабируемости

- **Проблемная ситуация:** В команде 3+ дизайнера; компоненты дублируются с мелкими отличиями; при смене бренда нужно обновить 200+ экранов вручную; разработчики используют захардкоженные цвета.
- **Силы:** Без системы каждый дизайнер делает «по-своему»; design-dev gap: в Figma одни значения, в CSS другие; быстрый рост продукта = нарастающий хаос.
- **Ход (решение):** Строить компонентную иерархию по Atomic Design (Brad Frost, 2013):
  - **Атомы**: цвет, типографика, иконка, spacing, shadow (зафиксированы дизайн-токенами);
  - **Молекулы**: button, input field, card (атомы + поведение);
  - **Организмы**: form, navbar, product card list;
  - **Шаблоны**: page layouts без реального контента;
  - **Страницы**: реальный контент в шаблоне.
  Дизайн-токены в Figma Variables (2023+) синхронизируются с кодом через Tokens Studio plugin.
- **Последствия:** Изменение токена цвета обновляет все компоненты; dev получает точные значения из Figma Dev Mode; команда говорит на общем языке (Button/Primary не кнопка «сохранить»).
- **Trade-offs:** Начальные инвестиции 2-4 недели на построение системы; Atomic Design пригоден не для всех доменов (контентные сайты — другой подход); токены усложняют адаптацию для multi-brand.
- **Пример:** Airbnb DLS (Design Language System) — одна из первых публичных дизайн-систем на Atomic Design. Material Design 3 — полная реализация с токенами для Android/Web/Flutter.
- **Пограничные случаи:** Одноразовые промо-страницы: не тащить в дизайн-систему; legacy кодовая база без токенов: вводить токены постепенно, начиная с цвета.
- **Антипаттерны:** «Снежинки» (уникальный компонент для каждого экрана); дизайн-система без документации использования; компоненты без состояний (hover/active/disabled/focus/error); токены только для цвета (забыть spacing, typography, radius).
- **Связи:** стыкуется с [P-08] (токены контраста), [P-09] (responsive breakpoints как токены); противоречит ad-hoc дизайну.
- **Когда НЕ применять:** MVP стадия с 1 дизайнером и быстро меняющимся продуктом — overhead системы не окупается; landing pages.
- **SoTA-источник:** Brad Frost «Atomic Design» (atomicdesign.bradfrost.com, 2013); «Design Tokens + Atomic Design» (bradfrost.com, 2024); Material Design 3 (Google, 2023).
- **Проверка качества (E.21):** Figma Library имеет Atoms/Molecules/Organisms секции; все цвета и отступы — через токены, не захардкоженные; компоненты имеют все interactive states.

### P-11: JTBD + Persona как взаимодополняющие инструменты

- **Проблемная ситуация:** Команда либо создаёт детальные персоны («Маша, 32, маркетолог, любит йогу»), которые не помогают принимать дизайн-решения, либо отказывается от персон ради JTBD и теряет контекст пользователя.
- **Силы:** JTBD-евангелисты объявляют персоны «мёртвыми»; персоны без поведенческих инсайтов = демографические стереотипы; выбор «или-или» политизируется в команде.
- **Ход (решение):** Использовать оба инструмента для разных вопросов:
  - **JTBD** отвечает на «что хочет сделать»: «Когда я [ситуация], я хочу [мотивация], чтобы [ожидаемый результат]». Применять для: приоритизации фич, продуктовых решений, позиционирования;
  - **Persona** отвечает на «кто и как»: контекст, ментальные модели, технические ограничения, контекст использования. Применять для: UX writing tone of voice, accessibility decisions, onboarding flow.
  NN/g (2016, актуально): «Personas are not dead — they need to be behavioral, not demographic».
- **Последствия:** Продуктовые решения приземлены на задачу (JTBD), дизайн-решения учитывают контекст пользователя (Persona); снижается «дизайн для воображаемого среднего пользователя».
- **Trade-offs:** Два инструмента = больше времени на исследования; риск противоречий между JTBD и Persona (разрешать через данные, не через «победу» одного инструмента).
- **Пример:** Spotify: JTBD «Когда я еду на работу, хочу слушать музыку без выбора треков, чтобы не отвлекаться» → фича «Daily Mix». Persona «Артём, 28, разработчик, слушает в наушниках ANC» → интерфейс без необходимости смотреть на экран, управление жестами.
- **Пограничные случаи:** Если данных мало — JTBD-гипотезы из customer interviews + proto-persona из демографии. При конфликте: данные важнее модели.
- **Антипаттерны:** «Чистые» демографические персоны без поведенческих insights; JTBD без контекста пользователя (кто выполняет job влияет на решение); дизайн «для всех» без сегментации.
- **Связи:** стыкуется с [P-03] (empty state: first-run experience для конкретной персоны); стыкуется с P-09 (mobile-first: кто и как использует мобильный).
- **Когда НЕ применять:** На самых ранних стадиях (нет данных) — только customer interviews; для B2B с 1-2 типами пользователей персоны избыточны — достаточно JTBD.
- **SoTA-источник:** NN/g «Personas vs. Jobs-to-be-Done» (nngroup.com, 2016); Clayton Christensen, Taddy Hall «Competing Against Luck» (Harper Business, 2016); IxDF «Personas: A Simple Introduction» (interaction-design.org, Lene Nielsen).
- **Проверка качества (E.21):** В дизайн-документе для каждой ключевой фичи есть JTBD statement И behavioral persona (не демографический профиль); нет решений «потому что персона Маша это любит» без поведенческого обоснования.

## 3. Каталог антипаттернов (сквозной)

| Антипаттерн | Симптом | Почему вреден | Чем заменить |
|-------------|---------|---------------|-------------|
| **Happy Path Only** | Дизайн только populated success state | Пользователи видят пустые/сломанные экраны при edge cases | [P-01] — полная модель 5 состояний |
| **Universal Spinner** | Один spinner для операций 200ms–30сек | Пользователь не знает масштаб ожидания; bounce rate растёт | [P-02] — дифференциация по времени |
| **Generic Empty State** | «Нет данных» или «No items» во всех контекстах | Пользователь не понимает, что делать дальше | [P-03] — контекстно-специфичный empty state |
| **Developer Error Messages** | «Error 500», «NullPointerException» в UI | Пользователь в панике; идёт в support | [P-04] — конструктивные human-readable ошибки |
| **Color-Only Signaling** | Красный = ошибка, без иконки/текста | Нарушение WCAG 1.4.1; невидимо для colorblind | [P-08] — иконка + цвет + текст |
| **Desktop-First Shrink** | Мобильный = уменьшенный десктоп | Перегруженный мобильный; неудобные touch targets | [P-09] — mobile-first с приоритизацией |
| **Dark Patterns** | Roach motel, hidden subscriptions, confirm-shaming | Потеря доверия (56% теряют доверие — Dovetail 2023); штрафы DSA EU | Прозрачные, честные UX-паттерны |
| **Snowflake Components** | Уникальный компонент для каждого экрана | Технический долг; несогласованность UI | [P-10] — атомарный дизайн и токены |
| **Demographic Personas** | «Маша, 32, маркетолог» без поведенческих insights | Дизайн-решения не привязаны к реальному поведению | [P-11] — behavioral persona + JTBD |
| **Silent Failure** | Кнопка нажата — ничего не изменилось | Пользователь не знает, сработало ли действие; дублирует | [P-05] — минимально достаточный feedback |
| **Toast for Critical Info** | Критическое сообщение в auto-dismiss toast | Пользователь пропустит; нарушение WCAG 2.2.2 | Inline message или modal для критического |
| **Optimistic UI для деструктивных операций** | Мгновенное удаление без подтверждения сервера | Данные потеряны при ошибке, без возможности восстановления | [P-07] — критерии применимости Optimistic UI |

## 4. Карта противоречий (между школами)

| Вопрос | Школа A | Школа B | Как выбирать |
|--------|---------|---------|--------------|
| **Personas vs JTBD** | NN/g: personas валидны при правильном использовании (behavioral, не demographic) | Christensen/JTBD-сообщество: personas misleading, только JTBD | Использовать оба: JTBD для продуктовых решений, behavioral persona для дизайн-решений [P-11] |
| **Skeleton vs Spinner** | Facebook/YouTube/LinkedIn: skeleton всегда лучше spinner | NN/g: skeleton хорош для layout-heavy content; spinner ок для single actions | По типу контента: контентные блоки → skeleton; одиночные actions → spinner [P-02, P-06] |
| **Mobile-first vs Responsive** | Luke Wroblewski, Google: mobile-first as default strategy | Некоторые B2B-команды: responsive достаточно, mobile — secondary | По аудитории: B2C с >50% mobile → mobile-first; B2B enterprise tool → responsive+desktop-first допустим [P-09] |
| **Optimistic UI vs Conservative UI** | Twitter/Todoist: optimistic для любых частых действий | Banking/healthcare: conservative, всегда ждать подтверждения | По риску: low-risk reversible → optimistic; high-stakes irreversible → conservative [P-07] |
| **Design Systems Early vs Late** | Brad Frost, Airbnb: система с первого дня как инвестиция | Lean UX / startup teams: система замедляет, нужна на зрелом продукте | По стадии: MVP без системы допустимо; после product-market fit → система обязательна [P-10] |
| **WCAG AA vs AAA** | W3C / Accessibility advocates: стремиться к AAA | Прагматики: AA обязательно, AAA нереалистично для коммерческих продуктов | AA = минимальная планка (юридическое требование EU 2025); AAA — для публичных сервисов |
| **Toast vs Inline feedback** | Material Design / NN/g: toast для ephemeral info | Apple HIG: prefer inline feedback, toast только крайний случай | По платформе и критичности: Android → toast ок; iOS → prefer inline; critical info → inline всегда [P-05] |

## 5. Guide применения — что / когда / как

| Сигнал ситуации | Применить паттерн(ы) | Не применять |

| Поиск без результатов | [P-03] No-results empty state | «Нет данных» без guidance |
| Ошибка валидации формы | [P-04] Inline validation рядом с полем | Модальное окно для inline ошибок |
| Системная ошибка / сервер недоступен | [P-04] Non-blocking banner + retry | Silent failure |
| Успешное сохранение | [P-05] Toast 3-5 сек + undo | Full-page success modal |
| Успешная оплата | [P-05] Dedicated success screen с summary | Toast (слишком мало для критического события) |
| Дизайним мобильный | [P-09] mobile-first 390px | Desktop shrink |

| Любой экран с цветовыми статусами | [P-08] цвет + иконка + текст | Color-only signaling |
