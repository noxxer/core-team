# DPF: CTO — Техническая стратегия

> Предметный учебник для AI-агента (Domain Principles Framework). Идёт **после FPF**:
> FPF даёт общие принципы работы, DPF — SoTA-ходы, термины, ошибки и границы этого домена.
> Паттерны — декларативный метод мышления, **не** инструкция на выполнение (E.8).

## 0. Scope & Bounded Context (FPF A.1.1)

- **В фокусе:** Ремесло CTO в части технической стратегии: выбор и эволюция технологического стека, управление техническим долгом, принятие и документирование архитектурных решений (ADR) длинного цикла, построение tech radar, технические аудиты, маршрутизация задач между командами, построение evidence graph для обоснования решений.
- **Вне фокуса:** Продуктовая стратегия (это PM/CPO), операционное управление разработчиками (это Engineering Manager), детали архитектуры конкретных систем (это Architect), HR и рекрутинг (это людской блок), финансовое планирование бюджета (это CFO).
- **Перевод терминов:** «Техдолг» здесь = только управляемый технический долг, не любая неидеальность кода; «стратегия» = видимые, подотчётные, улучшаемые, повторяемые решения (определение по Larson 2024); «аудит» = систематическая проверка технического состояния с Evidence Graph.
- **Maturity:** refined — провенанс-пробелы §6 закрыты (5/5); все 11 каналов Collection Protocol пройдены. Открытые пробелы уровня reliable: верификация ≥8 паттернов против 2+ независимых кейсов из первичных источников; операционализация противоречия §4 «скорость vs правильность» применительно к конкретной стадии компании.

## 2. Паттерны (FPF E.8)

### P-01: Boring Technology First

- **Проблемная ситуация:** Команда выбирает стек для новой системы. Давление в сторону новых технологий («посмотрим на X, все о нём говорят»). CTO должен принять решение быстро, при неполных данных.
- **Силы:** Инженеры хотят изучать новое; маркетинг привлекательности технологии реален. Противовес: нестандартные технологии увеличивают когнитивную нагрузку, снижают нанимаемость, уменьшают community поддержку.
- **Ход (решение):** По умолчанию выбирать скучную (proven, well-understood) технологию. Резервировать «innovation tokens» (~3 на команду) для действительно стратегических ставок, где нестандартность даёт конкурентное преимущество. Всё остальное — commodity stack.
- **Последствия:** Более предсказуемая разработка, лёгкий найм, зрелая экосистема инструментов. Риск: потеря в привлекательности для «рокет-сайнтистов».
- **Trade-offs:** Предсказуемость ↔ инженерный энтузиазм; скорость найма ↔ конкурентное позиционирование технологий.
- **Пример:** Stripe выбирает PostgreSQL и Ruby on Rails для ядра платформы платежей — проверенные инструменты, где важна предсказуемость, а не novelty (Larson, опыт Stripe).
- **Пограничные случаи:** Если вся индустрия переходит на новую технологию (например, миграция с Java 8 на современные JVM) — оставаться на «скучном» становится риском. Технология устарела настолько, что boring = deprecated.
- **Антипаттерны:** «Хайп-стек» — выбор технологии по конференц-трендам без анализа fit; «технологический зоопарк» — разные технологии в каждой новой команде; «вечный proof-of-concept» — trial без decision.
- **Связи:** стыкуется с [P-02] (ADR фиксирует выбор), [P-05] (tech radar оценивает зрелость); противоречит [P-10] (Product-led growth иногда требует быстрых ставок на новое).
- **Когда НЕ применять:** Если ваш product IS the technology (tech company, где инновация — core value proposition); если зрелая технология физически не масштабируется до нужных требований.
- **SoTA-источник:** Dan McKinley «Choose Boring Technology» (2015, mcfunley.com); theartofcto.com Tech Stack Framework (2024); Larson «Crafting Engineering Strategy» ch. on migrations (2024).
- **Проверка качества (E.21):** Решение о стеке задокументировано в ADR с alternatives considered; команда может нанять 3+ инженеров с этим стеком за квартал; нет «почему мы используем X?» вопросов у новичков.

### P-02: ADR как append-only лог архитектурных решений

- **Проблемная ситуация:** В организации накапливаются архитектурные решения, понятные авторам, но непрозрачные для будущих команд. Через 1-2 года никто не помнит «почему мы сделали именно так».
- **Силы:** Решения принимаются в контексте, который меняется; устная культура не масштабируется; будущие инженеры предлагают те же альтернативы, уже рассмотренные и отклонённые.
- **Ход (решение):** Каждое значимое архитектурное решение фиксируется в коротком (1 страница) ADR: контекст → статус → решение → варианты (с объяснением отклонённых) → последствия. ADR хранится в репозитории рядом с кодом. Superseded ADR не удаляется, а помечается.
- **Последствия:** Архитектурная история как explicit knowledge; новые инженеры быстро понимают «почему»; будущие решения можно принимать зная, что уже было рассмотрено.
- **Trade-offs:** Накладные расходы на написание ADR (15-30 мин на решение) ↔ сохранение контекста; риск ADR-груза (сотни записей) ↔ ценность полного лога.
- **Пример:** Google Cloud и AWS Prescriptive Guidance рекомендуют ADR как стандартную практику для сложных архитектурных решений; GitHub хранит ADR прямо в репозитории.
- **Пограничные случаи:** Операционные решения (настройка сервера) не требуют ADR; слишком много ADR на мелкие решения обесценивает систему. Фильтр: «будет ли это решение вопросом через год?»
- **Антипаттерны:** «Мёртвый ADR» — написан, никто не читает, не поддерживается; «ADR-постфактум» — документирование уже реализованного без честного alternatives analysis; «один ADR — вся стратегия» — мегадокумент вместо атомарных записей.
- **Связи:** стыкуется с [P-01] (каждый выбор стека = ADR), [P-03] (техдолг решается через ADR о миграции), [P-07] (evidence graph строится на цепочке ADR).
- **Когда НЕ применять:** Стартап <5 человек в режиме поиска product-market fit — накладные расходы превышают пользу; тактические решения (какой индекс добавить в БД).
- **SoTA-источник:** AWS Architecture Blog «Master ADRs: Best Practices» (2024); Google Cloud Architecture Center ADR Overview; adr.github.io (Michael Nygard формат, 2011).
- **Проверка качества (E.21):** Новый инженер находит ответ на «почему мы используем X?» в ADR без вопросов к команде; при ревью PR reviewer ссылается на конкретный ADR; superseded решения связаны между собой.

### P-03: Управление техдолгом через явный бюджет (70/30 rule)

- **Проблемная ситуация:** Технический долг накапливается молча. «Мы разберёмся позже» → «позже» никогда не наступает. Через 2-3 года долг начинает критически тормозить разработку.
- **Силы:** Бизнес давит на фичи; инженеры не могут «продать» техдолг без бизнес-языка; без явного бюджета техдолг всегда проигрывает фичам в приоритизации.
- **Ход (решение):** Установить явное соотношение: 70% спринта — новые фичи, 30% — техдолг и улучшения. Зафиксировать как командный контракт. Декомпозировать долг по категориям (код, архитектура, зависимости, тесты, документация, инфраструктура) и приоритизировать по влиянию на ключевые бизнес-инициативы. Сделать долг видимым через метрики.
- **Последствия:** Компании, эффективно управляющие техдолгом, высвобождают до 50% инженерного времени для работы над продуктом (McKinsey, «Breaking technical debt's vicious cycle to modernize your business», апрель 2023, mckinsey.com). Без явного бюджета до 40% IT-затрат уходит на поддержку унаследованных систем [vendor-claim: Gartner «Reduce and Manage Technical Debt», gartner.com — точный год отчёта не верифицирован].
- **Trade-offs:** Скорость выхода фичей в краткосрочной перспективе ↔ скорость разработки через 12-18 мес; видимость долга ↔ организационный дискомфорт.
- **Пример:** Команды с явным 15-25% бюджетом на техдолг в каждом спринте показывают стабильно растущий velocity, а не падающий [hypothesis: практическая рекомендация, не контролируемое исследование; коррелирует с паттерном из McKinsey 2023 и Gartner best practices].
- **Пограничные случаи:** Молодой стартап в pivot-режиме — 90/10 оправдан; legacy-система в режиме sunset — 20/80 (преимущественно долг). Формула зависит от стадии.
- **Антипаттерны:** «Техдолг-квартал» — раз в год отдельный спринт для долга (накапливается ещё быстрее); «невидимый долг» — нет систематической инвентаризации; «долг без приоритета» — всё в одну кучу без ранжирования по влиянию.
- **Связи:** стыкуется с [P-02] (решение о погашении долга = ADR), [P-06] (DORA-метрики показывают накопление долга), [P-07] (evidence graph обосновывает бюджет перед бизнесом).
- **Когда НЕ применять:** Режим экстренной доставки (launch window) — допустимо временно; если долг изолирован и не блокирует roadmap.
- **SoTA-источник:** CTO Magazine «Prioritize Technical Debt for CTOs» (2024); Gartner «Reduce and Manage Technical Debt» (gartner.com, документ без фиксированной даты); McKinsey «Breaking technical debt's vicious cycle to modernize your business» (апрель 2023, mckinsey.com); netguru.com «Managing Technical Debt Blueprint» (2025).
- **Проверка качества (E.21):** Velocity команды не снижается квартал-к-кварталу; количество инцидентов, связанных с «известными проблемами», снижается; техдолг видим в backlog с бизнес-обоснованием каждой позиции.

### P-04: Tech Radar как инструмент технологической политики

- **Проблемная ситуация:** В организации нет общего языка для оценки технологий. Разные команды используют разные версии одного инструмента; новые технологии принимаются или отвергаются без явного процесса.
- **Силы:** Технологический ландшафт меняется быстро; централизованный контроль убивает инновацию; децентрализованный выбор создаёт зоопарк.
- **Ход (решение):** Запустить internal Tech Radar по модели ThoughtWorks: 4 квадранта (Техники, Инструменты, Платформы, Языки), 4 кольца (Adopt, Trial, Assess, Hold). Обновлять 2 раза в год. Процесс формирования — collaborative (CTO + tech leads команд). Radar — рекомендации, не мандаты.
- **Последствия:** Команды понимают технологическую политику без микроменеджмента; новые технологии получают явный «вход» через Assess → Trial → Adopt; легаси-технологии получают явный «выход» через Hold.
- **Trade-offs:** Процесс поддержки radar требует времени (2x в год сессии tech leads); риск превращения radar в «политику запретов» если Hold = «нельзя».
- **Пример:** ThoughtWorks публикует открытый Technology Radar с апреля 2010 года; Vol.34 (апрель 2026) охватывает 100+ технологий. Многие компании строят internal radar по их методологии.
- **Пограничные случаи:** Команды <10 инженеров — overhead radar превышает пользу; radar применим к технологическому выбору, но не к архитектурным паттернам (для этого ADR).
- **Антипаттерны:** «Radar-документ» — создан, но не используется в реальных решениях; «radar одного человека» — CTO решает в одностороннем порядке; «статичный radar» — не обновляется, устаревает.
- **Связи:** стыкуется с [P-01] (radar формализует «boring technology»), [P-02] (ADR при переходе технологии между кольцами), [P-08] (Wardley map помогает positioned технологии по эволюции).
- **Когда НЕ применять:** Организации с единым стеком и стабильным технологическим выбором; продукт, где технологический выбор жёстко ограничен платформой (iOS-только, SAP-партнёр и т.п.).
- **SoTA-источник:** ThoughtWorks «Build Your Own Technology Radar» (thoughtworks.com/insights/blog, 2024); ThoughtWorks Technology Radar Vol.34 (апрель 2026).
- **Проверка качества (E.21):** Когда инженер спрашивает «можем ли мы использовать X?», ответ находится в radar за <5 минут; новые технологии идут через Assess прежде чем попасть в production; Hold-технологии имеют явный план миграции.

### P-05: Маршрутизация задач через Team Topologies

- **Проблемная ситуация:** CTO получает запрос: «нам нужна платформа», «нам нужна фича», «у нас сложная задача по ML». Неправильная маршрутизация создаёт когнитивную перегрузку команды или дублирование работы.
- **Силы:** Разные задачи требуют разной оргструктуры; стандартные подходы («фича-команда») не масштабируются при росте.
- **Ход (решение):** Применить модель Team Topologies (Skelton & Pais): 4 типа команд × 3 режима взаимодействия. Stream-aligned team — основной тип, доставка ценности по потоку. Platform team — убирает когнитивную нагрузку со stream-aligned через self-service API. Enabling team — временная, помогает другим командам освоить capability. Complicated subsystem team — постоянная, для задач с высокой специализацией (ML, cryptography). CTO принимает решение о типе команды для новой задачи.
- **Последствия:** Уменьшение когнитивной нагрузки на команды; быстрый flow ценности; явные границы ответственности.
- **Trade-offs:** Структурирование команд требует времени и organizational change; риск бюрократизации при буквальном следовании; Team Topologies лучше работает при масштабе >50 инженеров.
- **Пример:** Spotify реализовал squad/tribe/chapter/guild модель (похожую, но отличную от Team Topologies); Netflix использует высокоавтономные команды с clear platform boundaries.
- **Пограничные случаи:** Стартап <20 человек: все типы команд в одной команде, явная маршрутизация не нужна. При M&A: интеграция разных топологий крайне сложна.
- **Антипаттерны:** «Platform team as gatekeeper» — платформенная команда блокирует вместо того, чтобы снижать нагрузку; «все команды stream-aligned» — теряется платформенная leverage; «комитет по архитектуре» вместо enabling team.
- **Связи:** стыкуется с [P-06] (DORA метрики покажут, правильно ли маршрутизирована задача), [P-09] (build/buy/borrow влияет на нужен ли platform team или vendor).
- **Когда НЕ применять:** Организация в режиме emergency (запустить за 2 недели); ранняя стадия (PMF не найден — оргструктура будет меняться).
- **SoTA-источник:** Matthew Skelton, Manuel Pais «Team Topologies» (2019, IT Revolution); teamtopologies.com/key-concepts; ThoughtWorks bliki «Team Topologies» (Fowler, 2019).
- **Проверка качества (E.21):** Команды могут назвать свой тип и interaction mode с соседними командами; нет ситуации «нам нужно согласование от 5 команд для деплоя»; cognitive load команды явно оценён.

### P-06: DORA-метрики как baseline технической стратегии

- **Проблемная ситуация:** CTO заявляет о «повышении качества» и «ускорении разработки», но нет объективного baseline для измерения прогресса. Субъективные ощущения не убеждают бизнес.
- **Силы:** Инженерные метрики обычно технические (LOC, story points) и легко манипулируемы; бизнес хочет бизнес-метрики; создание новой системы измерений — дорого.
- **Ход (решение):** Внедрить 4 DORA-метрики как объективный baseline: Deployment Frequency (как часто деплоимся), Lead Time for Changes (от commit до production), Change Failure Rate (% деплоев, вызвавших инцидент), Time to Restore Service (MTTR). Данные — из VCS и incident management, не из опросов. Категории: Elite (деплой несколько раз/день, <1ч lead time), High, Medium, Low.
- **Последствия:** Объективный язык для разговора с бизнесом о технических инвестициях; выявление системных проблем (высокий Change Failure Rate = проблема с тестированием или архитектурой).
- **Trade-offs:** DORA показывает доставку, но не качество продукта и не счастье разработчиков — дополнять SPACE; риск «оптимизации метрики» (деплоим чаще ценой качества).
- **Пример:** Atlassian, GitHub, Spotify, Microsoft используют DORA как стандарт; Google's DORA team публикует State of DevOps Report ежегодно с 2014.
- **Пограничные случаи:** Embedded/safety-critical software (медицина, авиация) — другая модель деплоя, DORA не применима напрямую; монолиты без CI/CD — сначала базовые инструменты, потом метрики.
- **Антипаттерны:** «DORA без контекста» — сравнение с индустрией без учёта типа продукта; «метрики как KPI для людей» — DORA для системы, не для оценки инженеров; «только DORA» — без SPACE или DX Core 4 теряется человеческое измерение.
- **Связи:** стыкуется с [P-03] (техдолг виден в DORA: High CFR = долг в тестах/архитектуре), [P-05] (Team Topologies влияет на Lead Time), [P-07] (Evidence Graph включает DORA как наблюдаемые факты).
- **Когда НЕ применять:** Организация без CD-pipeline — сначала CI/CD, потом метрики; команда <5 человек — сигнал слишком зашумлён.
- **SoTA-источник:** Nicole Forsgren, Jez Humble, Gene Kim «Accelerate» (2018); DORA/Google «State of DevOps Report» (ежегодно); getdx.com «DORA Metrics Complete Guide» (2025); SPACE Framework (Forsgren et al., 2021).
- **Проверка качества (E.21):** CTO может показать DORA-тренд за последние 3 квартала; при обосновании технического инвестирования перед советом — конкретные цифры, не ощущения; Change Failure Rate понятен и сравним с предыдущим периодом.

### P-07: Evidence Graph для обоснования технических решений

- **Проблемная ситуация:** CTO предлагает значимое техническое изменение (миграция, инфраструктурный сдвиг, смена стека). Стейкхолдеры сомневаются. Аргументы — «это best practice» и «так все делают» — не убеждают.
- **Силы:** Технические решения кажутся непрозрачными для бизнеса; «доверяйте эксперту» не масштабируется в governance; решения без evidence устаревают незаметно.
- **Ход (решение):** Строить explicit evidence graph: наблюдение (факт из системы) → интерпретация (что это означает) → вывод (какое решение следует) → решение (ADR). Каждый узел графа — с источником и датой. Решения без evidence → в «открытые вопросы», не в стратегию. При pересмотре решения — DRR (Decision Revision Request) с новым evidence.
- **Последствия:** Решения становятся объективно проверяемыми; новые данные ведут к пересмотру, а не к политическим спорам; evidence graph = аудиторский след для будущих команд.
- **Trade-offs:** Формализация evidence = время; риск «paralysis by analysis» при малых решениях — применять только для значимых.
- **Пример:** Will Larson в «Crafting Engineering Strategy» описывает strategy как «visible, accountable, improvable, repeatable decisions» — evidence graph обеспечивает все четыре свойства.
- **Пограничные случаи:** Срочные решения — действие сначала, evidence graph ретроспективно; решения в условиях полной неопределённости — зафиксировать как эксперимент, не как стратегию.
- **Антипаттерны:** «Авторитет без evidence» — «я 15 лет в индустрии, поверьте мне»; «evidence только в голове» — не задокументировано; «cherry-picked evidence» — только подтверждающие данные.
- **Связи:** стыкуется с [P-02] (ADR — финальный узел графа), [P-06] (DORA-данные — первичный evidence), [P-08] (Wardley map = визуализация части evidence graph).
- **Когда НЕ применять:** Тактические решения (<1 недели влияния); решения, явно обратимые без значительных затрат.
- **SoTA-источник:** FPF A.10 Evidence Graph (core-team framework); Will Larson «Crafting Engineering Strategy» (2024); Simon Wardley о situational awareness как предпосылке стратегии (wardleymaps.com).
- **Проверка качества (E.21):** Любой стейкхолдер может спросить «почему это решение?» и получить ответ в виде цепочки evidence → вывод → ADR без объяснений CTO; при появлении новых данных решение явно пересматривается (DRR), а не «забывается».

### P-08: Wardley Mapping для ситуационного осознания

- **Проблемная ситуация:** CTO принимает решения о build vs buy, инвестициях в платформу, стратегических ставках — без явной карты «где мы находимся». Решения принимаются интуитивно или по аналогии с другими компаниями.
- **Силы:** Технологический ландшафт сложен; что является инновацией для одной компании — commodity для другой; copy-paste стратегии («делаем как Netflix») без понимания позиции ведут к ошибкам.
- **Ход (решение):** Строить Wardley Map: ось Y — видимость компонента для пользователя (anchor: user need → value chain); ось X — стадия эволюции (Genesis → Custom-Built → Product/Rental → Commodity/Utility). Компоненты в Genesis/Custom → строить самим или дифференцироваться. Компоненты в Product/Commodity → покупать/арендовать. Карта обновляется при изменении landscape.
- **Последствия:** Явная стратегия build/buy/borrow привязана к реальному положению технологии в эволюции; решения об инвестициях обоснованы.
- **Trade-offs:** Wardley Mapping требует обучения (~несколько часов); карта не точна — это инструмент обсуждения, не истина; ошибки в позиционировании компонентов ведут к стратегическим ошибкам.
- **Пример:** Simon Wardley предсказал доминирование cloud computing (AWS/GCP) используя Wardley Maps, когда облако ещё было в стадии «Custom-Built» для большинства (конец 2000-х). Компании с правильной картой позиции начали outsource-ить инфраструктуру раньше конкурентов.
- **Пограничные случаи:** Карта быстро устаревает в молодых рынках (Genesis компоненты двигаются быстро); неправильно нарисованная карта хуже, чем её отсутствие.
- **Антипаттерны:** «Mapping ради mapping» — красивая карта без влияния на решения; «один CTO рисует карту» — без input от команд теряется реальность; «карта без даты» — устаревшая карта вводит в заблуждение.
- **Связи:** стыкуется с [P-09] (Wardley map отвечает на «build/buy/borrow»), [P-07] (map = visualization evidence graph); противоречит наивному benchmarking (P-10).
- **Когда НЕ применять:** Микрорешения (выбор библиотеки); компании с очень узким технологическим стеком (SaaS на одной платформе).
- **SoTA-источник:** Simon Wardley «Wardley Maps» (wardleymaps.com, с 2016); craftingengstrategy.com главы о Wardley Mapping (Larson, 2024); stratrix.com/learn/frameworks/wardley-mapping.
- **Проверка качества (E.21):** CTO может показать карту и объяснить почему конкретный компонент — в Commodity (покупаем) vs Custom (строим сами); решение build/buy/borrow для новой технологии может быть обосновано позицией на карте.

### P-09: Build/Buy/Borrow для технологических компонентов

- **Проблемная ситуация:** Команда хочет добавить capability (аутентификация, аналитика, платёжная система). CTO должен решить: строить внутри, купить готовое решение, или использовать open-source/партнёра.
- **Силы:** Build = контроль и дифференциация, но дорого и медленно. Buy = скорость, но vendor lock-in и ограничения. Open-source (Borrow) = гибкость, но operational overhead.
- **Ход (решение):** Применить фрейм через Wardley Map: компоненты в стадии Commodity (auth, email, payments) → Buy или Borrow. Компоненты в Custom/Genesis, которые являются core differentiator → Build. Критерии Build: (1) это source of competitive advantage, (2) нет готового решения нужного качества, (3) есть ресурс на поддержку. Покупать commodity — высвобождает ресурсы для differentiator.
- **Последствия:** Оптимизация ресурсов: build там, где это важно; buy там, где это commodity. Современный подход: «buy commodity, build differentiator» как гибрид.
- **Trade-offs:** Build → долгосрочный контроль vs краткосрочная скорость; Buy → скорость vs vendor dependence + ограничения; Borrow (OSS) → гибкость vs operational cost.
- **Пример:** Stripe строит платёжное ядро (differentiator), но использует AWS для инфраструктуры (commodity). Это standard pattern у большинства зрелых tech-компаний.
- **Пограничные случаи:** Regulated industries (banking, healthcare) — «buy» сложнее из-за compliance требований; очень маленькая команда — build = risk (уйдёт один человек, кто поддержит?).
- **Антипаттерны:** «NIH (Not Invented Here) syndrome» — строим всё сами по принципиальным соображениям; «все покупаем» — потеря контроля над ключевыми компонентами, vendor lock-in; «buy без exit strategy» — нет плана смены вендора.
- **Связи:** стыкуется с [P-08] (Wardley map определяет позицию компонента), [P-01] (commodity stack → Buy/Borrow); противоречит NIH-культуре.
- **Когда НЕ применять:** Временный/экспериментальный компонент — купить cheap SaaS и не думать; если рынок предоставляет именно то, что нужно, без компромиссов.
- **SoTA-источник:** theartofcto.com «Making Tech Stack Decisions» (2024); Simon Wardley о build/buy via maps; Daniel Na «Should you build it fast or build it correctly?» (blog.danielna.com, январь 2021).
- **Проверка качества (E.21):** Для каждого нового capability — явное решение build/buy/borrow с обоснованием через Wardley позицию или бизнес-критерий; no «we built X because we always build things».

### P-10: Стратегия выхода из Big Rewrite (Strangler Fig)

- **Проблемная ситуация:** Legacy-система тормозит бизнес. Появляется соблазн: «давайте перепишем с нуля, новым стеком, правильно». CTO должен решить: идти на полный rewrite или нет.
- **Силы:** Старый код эмоционально раздражает инженеров; rewrite кажется чистым листом; «мы теперь умнее». Противовес: старый код содержит встроенное знание о граничных случаях; rewrite занимает в 2-5x дольше планируемого; бизнес не стоит на месте.
- **Ход (решение):** По умолчанию — НИКОГДА не делать Big Rewrite (Joel Spolsky, 2000 — применимо до сих пор). Применять Strangler Fig Pattern (Martin Fowler): заменять legacy постепенно, изолируя и переписывая компоненты по одному, пока strangler полностью не обернёт оригинал. Явный trigger для оценки full rewrite: система физически не масштабируется (не «сложно», а «невозможно»); вся команда авторов ушла и нет документации; compliance требует полной замены.
- **Последствия:** Strangler Fig позволяет доставлять ценность в процессе; rewrite это 2-5 лет «без новых фич». Netscape потерял 3 года и рынок (2000).
- **Trade-offs:** Инкрементальная миграция = технический долг параллельных систем; но это контролируемый долг vs экзистенциальный риск rewrite.
- **Пример:** Netscape в 1998 году решил переписать браузер с нуля → Version 5.0 так и не вышла → IE занял доминирующую позицию → смерть Netscape (Spolsky, joelonsoftware.com, 2000).
- **Пограничные случаи:** Если система <6 месяцев разработки и команда та же — rewrite оправдан (недостаточно встроенного знания); если технология deprecated и нет поддержки (Python 2).
- **Антипаттерны:** «Rewrite и feature-freeze» — бизнес стоит, конкуренты идут; «rewrite без migration strategy» — два года «почти готово»; «fix the debt as you migrate» — попытка решить всё сразу утраивает сроки.
- **Связи:** стыкуется с [P-03] (техдолг оцениваем до решения о rewrite), [P-02] (миграционная стратегия = ADR), [P-08] (Wardley map помогает понять: перезапись ли это или ребрендинг commodity).
- **Когда НЕ применять:** Strangler Fig неприменим, если система — монолит без чётких границ и любое изменение требует полного деплоя (сначала модуляризация).
- **SoTA-источник:** Joel Spolsky «Things You Should Never Do, Part I» (joelonsoftware.com, 2000); Martin Fowler «Strangler Fig Application» (martinfowler.com); Herb Caudill «Lessons from 6 Software Rewrite Stories» (Medium, 2019); AWS Prescriptive Guidance «Migration Strategies» (7Rs framework, 2024).
- **Проверка качества (E.21):** Предложение rewrite встречает явный вопрос «почему не Strangler Fig?» с ответом; если rewrite одобрен — есть ADR с конкретными триггерами и критериями успеха; migration plan опубликован с milestones.

### P-11: Deliberate Engineering Strategy (стратегия как дисциплина)

- **Проблемная ситуация:** CTO принимает важные решения реактивно: под давлением дедлайнов, запросов команды, конкурентного давления. Через год обнаруживается, что «стратегия» — это просто сумма тактических решений без coherence.
- **Силы:** Операционные вопросы съедают стратегическое время; «срочное» вытесняет «важное»; стратегия без давления не возникает сама по себе.
- **Ход (решение):** Зафиксировать engineering strategy как явную дисциплину: (1) Strategy Diagnosis — что сейчас не работает, какие ограничения? (2) Guiding Policy — принципы принятия решений (не планы, а правила). (3) Coherent Actions — скоординированные действия в рамках policy. Обновлять минимум раз в год или при изменении ключевого контекста. Стратегия = «visible, accountable, improvable, repeatable decisions» (Larson).
- **Последствия:** Команды принимают локальные решения coherently, без эскалации каждого к CTO; CTO освобождается от тактики.
- **Trade-offs:** Явная стратегия ограничивает тактическую свободу команд; риск «strategy document» который никто не читает.
- **Пример:** Will Larson в роли CTO Carta разрабатывал explicit engineering strategy с использованием Wardley mapping и systems modeling для ключевых решений (API deprecation, platform investments, LLM adoption) (Larson, 2024).
- **Пограничные случаи:** В режиме поиска PMF — стратегия = «найти что работает», не нужна formalized структура; в M&A — стратегии сталкиваются, нужен явный integration ADR.
- **Антипаттерны:** «Стратегия в голове CTO» — непрозрачна, не масштабируется; «strategy deck» — красивые слайды без actionable decisions; «стратегия без пересмотра» — устаревает за 6-12 месяцев без DRR.
- **Связи:** стыкуется со всеми паттернами (P-01 — P-10); evidence graph (P-07) — backbone стратегии; Wardley map (P-08) — ситуационная основа.
- **Когда НЕ применять:** Стартап <10 человек в режиме поиска PMF — overhead превышает пользу. Но «нет стратегии» ≠ «имплицитная стратегия» = риск.
- **SoTA-источник:** Will Larson «Crafting Engineering Strategy» (2024, O'Reilly); Larson «An Elegant Puzzle» (2019); O'Reilly CTO Hour «Deliberate Engineering Strategy» (oreilly.com/radar, 2024).
- **Проверка качества (E.21):** Любой tech lead команды может пересказать engineering strategy своими словами; при новом решении («делаем ли мы X?») есть guiding policy для ответа без эскалации к CTO; стратегия пересматривалась за последние 12 месяцев.

### P-12: Технический аудит как регулярная практика

- **Проблемная ситуация:** CTO не имеет актуальной картины технического состояния компании. Проблемы обнаруживаются в момент кризиса (инцидент, найм ключевого инженера, due diligence перед инвестицией).
- **Силы:** Технический долг накапливается незаметно; команды адаптируются к своим ограничениям и перестают их замечать; CTO получает filtered view снизу вверх.
- **Ход (решение):** Проводить технический аудит раз в 6-12 месяцев по структуре: (1) архитектура (соответствие текущим требованиям, tech debt map), (2) security (OWASP Top 10, dependency vulnerabilities), (3) delivery (DORA-метрики, pipeline качество), (4) team topology (правильные ли команды для текущих задач), (5) tech radar актуальность (есть ли Hold-технологии без плана миграции). Аудит = evidence для следующего цикла стратегии.
- **Последствия:** CTO имеет актуальную картину; проблемы выявляются превентивно; due diligence перед инвестицией не становится сюрпризом.
- **Trade-offs:** Аудит требует времени команды; риск «аудит ради аудита» без action items.
- **Пример:** Pre-IPO tech due diligence — стандартная практика у крупных инвесторов. Компании, проводящие регулярные внутренние аудиты, проходят DD быстрее и без сюрпризов.
- **Пограничные случаи:** В режиме гиперроста аудит может замедлить. Но «я не знаю, что у нас происходит технически» — неприемлемая позиция для CTO в любой фазе.
- **Антипаттерны:** «Аудит перед кризисом» — реактивный; «аудит без follow-up» — создаёт список проблем без ответственных; «аудит самой командой» — конфликт интересов, нужен внешний взгляд или другая команда.
- **Связи:** стыкуется с [P-03] (аудит → декомпозиция техдолга), [P-04] (аудит обновляет radar), [P-06] (аудит включает DORA), [P-07] (аудит = evidence для strategy).
- **Когда НЕ применять:** Не проводить полный аудит в период активной миграции или M&A интеграции — слишком много движущихся частей; проводить точечные аудиты по критическим компонентам.
- **SoTA-источник:** SoftwareSeni «Technical Debt Assessment and Modernisation Strategy» (2024); OWASP security standards; DORA State of DevOps Report как benchmark.
- **Проверка качества (E.21):** CTO может показать tech debt map по категориям; security vulnerabilities известны и имеют план; DORA-метрики сопоставлены с Industry benchmarks.

| **Хайп-стек** | Выбор технологии по конференц-трендам без анализа fit | Технологический зоопарк; трудности найма; несовместимость | Boring Technology First [P-01] |

| **Невидимый техдолг** | Нет инвентаризации; «мы разберёмся позже» | Долг удваивается каждые 1-2 года; блокирует roadmap | Явный бюджет 70/30 [P-03] |
| **Мёртвые ADR** | ADR написаны, никто не читает, не обновляет | Ложное ощущение документированности; решения противоречат ADR | ADR как append-only лог [P-02] |
| **Platform team as gatekeeper** | Platform team блокирует деплои вместо self-service | Bottleneck; stream-aligned teams теряют автономию | Team Topologies маршрутизация [P-05] |
| **NIH (Not Invented Here)** | «Мы строим всё сами» без анализа | Ресурсы на commodity вместо differentiator | Build/Buy/Borrow [P-09] |
| **DORA как KPI людей** | «Твой deployment frequency низкий — ты плохо работаешь» | Оптимизация метрики ценой качества; демотивация | DORA как system metric [P-06] |
| **Benchmarking без контекста** | «Netflix так делает, значит и мы должны» | Контекст Netflix не ваш; cargo cult без понимания | Wardley Mapping [P-08] |
| **Radar без действий** | Tech radar красивый, Hold-технологии без плана миграции | Radar теряет доверие; статусы расходятся с реальностью | Tech Radar как политика [P-04] |
| **Аудит перед кризисом** | DD-аудит перед инвестицией как первый технический аудит | Сюрпризы, потеря доверия инвесторов, сниженная оценка | Регулярный аудит [P-12] |
| **Fix debt as you migrate** | Миграция + модернизация одновременно | 20x дольше чем rehost; проект зависает | Раздельные фазы: сначала migrate, потом modernize |

## 4. Карта противоречий (между школами)

| Вопрос | Школа A | Школа B | Как выбирать в нашем контексте |
|--------|---------|---------|--------------------------------|
| **Move fast vs Build right** | Product-led: скорость важнее качества пока не найден PMF; исправим потом | Engineering-led: качество с начала снижает общую стоимость; технический долг убивает компанию | Для продуктовых компонентов (user-facing) — скорость до PMF; для платформенных (organizational leverage) — правильность с начала (Daniel Na, blog.danielna.com, январь 2021) |
| **Centralized vs Decentralized tech stack** | Единый стек: предсказуемость, найм, переиспользование; «Platform as a product» | Команды выбирают сами: автономия, лучший инструмент для задачи; уменьшает bottleneck | Центральный radar с рекомендациями (не мандатами) + autonomy внутри Adopt/Trial ring |
| **Build platform early vs Wait for pain** | Инвестируй в platform team рано — дешевле потом | Не строй то, что ещё не нужно; YAGNI; платформа до потребности = overhead | Team Topologies: stream-aligned по умолчанию; platform team когда ≥3 stream-aligned команды испытывают одни и те же friction points |
| **Rewrite vs Strangler Fig** | Иногда технический долг настолько глубок, что только чистый лист поможет | Full rewrite — самая дорогая ошибка (Spolsky 2000) | По умолчанию Strangler Fig; trigger для rewrite — физическая невозможность масштабирования, не «это сложно» |
| **ADR-heavy vs Lightweight docs** | ADR для каждого значимого решения: дорогой процесс, но исчерпывающая история | Меньше документации, больше коммуникации; ADR устаревают | ADR для решений с >12 месяцами последствий; lightweight для тактических; threshold — «будет ли это вопросом через год?» |
| **DORA vs SPACE** | DORA: объективные system metrics, не манипулируемы | SPACE: holistic, включает developer experience и satisfaction | Начать с DORA как baseline; добавить SPACE когда delivery метрики стабилизированы и нужно понять «почему» |

| Организация растёт (>50 инженеров, несколько команд) | P-05 (Team Topologies), P-04 (Tech Radar) | Copy-paste структуры Netflix/Spotify без Wardley map |

| Значимое архитектурное решение принято | P-02 (ADR сразу), P-07 (зафиксировать evidence) | Устное решение «все знают» |
| Проблема с метриками доставки (много инцидентов, медленные деплои) | P-06 (DORA baseline), P-03 (проверить долг как причину), P-12 (аудит delivery pipeline) | Нанять больше людей без диагноза |
| Pre-investment due diligence | P-12 (актуальный аудит), P-07 (evidence graph решений), P-02 (ADR история) | Первый аудит в момент DD |
