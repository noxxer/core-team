# DPF: analyst — рыночная аналитика

> Предметный учебник для AI-агента (Domain Principles Framework). Идёт **после FPF**:
> FPF даёт общие принципы работы, DPF — SoTA-ходы, термины, ошибки и границы этого домена.
> Паттерны — декларативный метод мышления, **не** инструкция на выполнение (E.8).

## 0. Scope & Bounded Context (FPF A.1.1)

- **В фокусе:** Ремесло аналитика как роль в команде продукта/стартапа: постановка вопроса → сбор данных → структурирование (MECE/Pyramid) → конкурентный анализ → синтез инсайтов → оформление для ЛПР (pulse-отчёт, battlecard, market sizing)
- **Вне фокуса:** Финансовый анализ (P&L, DCF), data engineering, BI-дашборды как самоцель, академическое исследование (peer-review), государственная разведка
- **Перевод терминов:** В этом DPF «анализ» = «синтез→инсайт для решения», а не «сбор данных». «Конкурентная разведка» (CI) и «рыночная аналитика» используются как синонимы на уровне ремесла роли
- **Maturity:** refined — провенанс-пробелы закрыты (2026-06-29), 11 каналов пройдено; к reliable требуется E.21 независимая проверка практиком-аналитиком

## 2. Паттерны (FPF E.8)

### P-01: Вопрос до данных (Question-First)

- **Проблемная ситуация:** Аналитик начинает исследование, имея только тему («проанализируй рынок X»), без конкретного вопроса, ответ на который изменил бы решение
- **Силы:** Заказчик формулирует задачу расплывчато. Аналитик боится «уточнять», чтобы не выглядеть некомпетентно. Данные доступны легко, поэтому проще собирать, чем формулировать вопрос
- **Ход (решение):** До начала сбора данных зафиксировать: (1) «Какое решение будет принято на основе этого анализа?», (2) «Что нужно знать, чтобы принять это решение?», (3) «Как выглядит "достаточно хороший" ответ?». Ответы согласовать с ЛПР письменно
- **Последствия:** Анализ фокусируется на нужном; сокращается время сбора данных; заказчик заранее понимает, что получит
- **Trade-offs:** Требует 30–60 минут с ЛПР до старта работы. Иногда ЛПР не может чётко сформулировать вопрос — тогда нужна фасилитация, а не анализ
- **Пример:** McKinsey перед любым проектом формулирует «Key Question» — один вопрос на обложке стартового deck-а. Всё остальное — аргументы в пользу ответа на него (Pyramid Principle, Barbara Minto, 1987)
- **Пограничные случаи:** Разведочный (exploratory) анализ правомерен, когда вопрос неизвестен намеренно — но тогда выход = набор гипотез, а не рекомендация
- **Антипаттерны:** «Исследование ради исследования» — 60-страничный отчёт без рекомендации; «Ответ подтасован» — вопрос формулируется под желаемый результат (см. §3 AN-02)
- **Связи:** стыкуется с [P-02] (MECE-структуризация вопроса), [P-03] (гипотеза до данных); противоречит подходу «сначала данные, потом вопрос» (§4 C-01)
- **Когда НЕ применять:** Мониторинг (pulse-отчёты) и алерты — там вопрос фиксирован заранее и не требует переформулировки каждый цикл
- **SoTA-источник:** Barbara Minto, *The Pyramid Principle* (Pearson, 1987/2002); McKinsey Problem Solving approach, описание в открытых источниках Analyst Academy (theanalystacademy.com, 2024)
- **Проверка качества (E.21):** ЛПР может одной фразой сформулировать решение, которое он примет после прочтения выхода. Если не может — вопрос не зафиксирован

### P-02: MECE-структуризация пространства проблемы

- **Проблемная ситуация:** Аналитик строит список факторов/гипотез/сегментов, и он пересекается (двойной счёт) или неполный (пропущены важные ветки)
- **Силы:** Интуитивный список «из головы» почти всегда неполный и с пересечениями. Давление времени ведёт к первому подходящему разбиению. Нет простого способа проверить «полноту» без структуры
- **Ход (решение):** Построить Issue Tree: разбить вопрос на взаимно исключающие и исчерпывающие подвопросы. На каждом уровне задать тест ME («могут ли два элемента быть одновременно истинными?») и CE («существует ли сценарий, не покрытый ни одним элементом?»). Типичные готовые MECE-ветки: «выручка / затраты», «внутренние / внешние факторы», «рынок / продукт / исполнение»
- **Последствия:** Анализ становится полным и без дублей. Легко делегировать ветки разным аналитикам без пересечений. Структура видна на одном слайде
- **Trade-offs:** Создание строгого MECE занимает 1–3 часа. Иногда реальность не делится MECE-чисто — приходится выбирать «достаточно MECE» разбиение с явными допущениями
- **Пример:** McKinsey-консультанты строят issue tree в начале каждого кейса. Типичный пример: «Почему упала прибыль?» → «Выручка ↓ / Затраты ↑» → каждое — далее по драйверам (MConsultingPrep.com, 2025)
- **Пограничные случаи:** «Матрица 2×2» — частный случай MECE с двумя осями; работает, но рискует упростить. Исторические данные без структуры — антипаттерн, но в разведочном анализе допустим для первого прохода
- **Антипаттерны:** «Список факторов» без проверки ME и CE; «Нарезка по данным» — разбиение определяется доступными данными, а не логикой задачи (см. §3 AN-03)
- **Связи:** стыкуется с [P-01] (вопрос-первый), [P-04] (Pyramid для коммуникации); является основой для [P-03] (hypothesis-driven)
- **Когда НЕ применять:** Когда задача — придумать новое, а не декомпозировать известное (brainstorm, blue-ocean). MECE-жёсткость убивает нестандартные варианты
- **SoTA-источник:** Barbara Minto, *The Pyramid Principle* (1987); Wikipedia, MECE principle (обновлялась 2024); CareerInConsulting.com — MECE guide (2024)
- **Проверка качества (E.21):** Можно ли нарисовать дерево на одном листе без пересечений стрелок? Каждый лист-ветка имеет данные или гипотезу?

### P-03: Hypothesis-Driven анализ (гипотеза до данных)

- **Проблемная ситуация:** Анализ занимает слишком много времени; аналитик собирает «все» данные, прежде чем сделать вывод; результат — «отчёт обо всём»
- **Силы:** Неопределённость пугает: «а вдруг пропущу важное». Культура «нужны данные» создаёт стимул собирать, а не думать. Генеративный AI снижает стоимость сбора — и увеличивает объём лишних данных
- **Ход (решение):** После построения Issue Tree сформулировать гипотезу-ответ на каждую ветку («Мы думаем, что X, потому что Y»). Затем собирать только данные, которые опровергают или подтверждают гипотезы. Если гипотеза опровергнута — обновить, не удалять
- **Последствия:** Время анализа сокращается существенно [hypothesis — McKinsey/BCG описывают ускорение как ключевой эффект, конкретный множитель «2–5×» в первичных источниках не зафиксирован]. Вывод появляется раньше, чем собраны «все» данные. Аналитик может остановить сбор, когда уверенность достаточна для решения
- **Trade-offs:** Требует интеллектуальной смелости — сформулировать ответ до доказательства. Если начальная гипотеза сильно неверна — риск confirmation bias при сборе данных
- **Пример:** McKinsey и BCG работают hypothesis-first: на 2-й день проекта формулируется «пирамида гипотез», которая итеративно уточняется. Публично описано в Analyst Academy и CareerInConsulting (2024)
- **Пограничные случаи:** Explorative research (discovery) — когда домен совершенно неизвестен, начать с hypothesis невозможно; нужен один разведочный sprint без гипотез
- **Антипаттерны:** «Сначала собрать — потом думать» — ведёт к отчёту без рекомендации (§3 AN-01); «Зафиксированная гипотеза» — аналитик не меняет гипотезу при опровергающих данных (§3 AN-02)
- **Связи:** стыкуется с [P-02] (MECE как основа гипотезы); противоречит «индуктивному» подходу (§4 C-02)
- **Когда НЕ применять:** Compliance/audit работа, где нельзя «предполагать» — нужно задокументировать всё найденное; долгосрочное экспертное исследование
- **SoTA-источник:** CareerInConsulting.com — Hypothesis-driven approach (2024); MConsultingPrep.com — Issue Tree (2025); описано в книге Ethan Rasiel *The McKinsey Way* (McGraw-Hill, 1999)
- **Проверка качества (E.21):** Каждый тезис в финальном отчёте имеет ссылку на «гипотезу + данные, которые её подтвердили/опровергли». Нет тезисов без источника

### P-04: Pyramid для коммуникации инсайта

- **Проблемная ситуация:** Аналитик написал 40-страничный отчёт; ЛПР не читает его, или не понимает главного вывода
- **Силы:** Привычка академического письма («введение → данные → вывод»). Страх ошибиться — вывод откладывается на конец. Много данных = ощущение «полного» отчёта
- **Ход (решение):** Структура «перевёрнутой пирамиды»: (1) Главный инсайт / рекомендация — первый слайд. (2) 3 MECE-аргумента — второй уровень. (3) Данные и аналитика — третий уровень, в приложении. Каждый уровень самодостаточен: можно остановиться на Executive Summary и принять решение
- **Последствия:** ЛПР читает один экран и знает главное. Детали доступны по запросу. Время на «написание отчёта» сокращается, потому что структура определена заранее
- **Trade-offs:** Требует чёткого ответа ещё до написания. Если рекомендация спорная, ЛПР может отвергнуть её на первом слайде, не прочитав аргументы — риск «закрытия до диалога»
- **Пример:** McKinsey Deck: обложка = один вопрос + один ответ, страница 2 = три аргумента. Широко описано в WinningPresentations.com (2024) и Medium/LessonsFromMcKinsey (2024)
- **Пограничные случаи:** Плохая новость (проект провален, прогноз пессимистичный) — иногда лучше «ситуация → осложнение → вопрос → ответ» (SCQ-структура Минто), а не прямая рекомендация; снижает сопротивление
- **Антипаттерны:** «Путешествие данных» — хронологический нарратив «мы сделали → нашли → и вот...» без вывода в начале; «Слайд-кладбище» — 30 слайдов данных без структурного вывода
- **Связи:** стыкуется с [P-02] (MECE как основа аргументации), [P-01] (ответ на вопрос-за-анализом); является выходным форматом для [P-07] (pulse-отчёт)
- **Когда НЕ применять:** Регуляторная документация (требует индуктивной структуры); юридические меморандумы; академические статьи
- **SoTA-источник:** Barbara Minto, *The Pyramid Principle* (Pearson, 1987/2002); BetterUp Blog — Minto Pyramid (2024); StrategyU.co — Pyramid Principle Book Summary (2024)
- **Проверка качества (E.21):** Можно ли прочитать только заголовки слайдов и понять всю аргументацию? Если да — Pyramid соблюдена

### P-05: Многоуровневый сбор конкурентных данных (3-Tier CI)

- **Проблемная ситуация:** Конкурентный анализ опирается только на открытые сайты или только на один тип источника; выводы поверхностны или устаревают за 1–2 месяца
- **Силы:** Tier 1 данные (публичные) легко доступны, что создаёт иллюзию полноты. Tier 3 данные (первичные) дороги и медленны. AI ускоряет Tier 1, усиливая дисбаланс
- **Ход (решение):** Структурировать сбор по трём уровням: **Tier 1** (публичные) — сайт, pricing, job postings, patent filings, earnings calls, отзывы в G2/Gartner; **Tier 2** (полупубличные) — отчёты аналитиков, регуляторные filing, конференционные доклады; **Tier 3** (первичные) — win-loss интервью, общение с churned customers, expert networks. Трёхуровневое покрытие + три каденции обновления: ежегодный full refresh, ежеквартальный priority update, event-triggered (M&A, запуск продукта)
- **Последствия:** CI-программа устойчива к разовым изменениям (конкурент убрал pricing со страницы). Первичные данные выявляют инсайты, недоступные публично. Разные сотрудники могут вести разные Tier
- **Trade-offs:** Tier 3 требует бюджета (expert networks, исследовательские ресурсы). AI снижает стоимость Tier 1–2 на 60–70% (Klue State of CI 2025), но Tier 3 нельзя автоматизировать без потери качества
- **Пример:** Infomineo описывает «шестишаговый фреймворк»: define strategic question → scope competitive set → select framework → collect & validate → synthesize → build decision cadence (Infomineo.com, 2024)
- **Пограничные случаи:** Стартап с нулевым бюджетом на CI: начать с Tier 1 + один win-loss-звонок в неделю (Tier 3 минимальный). Крупная компания: все три Tier + dedicated CI-аналитик
- **Антипаттерны:** «Только Tier 1» — анализ только публичных данных; «CI без каденции» — разовый анализ вместо процесса; «Данные на полку» — CI без распространения до принимающих решений (§3 AN-04)
- **Связи:** стыкуется с [P-06] (win-loss как Tier 3), [P-07] (pulse-отчёт как вывод CI); противоречит «точечному CI» (§4 C-03)
- **Когда НЕ применять:** Микро-стартап на pre-product стадии — вместо CI нужна customer discovery (JTBD-подход)
- **SoTA-источник:** Infomineo.com — Competitive Analysis Framework Practitioner's Guide (2024); Alpha Sense — Competitor Analysis Framework (alpha-sense.com, 2024); Klue State of CI (2025)
- **Проверка качества (E.21):** CI-программа имеет задокументированные source map (что/откуда/как часто), и последнее обновление не старше 6 месяцев по каждому priority-конкуренту

### P-06: Win-Loss анализ как первичный CI

- **Проблемная ситуация:** Команда не понимает, почему проигрывает конкурентам; объяснения «мы дороже» или «у них лучше функциональность» не ведут к конкретным изменениям
- **Силы:** Послепродажные данные (CRM) показывают факт (выиграл/проиграл), но не причину. Продавцы дают субъективные объяснения. Покупатели дают разные ответы в анкете и в разговоре. Память деградирует с каждым днём после сделки
- **Ход (решение):** Проводить 20–30 минутные структурированные интервью покупателей в течение 14 дней после закрытия сделки (выиграл + проиграл). Интервьюер — не продавец (нейтральное третье лицо или специальный CI-аналитик). Вопросы строятся на технике «Five Whys» и «Laddering»: идти от поверхностного ответа («дороже») к реальному драйверу («ваш onboarding занимает 3 месяца, у конкурента — 2 недели»). Теги для агрегации шаблонов
- **Последствия:** Реальные конкурентные драйверы становятся известны продукту, маркетингу, сейлзу. Win-rate растёт при внедрении battlecard-ов на основе реальных возражений
- **Trade-offs:** Покупатели не всегда соглашаются на интервью (30–40% acceptance rate). AI-парсинг звонков (Gong, Chorus) даёт масштаб, но хуже качество нюансов. 50% организаций уже используют AI в win-loss программах (Klue, 2025)
- **Пример:** Klue описывает 7-шаговую методологию (Klue.com, 2025): define purpose → build interview guide → recruit respondents → conduct 20-min interview → tag themes → synthesize insights → distribute battlecards
- **Пограничные случаи:** B2C/self-serve продукты — интервью заменяются структурированными exit surveys; enterprise с длинным циклом (12+ мес.) — интервью через 30 дней ещё работают, после 60 — информация деградирует
- **Антипаттерны:** «Анкета вместо интервью» — теряется до 80% нюансов (Klue, 2025); «Продавец интервьюирует» — покупатель говорит то, что не обидит продавца; «Win-loss без распределения» — инсайты не доходят до battlecard-ов
- **Связи:** стыкуется с [P-05] (Tier 3 CI), [P-08] (battlecard строится на win-loss данных)
- **Когда НЕ применять:** ACV < $1K — стоимость интервью выше стоимости сделки; pre-product-market-fit — покупателей нет
- **SoTA-источник:** Klue — Win-Loss Analysis Guide 7-Step (klue.com, 2025); Pragmatic Institute — Win/Loss Analysis (pragmaticinstitute.com, 2024); CompetitiveIntelligenceAlliance.io — Advanced Interview Techniques (2024)
- **Проверка качества (E.21):** За квартал собрано ≥20 интервью (win+loss), тегирование консистентно, топ-3 конкурентных инсайта обновили battlecard-ы и product roadmap

### P-07: Pulse-отчёт как регулярная CI-каденция

- **Проблемная ситуация:** Руководство узнаёт о важных ходах конкурентов из новостей или от клиентов, а не от аналитиков; CI-программа существует, но не интегрирована в ритм принятия решений
- **Силы:** Руководство перегружено: полный CI-отчёт не читается. Разовые отчёты устаревают быстро. Нет регулярного «слота» для CI в расписании ЛПР
- **Ход (решение):** Pulse-отчёт = 1–2 страницы, фиксированный формат, регулярный выпуск (еженедельно для команд, ежемесячно для руководства). Структура: (1) Топ-3 изменения за период + их значимость; (2) Сигналы (новые вакансии, патенты, отзывы, PR); (3) Рекомендуемые действия. Anchor-секции неизменны выпуск за выпуском — для сравнимости
- **Последствия:** CI встраивается в ритм команды. Руководство имеет регулярный «момент» для реакции на конкурентную среду. Аналитик не пишет ad-hoc ответы на вопросы вида «а что делает конкурент X?»
- **Trade-offs:** Риск «коллектора данных без инсайтов» — отчёт превращается в сводку новостей. Нужна дисциплина: если нечего добавить в топ-3 — лучше пропустить выпуск, чем добавить шум
- **Пример:** Описание формата в Sprinklr CI Report guide (2025): тема → сигналы → impact → recommended action → owner. Формат MFS Market Pulse (mfs.com) для инвестиционных команд
- **Пограничные случаи:** Быстро движущийся рынок (AI-индустрия) — еженедельная каденция; стабильный рынок (тяжёлая промышленность) — ежемесячная; кризис (конкурент объявил о поглощении) — event-triggered немедленный алерт
- **Антипаттерны:** «Новостной дайджест» — сводка событий без интерпретации; «Пульс без рекомендации» — отчёт без «что делать» (§3 AN-01); «Перегрузка частотой» — ежедневные отчёты снижают внимание
- **Связи:** стыкуется с [P-05] (CI как источник данных для pulse), [P-04] (Pyramid для структуры); [P-08] (battlecard обновляется по pulse)
- **Когда НЕ применять:** Замена годового стратегического анализа (pulse дополняет, не заменяет); замена глубокого win-loss исследования
- **SoTA-источник:** Sprinklr — Competitive Intelligence Report Guide (2025); Market Analysis Template Guide (MixBright, 2025); SurveyMonkey Pulse Survey Templates (2025)
- **Проверка качества (E.21):** ЛПР упоминает конкретный инсайт из pulse при принятии решения в течение месяца; формат стабилен ≥3 выпуска подряд

### P-08: Battlecard как дистилляция CI для сейлза

- **Проблемная ситуация:** Продавцы встречают возражения типа «конкурент X дешевле» и не знают, что ответить; CI-инсайты существуют в отчётах, но не доходят до разговора с клиентом
- **Силы:** Полный отчёт о конкуренте слишком объёмен для чтения в продажах. У продавца нет времени изучать конкурентов в деталях. Возражения возникают неожиданно, нужен быстрый ответ
- **Ход (решение):** Один экран на одного конкурента: (1) Краткий профиль (кто, ICP, позиционирование); (2) Наши сильные стороны vs этот конкурент; (3) Их сильные стороны (честно); (4) Скрипты ответов на топ-3 возражения (слово в слово); (5) Сигналы «когда этот конкурент точно проиграет нам» (deal signals). Обновляется после каждого значимого win-loss цикла
- **Последствия:** Сейлзы используют CI в переговорах; win-rate против конкретного конкурента растёт. Battlecard живой — обновляется данными, а не архивируется
- **Trade-offs:** Battlecard упрощает реальность. Риск «заучить скрипт» вместо понимания. Требует регулярного обновления — устаревший battlecard хуже чем отсутствие
- **Пример:** Klue — признанный лидер рынка по качеству battlecard-ов (G2 score 9.5/10, Klue.com, 2025). Компании, использующие Klue, отмечают рост win-rate против конкретных конкурентов после внедрения battlecard-ов
- **Пограничные случаи:** B2C — battlecard заменяется FAQ для саппорта; слишком много конкурентов (>20 релевантных) — нужна приоритизация, покрывать top-5 по частоте встреч
- **Антипаттерны:** «SWOT на конкурента» вместо battlecard — не actionable в продажах; «Статичный battlecard» — обновляется раз в год или реже; «Только наши плюсы» — потеря доверия, если продавец не знает реальных слабостей
- **Связи:** стыкуется с [P-06] (win-loss → battlecard content), [P-05] (Tier 1–2 → профиль конкурента), [P-07] (pulse → обновление battlecard)
- **Когда НЕ применять:** Pre-sales/marketing материалы — там нужен другой формат (positioning document); academic competitive analysis
- **SoTA-источник:** Klue CI Platform (klue.com, 2025); Crayon CI Platform (crayon.co, 2025); Kompyte by ZoomInfo (kompyte.com, 2025)
- **Проверка качества (E.21):** Battlecard обновлён <90 дней назад; продавцы могут назвать топ-3 возражения и ответы без чтения battlecard (усвоено)

### P-09: TAM/SAM/SOM — защищаемый bottom-up расчёт

- **Проблемная ситуация:** Аналитик представляет объём рынка, ссылаясь на Statista/Gartner: «рынок $50B, мы займём 1%»; инвесторы или руководство не верят числам
- **Силы:** Top-down данные доступны и выглядят авторитетно. Bottom-up расчёт требует времени и данных об ICP. «1% от TAM» кажется скромным, но инвесторы видят этот паттерн тысячи раз
- **Ход (решение):** Bottom-up: TAM = количество ICP-компаний × ACV в сегменте. SAM = TAM × доля сегментов, где реально конкурируем. SOM = SAM × реалистичная доля захвата за 3–5 лет (на основе темпов роста и pipeline). Верифицировать cross-check: top-down (рынок по данным аналитиков) и bottom-up должны сходиться в ±15%. Если разрыв > 15% — пересмотреть допущения
- **Последствия:** Расчёт защищаем в переговорах с инвесторами. Командам понятно, где реальная точка роста. SOM привязан к реальному pipeline, а не к абстрактной доле
- **Trade-offs:** Bottom-up требует данных о количестве ICP (часто недоступны точно). Верхние числа могут оказаться меньше, чем top-down — это политически сложно
- **Пример:** В 2026 VCs и growth-инвесторы всё сильнее ожидают bottom-up deck: top-down TAM без bottom-up верификации воспринимается как «Statista-summary» (Waveup.com, 2025 — Series A specifically не выделены, относится к VC-инвесторам в целом [industry-claim])
- **Пограничные случаи:** Совершенно новый рынок — bottom-up невозможен, используется analogy-based sizing (похожие рынки в других географиях/отраслях)
- **Антипаттерны:** «1% fallacy» — «если возьмём 1% рынка $100B = $1B»; «TAM = SOM» — путаница уровней; «Statista-тезис» — ссылка без bottom-up верификации (§3 AN-05)
- **Связи:** стыкуется с [P-01] (вопрос — «насколько велик рынок?»), [P-02] (MECE-сегментация рынка для TAM/SAM); противоречит top-down-only школе (§4 C-04)
- **Когда НЕ применять:** Внутренние operational решения, где рыночный размер не влияет на выбор; зрелые рынки с устоявшимися долями (расчёт тривиален)
- **SoTA-источник:** Waveup.com — Market Sizing Top-down vs Bottom-up (2025); Carta.com — TAM SAM SOM (2024); ZenithData.com — TAM SAM SOM Practical Guide (2025)
- **Проверка качества (E.21):** Top-down и bottom-up оценки расходятся < 15%; SOM основан на реальном pipeline/conversion rate, а не на %TAM

### P-10: Сегментация по JTBD (работа, а не демография)

- **Проблемная ситуация:** Традиционная демографическая сегментация (возраст, индустрия, размер компании) дает нечёткие инсайты; продукт «для всех и ни для кого»; позиционирование размыто
- **Силы:** Демография удобна для сбора данных. Стандартные сегменты привычны для отчётности. «Работа» (job) абстрактнее и требует качественного исследования
- **Ход (решение):** Сегментировать по «работам»: что покупатель «нанимает» продукт делать? Например, не «B2B SaaS компании 50–200 сотрудников», а «команды, которые нанимают продукт, чтобы не нанимать аналитика». Для каждой «работы» определить: (1) functional job (что делается), (2) emotional job (как должен чувствовать себя покупатель), (3) social job (как это выглядит для окружения). Конкурент в JTBD — всё, что «нанимается» для той же работы: статус-кво, Excel, другой класс решений
- **Последствия:** Конкурентное позиционирование сужается и усиливается. Продукт строится под конкретную «работу», а не под «сегмент». Ценностное предложение формулируется через Job Statement
- **Trade-offs:** JTBD требует качественных интервью (не анкет). Тяжело агрегировать в количественные данные. Маркетинг привык к демографическим таргетингам — JTBD не ложится напрямую в рекламные платформы
- **Пример:** Milkshake Study — классический пример Клейтона Кристенсена: McDonald's выяснил, что половина milkshake-ов покупается утром как «еда в дороге» (работа = быстрый завтрак, который не разольётся в машине) — совершенно другая «работа», чем «сладкое к обеду» (Strategyn.com, 2024)
- **Пограничные случаи:** Сложные enterprise-закупки с 5+ стейкхолдерами — каждый имеет свою «работу»; JTBD отображает только функциональный слой, требуются дополнительные инструменты для организационной политики
- **Антипаттерны:** «Мы для всех» без «работы»; JTBD-интервью заменяются анкетами (теряется контекст); JTBD без количественной верификации на big N (§3 AN-03)
- **Связи:** стыкуется с [P-06] (win-loss даёт JTBD-данные), [P-09] (JTBD → SAM через ICP-работы); противоречит demographic-first STP школе (§4 C-05)
- **Когда НЕ применять:** Commodity-рынки с одинаковой «работой» у всех; зрелый рынок с устоявшимися категориями, где JTBD уже известен
- **SoTA-источник:** Tony Ulwick, *Jobs to Be Done: Theory to Practice* (Strategyn, 2016); Strategyn.com — JTBD overview (2024); Flevy.com — JTBD-based Segmentation (2024)
- **Проверка качества (E.21):** Каждый целевой сегмент описан через Job Statement («когда [ситуация], я хочу [мотивация], чтобы [результат]»); конкуренты определены в разрезе «работы», а не категории продукта

### P-11: CI-программа как процесс, а не проект

- **Проблемная ситуация:** Компания делает «конкурентный анализ» раз в год для стратегической сессии; за 12 месяцев ситуация меняется, анализ устаревает
- **Силы:** Разовый анализ требует меньших инвестиций. Непрерывный мониторинг кажется операционной нагрузкой. CI воспринимается как «задача аналитика», а не как системный процесс
- **Ход (решение):** Превратить CI в непрерывный процесс с тремя каденциями: Annual (стратегическая: полный refresh для planning), Quarterly (тактическая: update priority конкурентов для leadership review), Event-triggered (оперативная: конкурент объявил M&A/запуск/CEO-смену → немедленный briefing). Источники автоматизированы (alerts, job postings parser, patent tracker). Внутри компании — CI-чемпионы в продажах и продукте, которые передают Tier 3 сигналы
- **Последствия:** Решения принимаются на актуальных данных. CI-команда отвечает на запросы из реального события, а не готовит «запасной анализ». CI устаревший > 6 месяцев считается outdated (consensus среди практиков, AlphaSense 2024)
- **Trade-offs:** Требует выделенного ресурса или инструмента (Klue/Crayon: $15–40K+/год для enterprise [vendor-claim, Vendr.com 2026 — точная цена quote-based, публично не раскрывается]). AI снижает стоимость Tier 1 на 60–70%, но не отменяет нужды в аналитике для синтеза
- **Пример:** McKinsey рекомендует «continuous intelligence development» как основу стратегического планирования: make predictions, track, expand to more competitors over time (McKinsey.com, Getting into Your Competitor's Head, 2024)
- **Пограничные случаи:** Стартап <10 человек: один аналитик + Google Alerts + еженедельный 30-минутный CI-ритуал = минимальный viable процесс
- **Антипаттерны:** «Годовой SWOT» как единственный CI-артефакт; CI без distribution (§3 AN-04); «Инструмент без аналитика» — CI-платформа без человека для синтеза (§3 AN-06)
- **Связи:** стыкуется с [P-05] (3-Tier CI), [P-07] (pulse как output процесса), [P-08] (battlecard как living document)
- **Когда НЕ применять:** Single-player/lifestyle бизнес без конкурентной динамики; pre-product стадия (нечего мониторить)
- **SoTA-источник:** McKinsey — Getting into Your Competitor's Head (mckinsey.com, 2024); Infomineo — Competitive Analysis Framework (2024); AlphaSense — Competitor Analysis Framework (2024); informationR.net — Causes of Competitive Intelligence Failures (isic1813, 2018) — документирует системные причины провала CI как разовых проектов
- **Проверка качества (E.21):** Последний конкурентный алерт обработан <2 недель назад; команда может назвать топ-3 текущих движения конкурентов без подготовки

### P-12: Разделение факт / мнение / гипотеза в аналитических документах

- **Проблемная ситуация:** Аналитический документ смешивает факты с интерпретациями; ЛПР не может отделить «что есть» от «что мы думаем»; решение принимается на мнении, ошибочно принятом за факт
- **Силы:** Естественный язык стирает границу между «данные показывают» и «мы считаем». Генеративный AI уверенно генерирует синтез, не маркируя гипотезы. Аналитик избегает маркировки гипотез, чтобы не казаться «неуверенным»
- **Ход (решение):** В каждом документе явно разделять три уровня: **Факт** (данные с источником и датой, верифицируемые), **Мнение** (интерпретация аналитика, явно помечена «мы считаем»), **Гипотеза** (проверяемое утверждение без текущей верификации, с планом проверки). AI-синтез по умолчанию помечать как «Гипотеза, требует верификации»
- **Последствия:** ЛПР знает, что «факт рынка» и «наша интерпретация» — разные вещи с разной уверенностью. Гипотезы создают backlog верификации, а не «закрытые вопросы»
- **Trade-offs:** Документ становится длиннее. Требует дисциплины от аналитика. Культурно сложно: явные гипотезы выглядят как неуверенность, хотя это честность
- **Пример:** FPF A.7 (Strict Distinction) — тот же принцип на уровне фреймворка: факты и решения хранятся отдельно (domain.md vs decisions/). В Core Team DPF этот паттерн — прямое следствие FPF A.10 (Evidence Graph)
- **Пограничные случаи:** Устный брифинг — маркировка менее формальна, но всё равно нужна («это наша гипотеза»); Доклад для совета директоров — упрощается до факты+рекомендации, гипотезы в приложение
- **Антипаттерны:** «Уверенный синтез без источников» — AI генерирует связный текст без верификации; «Мнение как факт» — «рынок ожидает X» без источника (§3 AN-07)
- **Связи:** стыкуется со всеми паттернами (требование провенанса — сквозное); особенно критично для [P-03] (гипотеза до данных)
- **Когда НЕ применять:** Внутренние брейнсторминг-заметки, где все знают, что это идеи
- **SoTA-источник:** FPF A.7 (Strict Distinction), A.10 (Evidence Graph); Barbara Minto — SCQ-структура (ситуация = факт, осложнение = интерпретация, вопрос = гипотеза); ScienceDaily — Intelligence Analysis Failures (2013)
- **Проверка качества (E.21):** В финальном документе каждый тезис имеет одну из трёх меток [Ф], [М], [Г] (или явный источник). Нет тезисов без маркировки

| AN-01: Отчёт без рекомендации | 40 страниц данных, последний слайд — «Заключение: рынок растёт» | ЛПР не знает, что делать с информацией | [P-04] Pyramid: рекомендация первой |
| AN-02: Confirmation bias CI | Ищем данные, подтверждающие уже принятое решение; опровергающие игнорируются | Выбор конкурентной стратегии на искажённой картине; Kodak-эффект | [P-03] + [P-12]: гипотеза до данных + явная маркировка |
| AN-03: Нарезка по данным | Сегменты определяются тем, что доступно в CRM/GA, а не логикой задачи | MECE не соблюдён; двойной счёт или пробелы | [P-02] MECE-структуризация; [P-10] JTBD-сегментация |
| AN-04: CI на полке | CI-отчёт написан, но не распространён до принимающих решения | Инвестиции в аналитику не окупаются; продавцы не знают о ходах конкурентов | [P-07] Pulse-каденция + [P-08] Battlecard distribution |
| AN-05: Statista-тезис (1% fallacy) | «Рынок $50B, мы возьмём 1% = $500M»; ссылка на third-party report без bottom-up | Инвесторы не верят; внутренние прогнозы оторваны от реальности | [P-09] Bottom-up TAM/SAM/SOM |
| AN-06: Инструмент без аналитика | CI-платформа (Klue/Crayon) куплена, но никто не синтезирует сигналы в инсайты | Данные накапливаются, решения не меняются | [P-11] CI как процесс с аналитиком-синтезатором |
| AN-07: Уверенный AI-синтез | LLM генерирует «факты» без источников; аналитик копирует без верификации | Решения на галлюцинациях; потеря доверия к CI-программе при первом же провале | [P-12] Разделение факт/мнение/гипотеза + верификация |
| AN-08: Годовой SWOT | Один раз в год SWOT по конкурентам; за год картина меняется кардинально | Решения принимаются на устаревших данных; конкурент успел запустить 3 продукта | [P-11] CI-процесс с тремя каденциями |
| AN-09: Interview-only win-loss | Только опрашивают выигранных клиентов; проигранным не звонят | Смещение выборки: не видим реальных причин потерь | [P-06]: проигранные сделки — приоритет для интервью |
| AN-10: Корпоративный шпионаж | Попытки получить CI через неэтичные/нелегальные методы (подкуп, обман) | Репутационный и юридический риск; Oracle vs Microsoft case 2000 (allconsultingfirms.com, 2024) | Легальный Tier 1–3 по [P-05] |

## 4. Карта противоречий (между школами)

> Где практики/школы расходятся. Без этой карты DPF превращается в «книгу рецептов».

| Вопрос | Школа A | Школа B | Как выбирать в нашем контексте |
|--------|---------|---------|--------------------------------|
| C-01: С чего начинать — вопрос или данные? | **Question-First** (McKinsey/Minto): сначала вопрос, потом сбор | **Data-First** (Data Science school): сначала exploratory analysis, вопрос возникает из данных | Если есть ЛПР с решением → Question-First [P-01]. Если домен неизвестен и нет ЛПР → один разведочный sprint Data-First, потом Question-First |
| C-02: Hypothesis-driven vs Inductive? | **Hypothesis-driven** (McKinsey/BCG): гипотеза до данных, быстрее | **Inductive** (academic): данные → паттерны → гипотеза, устойчивее | B2B strategy, time-boxed → hypothesis-driven [P-03]. Academic/policy research, нет time pressure → inductive |
| C-03: Разовый анализ vs Continuous CI? | **Project-based CI**: глубокий анализ раз в год | **Continuous CI** (SCIP, practitioners): CI как постоянный процесс | Fast-moving рынок (AI, SaaS) → Continuous [P-11]. Stable рынок (utilities, commodities) → Annual deep dive + event-triggered |
| C-04: Top-down vs Bottom-up market sizing? | **Top-down**: быстро, использует market reports | **Bottom-up**: медленнее, но защищаем в переговорах | Для инвесторов/Board → Bottom-up [P-09] + cross-check top-down. Для внутренней ориентировки → Top-down допустим |
| C-05: Demographic segmentation vs JTBD? | **STP/Demographic**: удобен для таргетинга в рекламе | **JTBD**: лучше предсказывает purchase и churn | Product positioning, roadmap priority → JTBD [P-10]. Paid acquisition targeting → STP. Идеально: JTBD для стратегии, STP для тактики |
| C-06: Количественный vs Качественный первичный CI? | **Quant-first**: большие выборки, статистика | **Qual-first**: глубина, нюансы, причинность | Win-loss insights (почему) → qual-first [P-06]. Market sizing, share tracking → quant-first [P-09]. N<30 → только qualitative выводы |

## 5. Guide применения — что / когда / как

| Сигнал ситуации | Применить паттерн(ы) | Не применять |
|-----------------|----------------------|--------------|
| ЛПР просит «проанализировать рынок» без конкретики | [P-01] зафиксировать вопрос-за-анализом | [P-09] TAM сразу без вопроса |
| Задача большая, непонятно, с чего начать | [P-02] Issue Tree → [P-03] гипотезы | Сразу собирать данные |
| Нужно убедить инвесторов в размере рынка | [P-09] bottom-up TAM + cross-check | Top-down только ссылка |
| «Почему мы проигрываем конкуренту X?» | [P-06] win-loss → [P-08] battlecard | Анкетирование сейлзов |
| Нужно донести сложный анализ до CEO за 5 минут | [P-04] Pyramid (Executive Summary first) | Хронологический нарратив |
| Руководство не знает, что делают конкуренты | [P-05] 3-Tier CI setup → [P-07] Pulse-каденция | Разовый SWOT |
| Продукт «для всех» и позиционирование размыто | [P-10] JTBD-сегментация | Demographic-only STP |
| AI синтез используется в аналитическом документе | [P-12] маркировка [Ф]/[М]/[Г] обязательна | Прямое копирование без верификации |
| Конкурент объявил о запуске нового продукта | [P-11] event-triggered alert → [P-07] немедленный pulse | Ждать ежеквартального цикла |
| Нужно понять реальные причины выбора конкурента | [P-06] win-loss interviews (нейтральный интервьюер, 14 дней) | Анкета / вопрос продавца |
