# DPF: facilitator

> Предметный учебник для AI-агента (Domain Principles Framework). Идёт **после FPF**:
> FPF даёт общие принципы работы, DPF — SoTA-ходы, термины, ошибки и границы ремесла фасилитации.
> Паттерны — декларативный метод мышления, **не** инструкция на выполнение (E.8).

## 0. Scope & Bounded Context (FPF A.1.1)

- **В фокусе:** ведение рабочей сессии команды (мультиагентной из LLM-ролей и/или человеческой) к ясному результату: открытие сессии, маршрутизация tension к нужной роли, структурирование расхождений, групповое решение с реальным commitment, конструктивная обратная связь, закрытие. Фокус — **процесс**, а не контент.
- **Вне фокуса:** стратегический контент решений (это CTO/architect), управление продуктом (product), модерация многодневных публичных форумов, психотерапия и коучинг личности (Personal Alignment упомянут, но не ядро), организационный редизайн на уровне устава (Holacracy governance берём как технику решения, не как мандат внедрять холакратию в проект).
- **Перевод терминов:** Core-Team `tension` ≈ Holacracy *tension*; `Decider Protocol` (CLAUDE.md) = Core Protocols *Decider*; `Perfection Game` = одноимённый Core Protocol; «3-стадийный протокол вопросов» — локальная надстройка над *Tension → обсуждение ролями → Founder*; «роли» в нашем контексте — субагенты, не люди, поэтому эмоциональные протоколы (Check In с mad/sad/glad/afraid) применяются **только к человеку-Founder**, а к ролям — их структурная часть (presence, commitment, protocol check).
- **Maturity:** `refined`. Пройдено 10 из 11 каналов Collection Protocol (не охвачен: плейбуки топ-5 компаний). Verify-стадия прошла (E.21). Провенанс-пробелы GAP-V1 закрыты пометкой [hypothesis] в тексте. До `reliable` не хватает: калибровки P-09/P-13 на реальных логах сессий Core Team; постраничного прохода Kaner / McCarthy «Software for Your Head»; доменных метрик качества решений с замерами на наших данных.

## 2. Паттерны (FPF E.8)

### P-01: Чек-ин открывает сессию (присутствие → качество результата)

- **Проблемная ситуация:** сессия стартует «с места в карьер», участники физически/контекстно не «в комнате», скрытые состояния (у человека — эмоции; у ролей — недочитанный контекст) тихо саботируют работу.
- **Силы:** давление «нет времени на ритуалы» vs то, что нечитанный контекст и непроговорённое состояние всплывут позже как переделка.
- **Ход:** перед работой — короткий вход. Для человека-Founder: Core Protocols Check In — «I feel [mad/sad/glad/afraid], I'm in» → ответ «Welcome». Для ролей-субагентов: структурный чек-ин = подтверждение прочтения ledger/glossary/своего context.md + «готов». Holacracy: check-in round, по одному, без обсуждения.
- **Последствия:** растёт presence и psychological safety; «глубина чек-ина прямо переводится в качество результата группы» (Core Protocols).
- **Trade-offs:** 1-3 минуты времени; риск формализма, если чек-ин превращается в отчёт.
- **Пример:** Core Team session-start hook инжектит стартовый ритуал чтения артефактов — это и есть структурный чек-ин ролей.
- **Пограничные случаи:** для чисто-агентных прогонов эмоциональная часть неприменима — остаётся только presence/contract-часть.
- **Антипаттерны:** «чек-ин как светская беседа»; чек-ин одной эмоцией для галочки; чек-ин за других («команда устала»).
- **Связи:** стыкуется с [P-12] (закрытие — парный ритуал); питает [P-08].
- **Когда НЕ применять:** мгновенный 1-вопросный lookup — церемония дороже задачи.
- **SoTA-источник:** thecoreprotocols.org/protocols/checkin.html; holacracy.org governance check-in (2026-06-29).
- **Проверка качества (E.21):** хотя бы один участник назвал реальное состояние/контекст-gap, а не «всё ок»; в агентном режиме — все роли подтвердили чтение своих артефактов до работы.

### P-02: Decider — быстрый единогласный commitment

- **Проблемная ситуация:** группе нужно решение, обсуждение ходит по кругу, неясно, кто за что отвечает после.
- **Силы:** скорость vs полнота вовлечения; желание «дожать» vs цена ложного согласия.
- **Ход:** Decider Protocol — конкретное предложение к действию → «1-2-3» → одновременный голос: yes (палец вверх) / no (вниз) / support-it (ладонь). Одновременность убивает якорение на лидера. Если ~50%+ «support-it» или цена интеграции велика — предложение отзывается. Несогласных ведём через Resolution [P-03].
- **Последствия:** решение становится обязывающим commitment всей группы (включая отсутствующих).
- **Trade-offs:** требует конкретного, исполнимого предложения; для размытых тем сначала нужна дивергенция [P-05].
- **Пример:** CLAUDE.md «Decider Protocol»: Clarity → Предложение → Голос → Резолюция.
- **Пограничные случаи:** в агентном режиме «голос» = структурированный yes/no/support от каждой релевантной роли, не палец.
- **Антипаттерны:** последовательное голосование (первый голос якорит); голосование по размытому «давайте улучшим X».
- **Связи:** влечёт [P-03]; останавливается «absolute no»; противоречит долгому консенсусу [P-06] по скорости.
- **Когда НЕ применять:** решение требует исследования/контента, который ещё не собран; вопрос — к Founder по 3-стадийному протоколу.
- **SoTA-источник:** thecoreprotocols.org/protocols/decider.html (2026-06-29).
- **Проверка качества (E.21):** после голоса каждый участник может назвать, к чему он лично привязан; нет «тихих» неголосовавших.

### P-03: Resolution — ввести outlier наименьшей ценой

- **Проблемная ситуация:** после голоса остались 1-2 несогласных; есть соблазн либо продавить, либо бесконечно спорить.
- **Силы:** момент группы vs право меньшинства; краткость vs полнота объяснения.
- **Ход:** проузеру задать каждому несогласному ровно один вопрос: «Что нужно, чтобы ты вошёл?» Ответ — **одно короткое утвердительное предложение** с конкретной правкой. Если человек начинает объяснять возражение — прервать и вернуть: «Что нужно, чтобы ты вошёл?» Простую правку внести и сверить взглядом; сложную — отозвать и пересобрать предложение.
- **Последствия:** расхождение конвертируется в конкретное исполнимое требование, а не в дебаты о позициях.
- **Trade-offs:** жёсткость формата может ощущаться как давление; «absolute no» всё равно снимает предложение.
- **Пример:** Core Protocols Resolution как фаза 7 Decider.
- **Пограничные случаи:** если правок много и они конфликтуют — это сигнал, что нужна дивергенция [P-05]/IDM [P-06], а не resolution.
- **Антипаттерны:** «уговаривание»; разрешение длинных объяснений «почему я против»; торг с «absolute no».
- **Связи:** продолжение [P-02]; при сложных правках — эскалация в [P-06].
- **Когда НЕ применять:** когда возражение валидно и принципиально (вред/откат) — это объект для IDM-интеграции [P-06], не для «что нужно, чтобы ты вошёл».
- **SoTA-источник:** thecoreprotocols.org/protocols/resolution.html (2026-06-29).
- **Проверка качества (E.21):** outlier перешёл в yes/support после одной конкретной правки, а не «сдался».

### P-04: Perfection Game — обратная связь, которая прибавляет

- **Проблемная ситуация:** ревью артефакта скатывается в перечень недостатков, автор защищается, ценные идеи теряются.
- **Силы:** честность vs деструктивность; «у меня есть мнение» vs «оно actionable?».
- **Ход:** перфектор ставит оценку 1-10 = «насколько я могу добавить ценности» (10 = улучшить не могу; 5 = «опишу, как сделать вдвое лучше»), затем «Что мне понравилось: X» и «Чтобы стало 10, нужно: Y» — только конкретика. Если не можешь назвать ни ценность, ни конкретное улучшение — обязан поставить 10.
- **Последствия:** агрегируются лучшие идеи; критика без actionable-пути запрещена структурой.
- **Trade-offs:** дисциплина формата; «10 по умолчанию» иногда прячет «не разобрался».
- **Пример:** Core Team использует Perfection Game в core-protocols как механизм фидбэка.
- **Пограничные случаи:** для бинарных gate-проверок (тест прошёл/нет) формат избыточен — там нужен ревью-чеклист (test-роль).
- **Антипаттерны:** «вот что плохо» без «как до 10»; оценка вкуса, а не добавляемой ценности; занижение балла без конкретного улучшения.
- **Связи:** стыкуется с [P-08] (интервенция через предложение, не диагноз-ярлык).
- **Когда НЕ применять:** security/correctness-ревью, где нужно прямо назвать дефект — там Perfection Game размывает остроту (см. §4).
- **SoTA-источник:** thecoreprotocols.org/protocols/perfectiongame.html (2026-06-29).
- **Проверка качества (E.21):** каждый снятый балл сопровождён конкретным «до 10»; автор уходит с actionable-списком, не с обидой.

### P-05: Разделить дивергенцию и конвергенцию — назвать Зону стона

- **Проблемная ситуация:** генерация опций и выбор между ними слиты в один разговор; доминирующие голоса закрывают пространство до того, как опции исследованы.
- **Силы:** комфорт быстрого схождения vs качество решения; тревога «застряли» в середине.
- **Ход:** явно вести группу по диамонду Kaner: сначала **дивергенция** (расширяем опции, без оценки), затем осознанно входим в **Groan Zone** (дискомфорт интеграции рамок — это норма, не провал), затем **конвергенция** и закрытие. Назвать фазу вслух: «сейчас расширяем, не выбираем».
- **Последствия:** меньше преждевременного схождения и groupthink; решения устойчивее.
- **Trade-offs:** дольше; Groan Zone эмоционально тяжёл, есть соблазн «спасти» группу преждевременным схождением.
- **Пример:** ретроспективы и дизайн-сессии по Diamond of Participatory Decision-Making.
- **Пограничные случаи:** для рутинных операционных решений диамонд избыточен — хватает Decider [P-02].
- **Антипаттерны:** «collapsing generation and selection» в один разговор; трактовка Groan Zone как сбоя и аварийное схождение.
- **Связи:** предшествует [P-02]/[P-06]; усиливается структурами участия [P-09].
- **Когда НЕ применять:** кризис/инцидент с дефицитом времени — там нужна директивная конвергенция, не дивергенция.
- **SoTA-источник:** Sam Kaner, Facilitator's Guide to Participatory Decision-Making (Diamond, Groan Zone), 3rd ed. (2026-06-29).
- **Проверка качества (E.21):** до выбора на столе было ≥3 реальных опции (перекликается с NQD); группа прошла дискомфорт, а не проскочила его.

### P-06: Consent поверх consensus — тест валидного возражения (IDM)

- **Проблемная ситуация:** консенсус «чтобы все хотели» вязнет; одно «мне не нравится» блокирует прогресс.
- **Силы:** инклюзивность vs скорость; «любить решение» vs «мочь с ним жить».
- **Ход:** Holacracy Integrative Decision-Making: предложение от носителя tension → уточняющие вопросы (только понять, не реагировать) → раунд реакций (каждый, кроме проузера, говорит свободно) → опция правки проузером → **раунд возражений**: «нанесёт ли это вред / откатит ли назад?». Возражение валидно, только если показывает вред/откат — не «есть вариант лучше». Валидное → интеграция: правим так, чтобы вред исчез, но tension проузера всё ещё закрывался.
- **Последствия:** быстрые, исполнимые, саморегулирующиеся решения; меньше блокировок предпочтениями.
- **Trade-offs:** требует дисциплины различать реакцию/возражение; фасилитатор должен уметь «тестировать» возражение.
- **Пример:** holacracy.org governance meeting; sociocracy consent rounds.
- **Пограничные случаи:** для решений с высокой ценой ошибки одного consent мало — нужна evidence-проверка (CTO/DRR).
- **Антипаттерны:** трактовать предпочтение как возражение; пропускать раунд реакций; «integration» как уговаривание.
- **Связи:** альтернатива [P-02] для governance-уровня; стыкуется с [P-11] (Investigate в уточняющих вопросах); см. §4.
- **Когда НЕ применять:** простое обратимое операционное решение — Decider [P-02] дешевле.
- **SoTA-источник:** holacracy.org governance-meeting-process; «Why Consent is Better than Consensus»; Circle Forward consent vs consensus (2026-06-29).
- **Проверка качества (E.21):** каждое заблокировавшее возражение прошло тест «вред/откат»; принятое решение интегрировало возражение, а не отбросило tension автора.

### P-07: Tension first — зафиксировать прежде, чем обсуждать

- **Проблемная ситуация:** проблема всплывает в реплике и тут же тонет в дискуссии без owner и без записи.
- **Силы:** энергия «давайте сразу решим» vs дисциплина фиксации; страх бюрократии.
- **Ход:** любое напряжение сначала формулируется как агенда-айтем/задача в одну-две строки (Holacracy agenda-building: один-два слова, без обсуждения; Core Team: TaskCreate до обсуждения). Tension всегда привязан к конкретной роли/ответственности. Только потом — маршрутизация к нужной роли.
- **Последствия:** ничего не теряется; обсуждение получает owner и scope; видна загрузка.
- **Trade-offs:** микро-задержка на запись; риск раздувания бэклога.
- **Пример:** Core Team 3-стадийный протокол — стадия 1 TaskCreate обязательна до обсуждения.
- **Пограничные случаи:** тривиальный вопрос с очевидным ответом фиксировать избыточно.
- **Антипаттерны:** «обсудим прямо сейчас» без фиксации; tension без привязки к роли (повисает в воздухе).
- **Связи:** предшествует [P-08] (маршрутизация) и всем решающим паттернам.
- **Когда НЕ применять:** инцидент-аврал, где фиксация задержит критическое действие (зафиксировать постфактум).
- **SoTA-источник:** holacracy.org agenda-building; Core Team core-protocols (Tension Processing) (2026-06-29).
- **Проверка качества (E.21):** к концу сессии каждый поднятый tension имеет статус (решён/в работе/отложен с причиной), ни один не «испарился».

### P-08: Диагностировать → вмешиваться из взаимного обучения, не из контроля

- **Проблемная ситуация:** группа буксует; фасилитатор под стрессом скатывается в директивный «я знаю, как надо» (unilateral control).
- **Силы:** соблазн «дожать результат» vs то, что контроль гасит вовлечённость и информацию.
- **Ход:** Schwarz: сначала **наблюдать** (что конкретно ограничивает эффективность группы), затем выбрать момент и **вмешаться** прозрачно, делясь наблюдением и проверяя его, а не навязывая вывод. Опираться на 4 ценности (валидная информация, свободный осознанный выбор, внутренний commitment, сострадание) и явные ground rules.
- **Последствия:** растёт внутренний commitment и качество информации; интервенции прозрачны.
- **Trade-offs:** медленнее, чем директива; требует от фасилитатора видеть собственный сдвиг в контроль.
- **Пример:** Skilled Facilitator Diagnose-Intervene cycle, mutual learning mindset.
- **Пограничные случаи:** при явном нарушении процесса допустим быстрый directive Protocol Check [P-10].
- **Антипаттерны:** скрытая интервенция (фасилитатор «рулит» к своему ответу неявно); ground rules, противоречащие прозрачности.
- **Связи:** стыкуется с [P-04], [P-10]; противоречит контентному продавливанию (§4 нейтральность).
- **Когда НЕ применять:** обучающий контекст, где у группы явно нет знаний и нужна экспертная подача (тогда роль = тренер, не фасилитатор).
- **SoTA-источник:** Roger Schwarz, The Skilled Facilitator, 3rd ed. (mutual learning, 4 values, 9 ground rules) (2026-06-29).
- **Проверка качества (E.21):** интервенция названа вслух как наблюдение и проверена («я вижу X, верно?»), а не подана как готовый вывод.

### P-09: Управлять airtime — структурой, а не призывами

- **Проблемная ситуация:** доминирующие голоса забивают тихих; «давайте дадим слово всем» не работает на призывах.
- **Силы:** инклюзивность vs естественная иерархия голоса; время vs полнота.
- **Ход:** распределять контроль микроструктурой, а не словами. Liberating Structures 1-2-4-All: сначала индивидуально (1), затем пара (2), четвёрка (4), потом общий сбор — каждый успевает подумать прежде, чем громкие захватят эфир. Round-robin, тихий старт (письменно), TRIZ/Wise Crowds под задачу.
- **Последствия:** реально включаются все; меньше anchoring на первого спикера.
- **Trade-offs:** структура ощущается «механической»; занимает время на оркестрацию.
- **Пример:** liberatingstructures.com 1-2-4-All как дефолт включения.
- **Пограничные случаи:** в агентном режиме аналог — параллельный fan-out ролей с независимым выводом до агрегации (но см. P-13 про ложный консенсус). Влияние порядка ролей в fan-out на итог — [hypothesis: ожидаемый эффект из теории позиционного смещения, эмпирически не замерен на наших сессиях].
- **Антипаттерны:** «эй, а что думают остальные?» без структуры; открытый брейншторм, где 2 человека держат ~80% эфира [hypothesis — воспроизводимого первоисточника с замером не найдено; типичная практическая наблюдаемость сообщества LS].
- **Связи:** усиливает [P-05] (дивергенция); стыкуется с [P-01].
- **Когда НЕ применять:** диада или решение, где нужен один эксперт — оркестрация избыточна.
- **SoTA-источник:** liberatingstructures.com (1-2-4-All, TRIZ, Wise Crowds); Lipmanowicz & McCandless (2026-06-29). Оптимальный порог концентрации airtime как численная метрика — [hypothesis: замеры сообщества LS не опубликованы в рецензируемых источниках на 2026-06-29].
- **Проверка качества (E.21):** доля airtime/вклада не сконцентрирована на 1-2 участниках; от тихих поступили идеи, попавшие в решение.

### P-10: Нейтральность процесса + Protocol Check как самокоррекция

- **Проблемная ситуация:** фасилитатор незаметно проталкивает свой ответ; формулировка вопросов и акценты подыгрывают доминирующему мнению — группа это чует и зажимается.
- **Силы:** у фасилитатора есть мнение vs его роль — владеть процессом, не контентом; авторитет vs доверие.
- **Ход:** держать различение «процесс ≠ контент»: вести как, не диктовать что. Если возникает нарушение протокола (своё или чужое) — мгновенный **Protocol Check**: назвать, скорректировать, без стыда и наказания. Если у фасилитатора есть контентная позиция — сделать её явной и временно снять «шляпу фасилитатора».
- **Последствия:** доверие к процессу; «момент, когда люди чуют bias, — они начинают молчать» предотвращён.
- **Trade-offs:** требует постоянного self-monitoring; полная нейтральность иногда невозможна (см. §4 «когда фасилитатор НЕ нейтрален»).
- **Пример:** Core Protocols Protocol Check; IAF Competency F (model professional attitude); North Star «when the facilitator must not be neutral».
- **Пограничные случаи:** в ценностно-нагруженных вопросах декларируемая нейтральность сама становится позицией — честнее назвать свою рамку.
- **Антипаттерны:** наводящие вопросы с зашитым ответом; «фасилитатор-эксперт», который под видом процесса проводит свой контент.
- **Связи:** охраняет [P-08], [P-11]; нарушение — частый корень антипаттернов §3.
- **Когда НЕ применять:** когда ты в роли решающего эксперта/архитектора, а не фасилитатора — тогда контент твой, и это надо назвать.
- **SoTA-источник:** thecoreprotocols.org/protocols/protocolcheck.html; iaf-world.org Core Competencies; North Star Facilitators (2026-06-29).
- **Проверка качества (E.21):** участники не могут назвать «предпочитаемый фасилитатором ответ»; нарушения протокола ловятся за секунды и без перехода на личности.

### P-11: Investigate — любопытство прежде возражения

- **Проблемная ситуация:** чужая идея кажется неясной/неверной, и первый импульс — оспорить или поставить диагноз.
- **Силы:** желание возразить vs ценность сначала понять; скрытая повестка vs подлинное любопытство.
- **Ход:** Core Protocols Investigate — «отстранённый, но увлечённый исследователь»: задавать только хорошо сформулированные информационные вопросы («Что в X делает тебя Y?», «Приведи конкретный пример», «Что одно ты хочешь больше всего, решив X?»), без мнений, без наводящих, без «почему». Если не можешь оставаться беспристрастным — остановись.
- **Последствия:** идеи раскрываются прежде, чем их зарубили; меньше ложных возражений в [P-06].
- **Trade-offs:** медленнее, чем сразу возразить; легко скатиться в замаскированную критику.
- **Пример:** Investigate как фаза уточняющих вопросов в IDM [P-06].
- **Пограничные случаи:** в агентном режиме = роль-«devil's advocate» (guardian) задаёт уточнения до objection-раунда.
- **Антипаттерны:** вопросы с зашитым ответом; «почему ты...» (ставит в защиту); диагноз вместо вопроса.
- **Связи:** питает [P-06] (clarifying questions), [P-05] (дивергенция).
- **Когда НЕ применять:** когда факт уже установлен и нужно действие, а не дальнейшее исследование.
- **SoTA-источник:** thecoreprotocols.org/protocols/investigate.html (2026-06-29).
- **Проверка качества (E.21):** до возражения автор идеи сказал «да, именно так» — значит, его поняли, а не приписали ему позицию.

### P-12: Дизайн по цели + закрытие петли

- **Проблемная ситуация:** сессия собрана «потому что встречаемся» без острой цели; в конце нет решений и фиксации — энергия рассеивается.
- **Силы:** удобный размытый повод vs специфичная спорная цель; усталость к финалу vs необходимость закрыть петлю.
- **Ход:** на входе — Priya Parker: цель как «вышибала», она специфична, уникальна и спорна, и служит фильтром решений (кого звать, какие правила, что в scope). На выходе — обязательное закрытие: Holacracy closing round / Core Protocols Check Out, фиксация решений и owners, замер ROTI.
- **Последствия:** меньше «совещаний ни о чём»; решения и обучение зафиксированы.
- **Trade-offs:** дисциплина проектирования до и фиксации после; соблазн пропустить закрытие «мы и так устали».
- **Пример:** Art of Gathering (purpose as bouncer); Core Team `/end-session` = структурное закрытие (handoff, ledger, git-verify).
- **Пограничные случаи:** для непрерывного агентного конвейера «закрытие» = handoff-артефакт между сессиями.
- **Антипаттерны:** цель «обсудить X» (не фильтрует ничего); сессия без closing round и без записи решений.
- **Связи:** парный к [P-01]; питает все решающие паттерны (без цели нет критерия выбора).
- **Когда НЕ применять:** спонтанный микро-синк на 2 минуты.
- **SoTA-источник:** Priya Parker, The Art of Gathering; holacracy.org closing round; ayeconference.com ROTI (2026-06-29).
- **Проверка качества (E.21):** в конце каждый может назвать решения и свой следующий шаг; ROTI собран; ни одно решение не «повисло» без owner.

### P-13: Защита от ложного консенсуса в многоагентной фасилитации

- **Проблемная ситуация:** роли-субагенты быстро «соглашаются», но согласие — артефакт sycophancy/копирования, а не проверки; единогласный ответ всё ещё неверен.
- **Силы:** скорость схождения vs её обманчивость; «все роли за» vs отсутствие реальной независимости мнений.
- **Ход:** не путать единогласие с правотой. Обеспечить **независимость** до агрегации (роли формируют вывод изолированно, P-09 fan-out), затем явный objection/devil's-advocate-раунд (guardian), сверка с evidence (A.10), а не с убедительностью. Помнить: выигрыш «дебатов» часто = эффект majority voting, не делиберации; убедительная ложь может перебить верный аргумент.
- **Последствия:** меньше confident-wrong решений; consensus-stopping не маскирует высокий error rate.
- **Trade-offs:** дороже (доп. раунды, независимые прогоны); медленнее.
- **Пример:** MAST-таксономия (Cemri et al.) — 14 режимов отказа, кластеры: System Design / Inter-Agent Misalignment / Task Verification; failure modes multi-agent debate (sycophantic switch, copying).
- **Пограничные случаи:** для низкорисковых обратимых решений доп. раунды избыточны — хватает Decider [P-02].
- **Антипаттерны:** «все роли согласились → значит верно»; оркестратор-аккумулятор контекста на 4+ ролях (переполнение окна → потеря верификации); отсутствие task-verification на стыках ролей (CHK-WIRE/CHK-ORPHAN — локальный аналог).
- **Связи:** надстройка над [P-02]/[P-06] для агентного контекста; стыкуется с [P-11].
- **Когда НЕ применять:** одиночная роль/детерминированная задача без делиберации.
- **SoTA-источник:** arxiv 2503.13657 (Why Do Multi-Agent LLM Systems Fail?, MAST); arxiv 2509.05396 (failure modes in multi-agent debate) (2026-06-29).
- **Проверка качества (E.21):** перед фиксацией единогласного решения проведён хотя бы один независимый objection-проход и сверка с фактами, а не с «как уверенно звучит».

## 4. Карта противоречий (между школами)

| Вопрос | Школа A | Школа B | Как выбирать в нашем контексте |
|--------|---------|---------|--------------------------------|
| Как принимать решение | **Decider** (Core Protocols): быстрый одновременный голос + Resolution | **IDM/consent** (Holacracy/Sociocracy): раунды реакций и валидных возражений | Обратимое операционное → Decider [P-02]; governance/высокая цена ошибки → IDM [P-06] |
| Согласие или согласованность | **Consensus**: все должны хотеть | **Consent**: нет обоснованного возражения, «можешь жить» | Дефолт — consent (быстрее, не требует совпадения предпочтений); consensus только для конституционных вопросов (values.md) |
| Кто пишет ground rules | **Schwarz**: фасилитатор задаёт явные 9 правил (группа свои могут конфликтовать с процессом) | **Самоорганизация/LS**: группа вырабатывает свои нормы | В Core Team правила — структурные гейты (заданы), не выводятся каждой сессией; ближе к Schwarz |
| Тон обратной связи | **Perfection Game**: только плюс + «до 10» | **Security/correctness-ревью** (OWASP/FC): прямо назвать дефект | Дизайн/идея → Perfection Game [P-04]; корректность/безопасность → прямой дефект-репорт (test-роль), §0 вне фокуса PG |
| Нейтральность фасилитатора | **IAF/Schwarz**: процесс, не контент; нейтрален | **North Star / ценностные сессии**: иногда фасилитатор НЕ нейтрален и обязан это назвать | Держать нейтральность по дефолту [P-10]; при контентной позиции — явно снять «шляпу» |
| Эмоции в процессе | **Core Protocols Check In**: эмоции — ядро (mad/sad/glad/afraid) | **Holacracy**: эмоции вне governance, только tension/данные | К человеку-Founder — Check In; к ролям-агентам — структурная часть (presence/tension), без эмоций |
| Ценность многоагентных дебатов | **Debate улучшает reasoning** (ранние MAD-работы) | **Choi et al.**: выигрыш = majority voting, не делиберация; дебаты усиливают bluster | Не полагаться на «дебаты ради истины»; добавлять independence + evidence-сверку [P-13] |
