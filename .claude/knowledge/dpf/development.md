# DPF: dev — TDD и функциональная ясность

> Предметный учебник для AI-агента (Domain Principles Framework). Идёт **после FPF**:
> FPF даёт общие принципы работы, DPF — SoTA-ходы, термины, ошибки и границы этого домена.
> Паттерны — декларативный метод мышления, **не** инструкция на выполнение (E.8).

## 0. Scope & Bounded Context (FPF A.1.1)

- **В фокусе:** Реализация программной логики через Red-Green-Refactor TDD цикл, написание тестов до кода, сохранение поведения при рефакторинге, fail-fast контракты на границах функций и модулей.
- **Вне фокуса:** Архитектурные решения (→ architect), выбор инфраструктуры/стека (→ cto), тест-план на уровне системы (→ test), UI/UX дизайн (→ designer), продуктовые метрики (→ product).
- **Перевод терминов:** «Тест» в этом контексте = unit/integration test, написанный самим dev до кода; не тест-план test-роли и не системный приёмочный тест.
- **Maturity:** refined — провенанс-пробелы закрыты (числа лейблованы, мёртвые источники удалены/процитированы, 2 отсутствующих источника добавлены, dead-ссылки устранены); 11 каналов Collection Protocol пройдено. Открытый пробел: London/Chicago в §4 оставлено намеренно неразрешённым (проектно-зависимо; нужен overlay на конкретный стек от architect).

## 1. Глоссарий домена

| RU | EN (для кода/поиска) | Определение | Не является |
|----|----------------------|-------------|-------------|
| Красный этап | Red phase | Написание падающего теста до существования реализации; тест должен падать по правильной причине (отсутствие кода, не синтаксическая ошибка) | Написание заглушки в коде перед тестом |
| Зелёный этап | Green phase | Написание минимального кода, чтобы тест прошёл; никакого кода сверх нужного | Полная правильная реализация «сразу» |
| Рефакторинг | Refactor | Улучшение структуры кода при неизменном поведении (тесты остаются зелёными) | Добавление функциональности, переработка API |
| Контракт функции | Function contract | Явный набор предусловий (preconditions) + постусловий (postconditions) + инвариантов (invariants) для функции или метода | Docstring-комментарий без машинной проверки |
| Предусловие | Precondition | Условие на аргументы/состояние, которое должно быть истинным перед вызовом функции; ответственность вызывающего кода | Валидация бизнес-правил внутри функции |
| Постусловие | Postcondition | Условие на возвращаемое значение/состояние, которое гарантируется после успешного выполнения функции | Логирование результата |
| Инвариант | Invariant | Условие, которое всегда истинно для объекта/модуля между вызовами публичных методов | Временное состояние внутри метода |
| Fail-fast | Fail-fast | Принцип: при нарушении контракта немедленно поднять исключение, не продолжать выполнение с невалидным состоянием | Перехват и логирование без re-raise |
| Заглушка | Stub | Тестовый дублёр, возвращающий фиксированные значения без логики | Мок с проверкой вызовов |
| Мок | Mock | Тестовый дублёр, также проверяющий, что к нему обратились нужным образом | Stub; реальная реализация |
| Лондонская школа | London school (mockist / outside-in) | TDD стиль, начинающий с внешнего API, активно использующий моки для описания взаимодействий между объектами | Классическая школа |
| Классическая школа | Classic school (Chicago / inside-out / Detroit) | TDD стиль, начинающий с доменной логики, предпочитающий реальные объекты вместо моков | Лондонская школа |
| Мутационное тестирование | Mutation testing | Оценка качества тестов путём инъекции искусственных дефектов (мутантов) в код; метрика — mutation score | Code coverage |
| Property-based testing | Property-based testing (PBT) | Тестирование через описание свойств (инвариантов), которые должны выполняться для всех (псевдослучайных) входных данных | Example-based testing (один сценарий — один тест) |
| Экстремальное программирование | Extreme Programming (XP) | Методология разработки, в которой TDD является обязательной практикой наряду с парным программированием и непрерывной интеграцией | Scrum, SAFe |

## 2. Паттерны (FPF E.8)

### P-01: Red-Green-Refactor как неделимый цикл

- **Проблемная ситуация:** Разработчик пишет код «сначала», откладывая тесты на потом или пропуская рефакторинг после прохождения тестов.
- **Силы:** Кажется быстрее сразу написать «правильный» код; тест воспринимается как лишняя работа; рефакторинг пугает регрессиями.
- **Ход (решение):** Строго чередовать три фазы в порядке Red → Green → Refactor. Red: написать один падающий тест, убедиться, что он падает по правильной причине. Green: написать минимально достаточный код, чтобы тест прошёл — не более. Refactor: улучшить структуру, не меняя поведения; тесты остаются зелёными на всём протяжении.
- **Последствия:** Тест всегда существует раньше кода → невозможно написать непроверяемый код; маленькие шаги снижают когнитивную нагрузку; встроенный regression guard.
- **Trade-offs:** Требует дисциплины; начальная скорость кажется ниже; плохо работает при нечётком требовании (нет понимания, что писать в тест).
- **Пример:** Kent Beck в «TDD by Example» реализует класс Money через цикл: тест `test_multiplication` → stub-реализация `times()` возвращает хардкод → рефакторинг к реальной формуле; каждый шаг ≤ 2-3 строки изменений.
- **Пограничные случаи:** Рефакторинг библиотеки без тестов (legacy) — нельзя начать с Red, нужно сначала покрыть тестами (→ P-08 Characterization Tests); очень маленький инкремент может казаться бессмысленным, но защищает от drift.
- **Антипаттерны:** «Test After» — тесты написаны после кода, не фиксируют замысел; «Скип Рефакторинга» — остановка на Green без очистки дублирования ведёт к техдолгу.
- **Связи:** стыкуется с [P-02] (Минимальный зелёный шаг), [P-03] (Один assert на тест), [P-05] (Рефакторинг без изменения поведения); противоречит [P-09 London school vs Chicago] по порядку построения слоёв.
- **Когда НЕ применять:** Спайк/прототип для исследования — здесь цель понять, а не строить; но результат спайка выбрасывается или переписывается по TDD.
- **SoTA-источник:** Beck, Kent. *Test-Driven Development: By Example*. Addison-Wesley, 2002, ch.1-3. Fowler, martinfowler.com/bliki/TestDrivenDevelopment.html (дата проверки 2026-06). Thoughtworks — XP and TDD: https://www.thoughtworks.com/insights/blog/testing/xp-tdd (дата проверки 2026-06).
- **Проверка качества (E.21):** Тест создан раньше кода (timestamp коммита или git log); после Green нет дублирования в коде; тест проходит и после рефакторинга.

### P-02: Минимально достаточная реализация (Minimum Green Step)

- **Проблемная ситуация:** В фазе Green разработчик пишет «правильную» полную реализацию сразу, потому что «и так очевидно».
- **Силы:** Желание сделать хорошо сразу; кажется, что маленький шаг — пустая трата времени.
- **Ход (решение):** Написать ровно столько кода, сколько нужно, чтобы текущий тест прошёл. Допустимо вернуть хардкод, если следующий тест заставит его обобщить. «Fake it till you make it» и «Triangulate».
- **Последствия:** Код растёт ровно под давлением тестов; исключает gold-plating; каждая строка кода обоснована провалившимся тестом.
- **Trade-offs:** На первых шагах код выглядит «глупо» (хардкод); требует доверия к процессу.
- **Пример:** Тест `assert adder(2, 3) == 5` → первая реализация `return 5` → второй тест `assert adder(1, 1) == 2` → вынуждает написать `return a + b`.
- **Пограничные случаи:** Когда правильная реализация очевидна и тривиальна — «Obvious Implementation» (допускает Beck): можно написать сразу, но при малейшем сомнении — вернуться к Fake It.
- **Антипаттерны:** «Решаем задачу, не ту, которую тест просит» — пишем общий случай, когда тест проверяет частный.
- **Связи:** стыкуется с [P-01] (цикл RGR), [P-04] (Fail-Fast на входах — его добавляем отдельным Red шагом).
- **Когда НЕ применять:** Когда реализация тривиально однозначна и команда уверена; тогда «Obvious Implementation» — допустимое ускорение.
- **SoTA-источник:** Beck, Kent. *TDD by Example*, ch.2 (Fake It / Obvious Implementation / Triangulate), 2002.
- **Проверка качества (E.21):** Между каждым Red и Green — только минимальное изменение (diff ≤ 10-15 строк [hypothesis]); ни один метод не реализует функциональность, непроверенную тестом.

### P-03: Один тест — одно поведение

- **Проблемная ситуация:** Тест проверяет несколько независимых поведений, что делает диагностику провала сложной.
- **Силы:** Соблазн проверить «всё сразу» в одном тест-методе; копипаста; желание сэкономить на setup.
- **Ход (решение):** Каждый тест-метод проверяет ровно одно поведение и содержит один логический assert (или несколько assert, неразделимо связанных с одним поведением). Описание теста в имени метода — полное предложение: `test_returns_zero_for_empty_list`.
- **Последствия:** При провале сразу ясно, что сломано; тесты становятся живой документацией поведения.
- **Trade-offs:** Больше тест-методов; больше кода setup (решается fixtures/factories); имена тестов длиннее.
- **Пример:** Вместо `test_user()` с 15 assert-ами: `test_user_created_with_default_role()`, `test_user_raises_on_empty_name()`, `test_user_email_normalized_to_lowercase()`.
- **Пограничные случаи:** Тест на исключение — один assert на тип исключения и отдельный тест на сообщение; составные инварианты (все поля объекта корректны) — допускается несколько assert в одном тесте, если они описывают один неразделимый контракт.
- **Антипаттерны:** «The Giant» — один тест с 15+ assert; «The Mockery» — тест проверяет, что методы вызваны, а не что поведение корректно (см. §3).
- **Связи:** стыкуется с [P-01], [P-06] (Test as Documentation); противоречит [P-09] по уровню изоляции.
- **Когда НЕ применять:** Интеграционный тест «happy path» оправданно проверяет весь flow — но это не unit-тест.
- **SoTA-источник:** Martin, Robert C. *Clean Code*, ch.9 (Unit Tests / F.I.R.S.T.), 2008. Codurance TDD anti-patterns ch.1 (The Giant), 2023.
- **Проверка качества (E.21):** В тест-методе не более одного act-секции; имя метода читается как утверждение о поведении.

### P-04: Fail-Fast на границах контракта

- **Проблемная ситуация:** Функция получает невалидные аргументы, продолжает выполнение, и ошибка всплывает в совершенно другом месте — трассировка указывает не туда.
- **Силы:** Проверки кажутся лишними (вызывающий код «должен» передавать правильные данные); оборонительное программирование выглядит паранойей.
- **Ход (решение):** Валидировать предусловия (preconditions) в начале каждой публичной функции. При нарушении — поднять специфическое исключение с содержательным сообщением немедленно. Не продолжать выполнение с невалидным состоянием. Для stateful систем — завершить операцию/транзакцию, а не продолжать.
- **Последствия:** Точная диагностика (ошибка возникает у источника, а не следствия); защита целостности данных; короткий feedback loop.
- **Trade-offs:** Небольшой overhead на проверках в hot path; может чрезмерно защищать «от дурака» внутренние функции (оправданно только для публичного API).
- **Пример:** Вместо `def divide(a, b): return a / b` → `def divide(a: float, b: float) -> float: if b == 0: raise ZeroDivisionError("divide: b must be non-zero"); return a / b`.
- **Пограничные случаи:** Внутренние приватные функции — fail-fast обязателен только если они вызываются из публичного контекста или функция большая; конфигурация приложения — проверять на старте (eager validation), не при первом использовании.
- **Антипаттерны:** «Fail-silent» — `except Exception: logger.log(e)` без re-raise; «Return None при ошибке» — маскирует проблему, код продолжает с None; «Blanket catch» — `except Exception: pass`.
- **Связи:** стыкуется с [P-02] (отдельный Red тест на каждое предусловие), [P-07] (иерархия исключений); принцип из FC skill (fail-fast, явная обработка ошибок).
- **Когда НЕ применять:** Внутренние вспомогательные функции, вызываемые только одним местом в контролируемом контексте — избыточно; performance-критичный hot path с уже валидированными данными.
- **SoTA-источник:** Meyer, Bertrand. *Design by Contract*. ETH/Eiffel, 1992. Enterprisecraftsmanship.com — Fail-Fast Principle (дата проверки 2026-06).
- **Проверка качества (E.21):** При передаче None/пустой строки/неверного типа — исключение поднимается в первой публичной функции; tracebacks в тестах указывают на источник, а не следствие.

### P-05: Рефакторинг без изменения поведения (Behavior-Preserving Refactor)

- **Проблемная ситуация:** Разработчик улучшает структуру кода и одновременно меняет поведение; или рефакторинг делается без тестового прикрытия.
- **Силы:** Соблазн «заодно исправить» баг или добавить фичу; рефакторинг без тестов кажется нормальным, потому что «код тот же».
- **Ход (решение):** Рефакторинг — отдельная, изолированная фаза после Green. Никакого нового поведения, никаких новых тестов в этой фазе. Использовать каталог Fowler (Extract Function, Inline Variable, Replace Conditional with Polymorphism и др.) как атомарные безопасные шаги. Тесты остаются зелёными на всём протяжении.
- **Последствия:** Тесты служат safety net; рефакторинг становится безопасным и регулярным; техдолг снижается итеративно.
- **Trade-offs:** Без тестового прикрытия рефакторинг рискован; медленная работа до наработки каталога атомарных рефакторингов.
- **Пример:** Fowler, *Refactoring* 2nd Ed.: Extract Function — выделить клок кода в именованную функцию, убедиться, что тесты проходят; Decompose Conditional — разбить сложный if на именованные функции.
- **Пограничные случаи:** Legacy-код без тестов — нельзя рефакторить без Characterization Tests (→ P-08); изменение публичного API (сигнатуры) — это изменение контракта, не рефакторинг.
- **Антипаттерны:** «Рефакторинг + новая фича за один коммит» — смешивает изменение поведения и структуры, неотделимо в git history; «Рефакторинг в Red фазе» — нельзя.
- **Связи:** стыкуется с [P-01], [P-08] (для legacy); каталог Fowler — конкретные атомарные ходы.
- **Когда НЕ применять:** При отсутствии тестового прикрытия — сначала P-08; когда изменение структуры неотделимо от изменения поведения — это фича, а не рефакторинг.
- **SoTA-источник:** Fowler, Martin. *Refactoring: Improving the Design of Existing Code*, 2nd Ed. Addison-Wesley, 2018. refactoring.com/catalog/ (Extract Function, Decompose Conditional).
- **Проверка качества (E.21):** Diff рефакторинг-коммита не добавляет новых тестов; все тесты зелёные до и после; git log разделяет рефакторинги и фичи.

### P-06: Тест как спецификация (Test as Executable Specification)

- **Проблемная ситуация:** Требования существуют только в Jira/Confluence, тесты — «ради coverage», код расходится с намерением.
- **Силы:** Тест воспринимается как проверка, а не документация; имена тестов невыразительны; тесты копируются без осмысления.
- **Ход (решение):** Писать тест как утверждение о поведении системы с точки зрения пользователя/потребителя API. Имя теста — полное предложение в формате `given_<context>_when_<action>_then_<outcome>` или аналоге. Тест-файл читается как спецификация компонента.
- **Последствия:** Тесты становятся живой документацией; при изменении поведения тест-имена указывают на нарушенные спецификации; упрощается онбординг.
- **Trade-offs:** Длинные имена тестов; необходимость поддерживать читаемость при рефакторинге.
- **Пример:** `def test_order_total_includes_tax_for_us_customers(): ...` вместо `def test_order()`.
- **Пограничные случаи:** BDD (Gherkin/pytest-bdd) — формализованная версия этого паттерна для приёмочных тестов; может быть избыточен для мелких утилитных функций.
- **Антипаттерны:** «The Liar» — тест с выразительным именем, но проверяющий другое; тесты без имён (`test_1`, `test_2`).
- **Связи:** стыкуется с [P-03] (один тест — одно поведение); расширяется в BDD через pytest-bdd.
- **Когда НЕ применять:** Тест на внутреннюю деталь реализации (которую не стоит документировать) — здесь лучше не тест, а рефакторинг к более ясному коду.
- **SoTA-источник:** Freeman, Steve; Pryce, Nat. *Growing Object-Oriented Software, Guided by Tests*, ch.2 (A First Test), Addison-Wesley, 2009. arXiv:2411.04141 — A Comparative Study on TDD and BDD, 2024 (сравнение BDD как формализации acceptance-тестов поверх TDD).
- **Проверка качества (E.21):** Новый разработчик читает тест-файл и понимает поведение модуля без чтения реализации.

### P-07: Иерархия специфических исключений

- **Проблемная ситуация:** Все ошибки выбрасываются как `ValueError` или `Exception`, невозможно обрабатывать разные виды ошибок по-разному.
- **Силы:** Проще использовать встроенные типы; создание иерархии кажется overhead.
- **Ход (решение):** Создать иерархию исключений домена: базовый класс для модуля/компонента, специфические потомки для каждой категории ошибки. Имена содержательны и описывают проблему. При wrapping исключений сохранять original cause через `raise DomainError("...") from original_exc`.
- **Последствия:** Вызывающий код может перехватывать точно нужный тип; диагностика по имени класса; тесты проверяют конкретный тип исключения.
- **Trade-offs:** Небольшой объём boilerplate при создании иерархии; нужно поддерживать в актуальном состоянии.
- **Пример:** `class PaymentError(DomainError): pass` → `class InsufficientFundsError(PaymentError): pass` → `class PaymentGatewayError(PaymentError): pass`. Тест: `with pytest.raises(InsufficientFundsError): process_payment(amount=10000)`.
- **Пограничные случаи:** Иерархия слишком глубокая (>3 уровней) — запах избыточной сложности; иерархия слишком плоская — нет различений.
- **Антипаттерны:** `except Exception: pass` — blanket catch без re-raise; одноимённые ошибки для разных ситуаций; `raise Exception("something went wrong")` без контекста.
- **Связи:** стыкуется с [P-04] (Fail-Fast); FC skill принцип «явная обработка ошибок».
- **Когда НЕ применять:** Библиотечный код без специфической семантики ошибок — допустимо использовать стандартные `ValueError`/`TypeError`.
- **SoTA-источник:** Martin, Robert C. *Clean Code*, ch.7 (Error Handling), 2008. FC skill principles.md (принцип 3 — явная обработка ошибок).
- **Проверка качества (E.21):** В тесте `pytest.raises(SpecificError)` — не `Exception`; `except SomeSpecificError` в продакшн-коде с конкретной обработкой.

### P-08: Characterization Tests для Legacy-кода

- **Проблемная ситуация:** Нужно изменить код без тестового прикрытия; рефакторинг рискован; поведение неизвестно.
- **Силы:** Legacy-код «работает», никто не решается его трогать; добавление тестов требует понимания кода; страх что-то сломать.
- **Ход (решение):** Написать «characterization tests» — тесты, фиксирующие текущее поведение (даже если оно неверное). Запустить код, посмотреть реальный вывод, написать тест на этот вывод. Только после создания safety net — рефакторинг (→ P-05).
- **Последствия:** Safety net создан; можно безопасно рефакторить; поведение задокументировано; баги зафиксированы как «ожидаемое поведение» до решения об их исправлении.
- **Trade-offs:** Тесты могут закреплять баги; требует времени; не документирует намерение, только поведение.
- **Пример:** Feathers, *Working Effectively with Legacy Code*, ch.13: wrapping функции в тест, принимающий реальный вывод как ожидаемый.
- **Пограничные случаи:** Неопределённое поведение (randomness, time) — нужно изолировать через dependency injection перед созданием characterization test.
- **Антипаттерны:** «Рефакторинг без characterization tests» — нарушение Code-Change Discipline (FPF A.7); рефакторинг legacy с only мануальным тестированием.
- **Связи:** стыкуется с [P-05] (рефакторинг после); предваряет любой [P-01] цикл на legacy коде.
- **Когда НЕ применять:** Код, который будет удалён, а не изменён — нет смысла покрывать.
- **SoTA-источник:** Feathers, Michael. *Working Effectively with Legacy Code*, ch.13 (I Need to Make a Change but I Don't Know What Tests to Write). Prentice Hall, 2004.
- **Проверка качества (E.21):** Тест-файл содержит комментарий `# characterization test: documents current behavior, not intended behavior`; при изменении поведения тест намеренно обновляется.

### P-09: Выбор школы TDD под задачу (London vs Chicago)

- **Проблемная ситуация:** Разработчик применяет один стиль TDD (только моки или никаких моков) ко всем задачам.
- **Силы:** Проще придерживаться одной техники; школы представлены как конкурирующие.
- **Ход (решение):** Выбирать стиль по характеру задачи: **London/Outside-in** — когда дизайн взаимодействий между объектами неизвестен, нужно проектировать интерфейсы «снаружи»; моки как инструмент дизайна. **Chicago/Inside-out** — когда доменная логика хорошо понята, зависимости простые; тестировать реальные объекты. Использовать оба как взаимодополняющие инструменты.
- **Последствия:** Моки применяются там, где они дают дизайн-сигнал (London), а не везде автоматически; тесты менее хрупки.
- **Trade-offs:** Требует опыта для распознавания подходящей ситуации; вначале дезориентирует.
- **Пример:** London: новый сервис с неизвестными зависимостями → мок зависимостей, проектируем интерфейс через тест. Chicago: расчёт налогов (чистая функция) → никаких моков, только входы и выходы.
- **Пограничные случаи:** Моки для внутренних деталей (не внешних зависимостей) → anti-pattern «Mockery»; отсутствие моков для external API → хрупкие тесты, зависящие от сети.
- **Антипаттерны:** «Mockery» — мок на всё подряд, тест проверяет реализацию, а не поведение; «Интеграция без изоляции» — тест зависит от реальной БД, сети.
- **Связи:** стыкуется с [P-03] (изоляция), [P-04] (контракты — то, что мок должен соблюдать); §4 Карта противоречий.
- **Когда НЕ применять:** Чистые функции без зависимостей — моки не нужны никогда.
- **SoTA-источник:** Freeman, Pryce. *GOOS*, 2009 (London). Beck. *TDD by Example*, 2002 (Chicago). Reales, Jose Maria Valera. «London vs Chicago — It's an integration, not a choice». Medium, 2023. https://chemaclass.medium.com/london-vs-chicago-6823bbb2fa40 (дата проверки 2026-06).
- **Проверка качества (E.21):** Моки применяются только к внешним зависимостям (БД, HTTP, файловая система); тест не проверяет порядок внутренних вызовов.

### P-10: Property-Based Testing для инвариантов домена

- **Проблемная ситуация:** Example-based тесты покрывают только продуманные разработчиком случаи; edge cases пропускаются; инварианты не проверяются систематически.
- **Силы:** Описание свойств сложнее, чем написание примеров; непривычный инструмент.
- **Ход (решение):** Использовать Hypothesis (Python) / QuickCheck (Haskell) / fast-check (JS) для тестирования инвариантов и свойств, которые должны выполняться для всех входных данных. Идентифицировать свойства: «encode(decode(x)) == x», «сортировка идемпотентна», «результат не зависит от порядка коммутативных операций». Запускать параллельно с example-based тестами.
- **Последствия:** Фреймворк находит edge cases, которые человек не придумал бы; shrinking автоматически минимизирует провальный input; тест выражает общий закон, а не конкретный пример.
- **Trade-offs:** Сложнее идентифицировать свойства (требует понимания домена); медленнее на больших property-space; не заменяет example-based, дополняет.
- **Пример:** `@given(st.lists(st.integers())) def test_sort_is_idempotent(lst): assert sorted(sorted(lst)) == sorted(lst)`. Hypothesis нашёл баг в Python stdlib через эту технику.
- **Пограничные случаи:** Stateful системы — Hypothesis stateful testing (RuleBasedStateMachine); слишком широкие свойства дают ложную уверенность.
- **Антипаттерны:** Использовать PBT только для randomized smoke-тестов без формулировки свойства; игнорировать провальный shrunk-пример.
- **Связи:** стыкуется с [P-04] (инварианты = постусловия), [P-06] (спецификация через свойства).
- **Когда НЕ применять:** UI-поведение, неопределённые бизнес-правила, где «правильный» ответ неизвестен без оракула.
- **SoTA-источник:** hypothesis.works — What is Property-Based Testing (дата проверки 2026-06). arXiv:2510.09907 — Agentic Property-Based Testing: Finding Bugs Across the Python Ecosystem, 2025. https://arxiv.org/abs/2510.09907
- **Проверка качества (E.21):** Hypothesis выполняет ≥100 примеров за прогон (default `max_examples=100`, hypothesis.readthedocs.io/en/latest/settings.html); при провале shrunk-пример добавляется как фиксированный example-тест.

### P-11: Mutation Testing как метрика качества тестов

- **Проблемная ситуация:** Code coverage 90%+ есть, но тесты «зелёные» и на сломанном коде — «coverage theater».
- **Силы:** Mutation testing медленнее обычных тестов; инструменты менее известны; результаты сложнее интерпретировать.
- **Ход (решение):** Запускать mutmut (Python) / Stryker (JS/Java) периодически (в CI на PR или еженедельно). Mutation score < 70% [vendor-claim: Codecov Blog] → тесты не проверяют поведение. Целевой score: 80-85%+ [vendor-claim: Codecov Blog] для критической логики. Выжившие мутанты — кандидаты на новые тесты.
- **Последствия:** Объективная метрика качества тестов, независимая от coverage; обнаруживает «тесты ради тестов».
- **Trade-offs:** В 3-10 раз медленнее обычного CI [hypothesis: зависит от инструмента и размера кодовой базы]; false-positive эквивалентные мутанты; требует настройки scope (не запускать на всей базе кода).
- **Пример:** Mutant меняет `>` на `>=` в валидации возраста — если тест не падает, условная граница не протестирована.
- **Пограничные случаи:** Эквивалентные мутанты (семантически идентичный код) — ложные выжившие, нужно игнорировать вручную; I/O-код сложно тестировать на мутантах.
- **Антипаттерны:** Ориентироваться только на code coverage без mutation score; запускать mutation testing на всём коде без scope-ограничений.
- **Связи:** дополняет [P-01] (качество написанных тестов), [P-03] (один assert = один детектор мутанта).
- **Когда НЕ применять:** Прототипы, throw-away код; очень медленные интеграционные тесты.
- **SoTA-источник:** arXiv:2307.11534 — TDD maturity model through anti-patterns framework, 2023. Codecov Blog — Mutation Testing: How to Ensure Code Coverage Isn't a Vanity Metric. https://about.codecov.io/blog/mutation-testing-how-to-ensure-code-coverage-isnt-a-vanity-metric/ (дата проверки 2026-06).
- **Проверка качества (E.21):** После добавления новых тестов mutation score ≥ 80% [vendor-claim: Codecov Blog] на целевом модуле; «выжившие» мутанты задокументированы как accepted-risk или покрыты новым тестом.

### P-12: Структура теста AAA (Arrange-Act-Assert)

- **Проблемная ситуация:** Тест-код перемешивает setup, выполнение и проверку, становится нечитаемым.
- **Силы:** Кажется проще писать линейно; для простых тестов структура кажется избыточной.
- **Ход (решение):** Явно разделять тест на три секции через пустые строки или комментарии: **Arrange** (создать объекты, настроить состояние), **Act** (вызвать тестируемое поведение — одна строка), **Assert** (проверить результат). Секция Act — всегда одна строка.
- **Последствия:** Тест читается как история: «дано — когда — тогда»; при провале сразу ясно, где проблема; структура совпадает с Given-When-Then BDD.
- **Trade-offs:** Verbose для простейших тестов; fixtures могут скрывать Arrange.
- **Пример:** `def test_discount_applied_for_premium_user(): user = User(tier="premium"); order = Order(user=user, total=100); result = calculate_discount(order); assert result == 20`.
- **Пограничные случаи:** Тест на исключение — Act находится внутри `with pytest.raises(...)`, Assert — проверка типа и сообщения исключения.
- **Антипаттерны:** «The Liar» — Act и Assert перемешаны; несколько Act в одном тесте (→ The Giant).
- **Связи:** стыкуется с [P-03], [P-06]; Given-When-Then = BDD-аналог AAA.
- **Когда НЕ применять:** Параметризованные тесты с `@pytest.mark.parametrize` — AAA неявно, но присутствует.
- **SoTA-источник:** Martin, Robert C. *Clean Code*, ch.9 (Unit Tests), 2008. GDS Way TDD guide (дата проверки 2026-06).
- **Проверка качества (E.21):** В каждом тест-методе ровно одна строка Act; секции визуально разделены.

## 3. Каталог антипаттернов (сквозной)

| Антипаттерн | Симптом | Почему вреден | Чем заменить (P-NN) |
|-------------|---------|---------------|---------------------|
| **Test After** | Тесты пишутся после кода, не до | Тест адаптируется под реализацию, а не описывает требование; design-feedback от TDD теряется | [P-01] Red-Green-Refactor |
| **The Giant** | Один тест с 15+ assert-ами; тестирует несколько поведений | При провале непонятно что именно сломалось; тест = «God class» | [P-03] Один тест — одно поведение |
| **The Liar** | Тест проходит, но не проверяет реальное поведение (async без await, time-dependent без mock) | Ложная уверенность; реальные баги не обнаруживаются | [P-12] AAA структура; [P-04] Fail-Fast |
| **The Slow Poke** | Тест-сьют выполняется минуты/часы | Разработчики перестают запускать тесты; TDD-цикл разрушен | Изолировать I/O через [P-09] (моки для внешних зависимостей) |
| **Excessive Setup** | Нужно создать 10 объектов перед тестом | Код не проектирован для тестируемости; сигнал о нарушении SRP | [P-09] правильный scope моков; рефакторинг зависимостей |
| **The Mockery** | Всё замокано, тест проверяет «был ли вызван метод», не поведение | Тест хрупкий; привязан к реализации; не обнаруживает логических ошибок | [P-09] моки только для внешних зависимостей; Chicago school для domain |
| **Fail-Silent** | `except Exception: logger.log(e)` без re-raise | Ошибка поглощена; система продолжает с невалидным состоянием; данные повреждены | [P-04] Fail-Fast; [P-07] иерархия исключений |
| **Coverage Theater** | Coverage 90%+ но тесты «зелёные» на сломанном коде | Метрика оптимизирована, а не качество; false confidence | [P-11] Mutation Testing |
| **Рефакторинг + фича в одном коммите** | Diff содержит и структурные изменения, и новое поведение | Невозможно bisect; review сложен; откат ломает всё | [P-05] изолированный рефакторинг-коммит |
| **Без Characterization Tests** | Рефакторинг legacy без safety net | Непредсказуемые регрессии; нарушение Code-Change Discipline (FPF A.7) | [P-08] Characterization Tests |
| **Return None при ошибке** | Функция возвращает None вместо исключения | Error hiding; ошибка всплывёт в другом месте как NullPointerException | [P-04] Fail-Fast; [P-07] специфические исключения |
| **Ловля Exception без re-raise** | `except Exception: pass` или без `raise` | Полное поглощение ошибки; бесследная потеря информации о состоянии | [P-04] + [P-07] |

*Источники: Codurance TDD anti-patterns ch.1/ch.2, 2023; arXiv:2307.11534, 2023; Martin *Clean Code* ch.7, 2008; enterprisecraftsmanship.com Fail-Fast.*

## 4. Карта противоречий (между школами)

| Вопрос | Школа A | Школа B | Как выбирать в контексте Core Team |
|--------|---------|---------|-------------------------------------|
| **Когда писать тест** | TDD-пуристы (Beck, Uncle Bob): тест всегда первый, нет исключений | Прагматики: иногда полезен «спайк» без тестов, потом выбрасывается | Спайк = throw-away; прод-код — только через TDD [P-01] |
| **Моки vs реальные объекты** | London school (Freeman/Pryce): активное использование моков как дизайн-инструмент; outside-in | Chicago school (Beck, Uncle Bob): предпочесть реальные объекты; мок = крайняя мера | Моки для внешних зависимостей (I/O, сеть) — всегда; для доменной логики — Chicago [P-09] |
| **Порядок построения** | Outside-in: начать с acceptance test → unit tests снизу | Inside-out: начать с доменной логики, потом интеграция | **Неразрешено (проектно-зависимо):** для новых сервисов склонение к London; для чистой логики — Chicago. Нужен overlay от architect на конкретный стек [P-09] |
| **Публичный API или реализация** | Тестировать только публичный API (классики) | Тестировать взаимодействия между объектами (mockists) | Тестировать поведение (публичный API); моки только для external boundary [P-03, P-09] |
| **Размер теста** | Google: Small/Medium/Large (Test Sizes, 2010); SMURF (2024) заменяет пирамиду | Classically: unit/integration/e2e пирамида | Для Core Team Python: Small=unit (без I/O), Medium=integration (один реальный сервис), Large=e2e |
| **TDD при нечётких требованиях** | TDD-пуристы: нельзя писать тест без спецификации | BDD-подход: спецификация = acceptance test в Gherkin (arXiv:2411.04141) | При нечётком требовании — сначала [P-06] Clarity Protocol, потом TDD |
| **Fail-Fast vs Defensive** | Fail-fast (Meyer, enterprisecraftsmanship): падать немедленно | Defensive programming (cloud era): предусмотреть degraded mode, retry | Fail-fast для доменной логики [P-04]; retry/circuit-breaker для внешних I/O |
