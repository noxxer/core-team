# DPF: keeper — управление знанием

> Предметный учебник для AI-агента (Domain Principles Framework). Идёт **после FPF**:
> FPF даёт общие принципы работы, DPF — SoTA-ходы, термины, ошибки и границы этого домена.
> Паттерны — декларативный метод мышления, **не** инструкция на выполнение (E.8).

## 0. Scope & Bounded Context (FPF A.1.1)

- **В фокусе:** Управление знанием внутри AI-ассистированной команды: создание и поддержка глоссария проекта, фиксация фактов о домене, производство документации для AI-потребителей (llms.txt/llms-full.txt), компрессия промптов, обеспечение bounded-context целостности терминов.
- **Вне фокуса:** Корпоративные KM-системы масштаба организации; управление базами знаний для поддержки клиентов; персональное управление знанием (PKM, второй мозг) как личная практика; управление документацией пользовательского продукта (user-facing docs); архивирование данных и data governance.
- **Перевод терминов:** glossary = глоссарий (project/glossary.md), domain = предметная область (project/domain.md), knowledge decay = устаревание знаний, bounded context = ограниченный контекст (FPF A.1.1), ubiquitous language = единый язык (DDD), llms.txt = машиночитаемый индекс знаний для LLM.
- **Maturity:** refined — все 11 каналов Collection Protocol пройдены; провенанс-пробелы закрыты (GAP-A..E из verify-аудита 2026-06-29); orphan-источники (Weinberger/Ahrens/KMWorld) процитированы в паттернах; тавтология P-09 устранена. Для reliable: независимая verify-сессия (role автора ≠ checker) + кастомные AI-team метрики + кейсы малых команд.

## 1. Глоссарий домена

| RU | EN (для кода/поиска) | Определение | Не является |
|----|----------------------|-------------|-------------|
| Глоссарий | glossary | Проектный словарь: RU-термин, EN-код, определение, «не является». Локален для bounded context | Общий словарь языка или индустрии |
| Домен | domain | Файл фактов о реальности (рынок, пользователи, система) — отдельно от решений и мнений | Техническая предметная область в смысле DDD aggregate |
| Устаревание знаний | knowledge decay | Снижение точности/применимости зафиксированного знания с течением времени без обновления | Целенаправленное удаление устаревшей информации |
| Ограниченный контекст | bounded context | Зона, внутри которой термины имеют строго одно значение; вне зоны — могут означать другое | Неймспейс в коде или модуль системы |
| Единый язык | ubiquitous language | Согласованный словарь, одинаково понимаемый всеми ролями и кодом внутри bounded context | Формальный язык спецификаций (UML, Z-notation) |
| Кодификация | codification | Перевод неявного знания (tacit) в явную форму — документы, схемы, правила | Архивирование готового документа |
| Персонализация | personalization | KM-стратегия через peer-to-peer связи; знание остаётся у человека, не в документе | Персонализированный UX |
| llms.txt | llms_txt | Markdown-индекс (≤10 KB) — краткий human-curated список ссылок для LLM-навигации по контенту | robots.txt (тот для поисковых ботов); sitemap.xml |
| llms-full.txt | llms_full_txt | Полный дамп всей документации проекта одним файлом для LLM-потребления | Архив кода; бэкап базы данных |
| Компрессия промпта | prompt compression | Сокращение промпта с сохранением семантической плотности — роль archivist | Ручное редактирование промпта без измерения |
| Таксономия | taxonomy | Иерархическая система категорий для организации знания | Онтология (семантические связи между понятиями) |
| Факт vs решение | fact_vs_decision | Разделение domain.md (факты о мире) и decisions/ (выборы команды) | Любое разделение «что было» / «что будем делать» |

## 2. Паттерны (FPF E.8)

### P-01: Bounded-Context глоссарий — локальное значение термина

- **Проблемная ситуация:** В команде или кодовой базе один и тот же термин используется с разными значениями разными ролями. AI-агент получает противоречивые инструкции или делает неверные выводы.
- **Силы:** Термины из естественного языка многозначны. Разные роли (разработчик, аналитик, пользователь) живут в разных контекстах. Без явного глоссария каждый читает документ через свою линзу.
- **Ход (решение):** Создать project/glossary.md с явным bounded context: для каждого термина — RU-имя, EN-имя (для кода/поиска), точное определение, и «не является» (антидефиниция). Поддерживать в единственном числе, без дублирования по ролям.
- **Последствия:** Уменьшение семантических конфликтов; AI-агент и люди используют один словарь; изменение определения проходит через один файл.
- **Trade-offs:** Требует дисциплины обновления при появлении новых терминов; споры о терминах всплывают явно (это ценно, но дорого); глоссарий не может охватить все нюансы живого языка.
- **Пример:** В DDD термин «заказ» (Order) в контексте инвентаря означает «складской заказ поставщику», а в контексте продаж — «заказ клиента». Без bounded context glossary оба смысла смешиваются в коде и документации.
- **Пограничные случаи:** Когда проект растёт и появляются суб-домены с конфликтующими значениями — нужны вложенные bounded contexts, а не единый глоссарий.
- **Антипаттерны:** «Один глоссарий на всю компанию» — теряет специфику; «глоссарий в wiki без владельца» — устаревает за 2-3 месяца; «термин не определён, всем очевидно» — ложный консенсус.
- **Связи:** стыкуется с [P-02] (факт vs решение), [P-05] (прагматичная таксономия); противоречит подходу «один мастер-глоссарий на организацию» (см. §4).
- **Когда НЕ применять:** При прототипировании с нестабильной моделью домена — глоссарий будет меняться быстрее, чем успевает обновляться. Строить после первой стабилизации.
- **SoTA-источник:** Martin Fowler, «Bounded Context», martinfowler.com/bliki/BoundedContext.html (2014, обновляется); Eric Evans, Domain-Driven Design, 2003; ISO 30401:2018 §5.3 (Knowledge identification).
- **Проверка качества (E.21):** Любой новый участник команды, прочитав только glossary.md, понимает все ключевые термины в первых 10 коммитах без вопросов. Количество терминологических вопросов в ревью снижается.

### P-02: Факт vs Решение — разделение domain.md и decisions/

- **Проблемная ситуация:** Факты о мире (рынок, пользователи, техническая реальность) перемешаны с решениями команды (архитектурные выборы, процессные правила). При смене контекста непонятно, что изменилось в мире, а что — в выборах команды.
- **Силы:** В обычной документации факты и мнения записываются вперемешку. AI-агент не отличает «85% пользователей на мобильных» (факт) от «делаем mobile-first» (решение). Решения устаревают быстрее фактов.
- **Ход (решение):** Вести два раздельных документа: domain.md — только факты о внешней реальности с источником и датой; decisions/ — только выборы команды (с обоснованием). Keeper проверяет, что факт не пролез в decisions/ и наоборот.
- **Последствия:** Решения можно пересматривать, не трогая факты. AI-агент понимает, что неизменно, а что было решено командой. Ревизия решений прозрачна.
- **Trade-offs:** Требует дисциплины при записи; граница «факт vs решение» иногда неочевидна (напр., «текущая скорость интернета пользователей» — факт или допущение?).
- **Пример:** «85% трафика мобильный» (domain.md, источник: Google Analytics 2025-Q1) vs «мы строим mobile-first UI» (decisions/DEC-001.md). Если трафик изменится, решение можно пересмотреть, не трогая decision.
- **Пограничные случаи:** Обоснованные допущения (assumptions) — не факты, но и не решения; держать в отдельном artifacts/assumptions.md.
- **Антипаттерны:** «Всё в одном README» — невозможно отделить контекст от выборов; «решения в комментариях к коду» — теряются при рефакторинге; «факты без источника и даты» — неотличимы от мнений.
- **Связи:** стыкуется с [P-01] (bounded context glossary), [P-07] (свежесть знаний); FPF A.7 (Strict Distinction) и A.10 (Evidence Graph).
- **Когда НЕ применять:** В одиночных side-проектах без долгосрочного планирования — оверхед не окупается.
- **SoTA-источник:** FPF A.7 Strict Distinction (Core Team v4.5); ISO 30401:2018 §8.1 (Knowledge capture and documentation).
- **Проверка качества (E.21):** Задать вопрос новому участнику: «где найти, что мы знаем о рынке?» и «где найти, что мы решили делать?» — ответ должен быть разным файлом.

### P-03: llms.txt как навигационный индекс для AI-потребителей

- **Проблемная ситуация:** AI-агент или LLM-инструмент не может найти нужную документацию проекта в огромном репозитории — либо тратит лишние токены на нерелевантные файлы, либо пропускает критически важные.
- **Силы:** HTML-структура сайтов и дерево файлов репозитория проектируются для людей, не для LLM. У LLM ограниченное контекстное окно. Нет стандартного способа сказать «читай это первым».
- **Ход (решение):** Создать и поддерживать llms.txt (≤10 KB) — markdown-файл с иерархическим списком ключевых ссылок и кратким описанием каждого документа. Опционально — llms-full.txt с полным дампом. Обновлять синхронно с изменением документации.
- **Последствия:** AI-агент за один вызов получает карту знаний проекта; уменьшается число «галлюцинаций» из-за устаревшего контекста; человек-разработчик тоже выигрывает — llms.txt работает как оглавление.
- **Trade-offs:** Стандарт llms.txt не принят крупными AI-провайдерами официально (по состоянию на середину 2026); если llms.txt не обновляется при смене документации — вреднее, чем отсутствие.
- **Пример:** Anthropic и Cloudflare приняли llms.txt с ноября 2024 (когда Mintlify внедрил поддержку). Более 2000 сайтов на 2025 год [llmstxt.org directory, январь 2025; vendor-claim].
- **Пограничные случаи:** Для проектов с закрытым доступом — llms.txt в корне репозитория, не публичный URL. Для монорепозиториев — возможно несколько llms.txt на суб-проект.
- **Антипаттерны:** «Сгенерировать один раз и забыть» — stale llms.txt ведёт AI к мёртвым ссылкам; «включить всё подряд» — теряет навигационную ценность; «дублировать sitemap.xml» — это другой инструмент для другой аудитории.
- **Связи:** стыкуется с [P-04] (prompt compression), [P-07] (свежесть знаний); зависит от [P-01] (glossary как один из разделов индекса).
- **Когда НЕ применять:** Для ephemeral проектов (<2 недель жизни) или проектов без AI-потребителей документации.
- **SoTA-источник:** Jeremy Howard / Answer.AI, llms.txt proposal, сентябрь 2024 (llmstxt.org); Mintlify adoption, ноябрь 2024; hitlseo.ai guide «llms.txt vs llms-full.txt: The Complete 2025 Guide».
- **Проверка качества (E.21):** LLM (или keeper в роли AI) может ответить на 5 типичных вопросов о проекте, прочитав только llms.txt — без обращения к другим файлам.

### P-04: Компрессия промптов — archivist-роль keeper'а

- **Проблемная ситуация:** Промпты ролей, накапливая правки и контекст, раздуваются до 3-5 KB и тратят значительную часть контекстного окна. При этом часть содержимого — дублирование, устаревшие примеры или избыточные объяснения.
- **Силы:** Авторы промптов думают о полноте, а не о токен-экономии. Каждое обновление добавляет строки, редко удаляет. LLM обрабатывает длинный промпт хуже компактного при прочих равных.
- **Ход (решение):** Keeper периодически проводит archivist-сессию: (1) вычитать промпт на дублирования и «воду»; (2) сжать без потери семантики; (3) измерить сжатие в токенах до/после; (4) проверить, что ключевые инструкции сохранены.
- **Последствия:** Экономия токенов (типично 20-40% [hypothesis; пример из Core Team: facilitator 4.2 KB→2.1 KB, −50%]); более чёткое следование роли инструкциям; место в контекстном окне для рабочего содержимого.
- **Trade-offs:** Риск потери нюансов при слишком агрессивном сжатии; требует понимания модели (что «видит» LLM в промпте); нельзя сжимать промпты без проверки поведения после.
- **Пример:** Промпт facilitator'а в Core Team v4.1 вырос до 4.2 KB; после archivist-прохода v4.5 — 2.1 KB при сохранении всех ключевых протоколов.
- **Пограничные случаи:** Промпты с множеством примеров — примеры нельзя убирать бездумно; они часто несут больше смысла, чем абстрактные правила.
- **Антипаттерны:** «Добавляем, никогда не убираем» — промпт-bloat; «сжали, не проверили» — потеряли критическое правило; «сжимаем раз в полгода» — слишком редко для активно развивающегося проекта.
- **Связи:** стыкуется с [P-03] (llms.txt экономия); противоречит подходу «максимальный контекст» при дорогих моделях (см. §4).
- **Когда НЕ применять:** Промпты на этапе проектирования роли — не трогать до стабилизации поведения.
- **SoTA-источник:** Anthropic, «Prompt Engineering Overview», docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/overview, 2024-2025; Tiago Forte, «Building a Second Brain», 2022 — distillation-принцип [аналогия из PKM, не прямое исследование LLM-агентов; специализированных академических работ по AI-agent prompt compression мало — область emergent].
- **Проверка качества (E.21):** Количество токенов промпта уменьшилось ≥15%; роль ведёт себя идентично до/после (проверяется на 5 контрольных сценариях).

### P-05: Прагматичная таксономия — структура сверху вниз с ростом снизу

- **Проблемная ситуация:** Новый проект или знание нуждается в организации. Слишком жёсткая иерархия с самого начала не подходит, так как домен не изучен. Слишком свободная — превращается в мусорную свалку.
- **Силы:** Таксономия «на вырост» требует предвидения домена. Органическая классификация по тегам хаотична и теряет навигационность при масштабе. KM-стандарты требуют структуры, но гибкости.
- **Ход (решение):** Начать с минимальной плоской структуры (glossary / domain / decisions / features / roles). Добавлять уровни только когда конкретная категория содержит >7 элементов. Ревизировать таксономию раз в квартал.
- **Последствия:** Структура отражает реальный рост, а не гипотетический дизайн; навигация остаётся понятной; не тратится время на «правильную» классификацию несуществующих сущностей.
- **Trade-offs:** Таксономия требует миграции при реструктуризации. Без ревизии со временем теряет структуру.
- **Пример:** Stripe начинали документацию с плоских файлов по продуктам; с ростом до сотен API-ресурсов добавили иерархию и автогенерацию разделов — но только когда масштаб это потребовал (Mintlify, 2024).
- **Пограничные случаи:** Для research-проектов плоская структура работает дольше, чем для продуктовых.
- **Антипаттерны:** «Таксономия из 5 уровней до начала работы» — empty folders; «теги вместо структуры» — теги без владельца превращаются в синонимы (user/users/usr все означают одно); «один человек придумал таксономию» — не будет использоваться остальными.
- **Связи:** стыкуется с [P-01] (bounded context); противоречит enterprise top-down taxonomy (см. §4).
- **Когда НЕ применять:** Проекты с известным, стабильным доменом — можно начать со зрелой структуры.
- **SoTA-источник:** Digital Workplace Group, KM Best Practices 2025; Enterprise Knowledge, «Top KM Trends 2025» (taxonomies as AI foundations, enterprise-knowledge.com, 2025); Sönke Ahrens, «How to Take Smart Notes», 2017 — Zettelkasten как альтернатива иерархической таксономии: атомарные заметки + двусторонние ссылки вместо строгой иерархии.
- **Проверка качества (E.21):** Любой участник команды без объяснений находит нужный артефакт менее чем за 2 навигационных шага от корня репозитория.

### P-06: Кодификация tacit-знания через структурированные шаблоны

- **Проблемная ситуация:** Ключевое знание команды — в головах опытных участников, не в документах. При смене состава или переключении AI-агента контекст теряется.
- **Силы:** Tacit-знание трудно формализовать без потери нюансов. Попытка «записать всё» требует огромных усилий и быстро устаревает. Но без кодификации знание недоступно AI и новым участникам.
- **Ход (решение):** Использовать структурированные шаблоны для ключевых типов знания (handoff, session, role-context, ADR). Шаблон задаёт минимальный набор вопросов, вытаскивая знание порциями без ощущения «напиши документ». После каждой сессии — роль обновляет role-context.md.
- **Последствия:** AI-агент читает role-context.md и получает накопленную память роли. Знание не теряется при «смерти» сессии. Шаблоны снижают когнитивный барьер.
- **Trade-offs:** Шаблоны могут стать формальностью («всё равно никто не читает»); требуют enforcement (хук TaskCompleted в Core Team).
- **Пример:** Core Team v4.5 — hooks/verify-task.sh блокирует закрытие задачи, если роль не обновила role-context.md за последние 5 минут. Это механический enforcement tacit-knowledge capture.
- **Пограничные случаи:** Если шаблон слишком длинный — роль заполняет формально. Оптимальная длина role-context.md — 50-200 строк.
- **Антипаттерны:** «Заполним когда будет время» — никогда; «один большой wiki-документ» — нет структурных вопросов, нет нужных ответов; «tacit-знание важно, поэтому делаем видео» — видео не читает AI-агент.
- **Связи:** стыкуется с [P-07] (свежесть), [P-08] (governance); ISO 30401 §8.1.
- **Когда НЕ применять:** Для незначимых артефактов и одноразовых задач.
- **SoTA-источник:** ISO 30401:2018 §8.1 (Knowledge capture); Airbnb knowledge-repo (github.com/airbnb/knowledge-repo) — notebooks as structured knowledge; Enterprise Knowledge, «AI-Assisted Tacit Knowledge Capture», 2025.
- **Проверка качества (E.21):** AI-агент, не участвовавший в прошлой сессии, восстанавливает актуальный контекст за <3 минуты, читая только role-context.md.

### P-07: Свежесть знаний — ротация и TTL контента

- **Проблемная ситуация:** Документация создаётся, но не обновляется. Через 3-6 месяцев значительная часть фактов устарела. AI, опирающийся на устаревший контент, выдаёт неверные рекомендации.
- **Силы:** Обновление документации — невидимая работа без немедленного результата. Приоритет всегда у новых фич, не у «зелёной» документации. Знание decay — процесс постепенный, незаметный до катастрофы.
- **Ход (решение):** Назначать TTL (time-to-live) для разных типов документов: domain.md — ревизия ежеквартально; glossary.md — при каждом новом термине; decisions/ — append-only, не устаревают; llms.txt — синхронно с документацией. Использовать метаданные `last_modified` как сигнал опасности.
- **Последствия:** Документация активно поддерживается; AI-агенты получают актуальный контекст; знание decay становится видимым метрикой.
- **Trade-offs:** TTL-политика требует discipline и tooling; нереалистичный TTL хуже отсутствия — порождает фиктивные обновления «дата изменена, контент не трогался».
- **Пример:** Bloomfire (2024): content freshness metric — процент документов с ревизией за последние 90-180 дней. Fluidtopics: «стаое знание хуже отсутствия знания — ведёт к wrong decisions».
- **Пограничные случаи:** Evergreen-контент (принципы, определения терминов) устаревает медленнее; event-driven контент (результаты встреч, решения) требует более частой проверки.
- **Антипаттерны:** «Обновим по запросу» — запрос приходит после неверного решения; «last_modified = git commit date» не гарантирует, что содержимое проверялось; «ротация без проверки» — обновили метаданные, не проверили факты.
- **Связи:** стыкуется с [P-02] (факт vs решение), [P-03] (llms.txt); FPF DRR (Decision Relevance Rate).
- **Когда НЕ применять:** append-only артефакты (decisions/) — не нужна ротация, только хронология.
- **SoTA-источник:** Bloomfire, «Knowledge Management Metrics 2024» (bloomfire.com); Fluidtopics, «8 KM Metrics», 2024; ISO 30401:2024/Amd §8.2 (Knowledge maintenance).
- **Проверка качества (E.21):** Доля документов с `last_modified` ≤ 90 дней > 80%. Количество «устаревших ссылок» в llms.txt = 0.

### P-08: Governance знания — владелец + цикл ревизии

- **Проблемная ситуация:** Документация существует, но непонятно кто за неё отвечает. Ошибки накапливаются, обновления откладываются, конфликты между версиями не разрешаются.
- **Силы:** Коллективная ответственность = ничья ответственность. В командах с несколькими ролями каждый думает, что кто-то другой обновит. Без explicit owner документ «ничей».
- **Ход (решение):** Для каждого ключевого документа назначить явного владельца (explicit owner). В Core Team — keeper владеет glossary.md, domain.md, llms.txt. Decisions/ — append-only, владелец=команда. Строить automated enforcement (хуки, CI-проверки).
- **Последствия:** Ясная точка ответственности; ошибки исправляются; конфликты разрешаются через известный канал.
- **Trade-offs:** Один владелец = bottleneck при его недоступности; ownership нужно передавать при смене состава.
- **Пример:** Stripe — каждый engineer является владельцем документации к своему коду; это часть job ladder и performance review (Mintlify, 2024). В Core Team — хук TaskCompleted блокирует задачу без обновления role-context роли-автора.
- **Пограничные случаи:** В распределённых командах ownership должен быть backup-aware — что если владелец недоступен?
- **Антипаттерны:** «Все могут редактировать» без процесса ревью → wiki rot; «центральная команда документации» без domain-владельцев → документация за бизнесом не поспевает; «ownership без enforcement» — правило без гарда.
- **Связи:** стыкуется с [P-07] (свежесть), [P-06] (кодификация); Digital Workplace Group рекомендует «assign knowledge domain owners» как top practice.
- **Когда НЕ применять:** Ephemeral документы и черновики — без формального owner.
- **SoTA-источник:** Digital Workplace Group, KM Best Practices 2025; APQC, Certified Knowledge Manager curriculum; ISO 30401:2018 §7.1 (Leadership and roles); KMWorld Conference 2025 (kmworld.com) — knowledge ownership и governance как топ-тема года среди KM-практиков.
- **Проверка качества (E.21):** Для каждого документа в project/ существует explicit owner (роль или человек), зафиксированный в frontmatter или README.

### P-09: Стратегия кодификации vs персонализации — осознанный выбор

- **Проблемная ситуация:** Команда пытается «задокументировать всё» — и тонет в документах, которые никто не читает. Или полагается только на людей («спроси Ивана») — и теряет знание при смене состава.
- **Силы:** Codification и personalization — конкурирующие стратегии (Hansen et al., Harvard Business Review, 1999). Codification: знание → документы → переиспользование. Personalization: знание → люди → прямые связи. Обе работают, но на разных типах знания.
- **Ход (решение):** Explicit vs tacit диагностика: явное, стандартизируемое знание (правила, процессы, факты) → кодифицировать. Сложное, контекстно-зависимое знание (решения в неопределённости, экспертные суждения) → персонализировать (роли-subagents, живые ревью, парное мышление). Для AI-команды: кодификация = обязательна для всего, что читает AI-агент; персонализация = сессионные ревью, Decider Protocol.
- **Последствия:** Снижается перегруженность «ненужными» документами; повышается качество тех, что остаются; AI-агент получает именно то, что ему полезно.
- **Trade-offs:** Диагностика explicit/tacit требует времени и опыта; граница размыта; риск под-кодификации (теряем знание) и пере-кодификации (тратим время на бесполезные документы).
- **Пример:** ResearchGate, 2024: «personalization and codification aren't mutually exclusive — organizations use blends depending on innovation archetype (inbound/outbound/hybrid)».
- **Пограничные случаи:** В AI-assisted командах баланс смещается к кодификации — AI не может участвовать в personalization-каналах (живых разговорах).
- **Антипаттерны:** «Документируем всё подряд» — creation без curation; «у нас нет документации, мы agile» — полная персонализация без кодификации; «копируем KM-практику McKinsey» (codification-heavy) для стартапа из 5 человек — несоответствие масштабу.
- **Связи:** стыкуется с [P-06] (tacit capture), [P-08] (governance); противоречит [см. §4 Codification vs Personalization].
- **Когда НЕ применять:** Когда знание принципиально неотделимо от опыта носителя — переговоры о ценностях, кризисное принятие решений, живой творческий процесс: кодификация не даёт нужного качества, необходима живая персонализация. Также: в первые 2 недели нового проекта с нестабильной доменной моделью — преждевременная кодификация создаёт документы, которые придётся выбросить.
- **SoTA-источник:** Morten T. Hansen et al., «What's Your Strategy for Managing Knowledge», HBR, March 1999; David Weinberger, «Too Big to Know», 2012 — networked knowledge как аргумент в пользу personalization: знание не хранится в документах, а распределено между людьми и связями; ResearchGate, «Balancing Knowledge Strategy: Codification and Personalization», 2024; realkm.com, 2024.
- **Проверка качества (E.21):** Команда может назвать, что конкретно кодифицируется, а что передаётся через живые встречи — и обоснование различия.

### P-10: DDD Ubiquitous Language — синхронизация код-документация-разговор

- **Проблемная ситуация:** Код использует технические имена (CustomerEntity), документация — бизнес-термины (покупатель), а в разговорах звучит что-то третье (клиент, юзер). Команда говорит о разном, думая, что говорит об одном.
- **Силы:** Естественная эволюция языка в команде без явного управления. Разные роли (dev, product, customer) имеют разный словарь. Рефакторинг кода ломает связь имён с документацией.
- **Ход (решение):** Единый язык (ubiquitous language) — проектный глоссарий, где каждый термин имеет имя, которое используется в коде (EN), в документации (RU), и в разговоре. Keeper следит за консистентностью. При введении нового термина — сначала в glossary.md, потом в код.
- **Последствия:** Меньше переводов при переключении контекстов; баги из-за семантических расхождений уменьшаются; AI-агент понимает термины из кода и документации одинаково.
- **Trade-offs:** Требует дисциплины при именовании; противоречит практике «называй как хочешь в коде, главное логика».
- **Пример:** В DDD, Bounded Context «Billing»: EN-имя Invoice (код), RU-имя «Счёт» (документация), «счёт» (разговор) — все три синхронизированы. Если dev называет это «ReceiptDocument» в коде — keeper поднимает несоответствие.
- **Пограничные случаи:** Legacy-код с устаревшими именами — не нужно переименовывать всё сразу; можно маппинг в glossary.md (код: ReceiptDocument → домен: Invoice).
- **Антипаттерны:** «Код — для программистов, документация — для остальных» — разрыв нарастает; «переводить термины по запросу» — путаница множится; «глоссарий только для onboarding» — не обновляется.
- **Связи:** стыкуется с [P-01] (bounded context glossary), [P-05] (таксономия); Eric Evans DDD.
- **Когда НЕ применять:** Прототипы с throw-away кодом — не тратить время на синхронизацию имён.
- **SoTA-источник:** Eric Evans, Domain-Driven Design, 2003; Martin Fowler, «Ubiquitous Language», martinfowler.com; paulserban.eu, «Bounded Contexts and Ubiquitous Language», 2024.
- **Проверка качества (E.21):** Произвольно взятый термин из кода и тот же термин в документации — совпадают ≥90% случаев (проверяется grep + glossary lookup).

### P-11: AI-ready контент — подготовка знания для LLM-потребления

- **Проблемная ситуация:** Документация написана для людей: длинные введения, форматирование HTML/PDF, контекстные ссылки на «вышеупомянутое», пересылки между страницами. LLM не может эффективно работать с таким форматом.
- **Силы:** Большинство существующих документов created before AI era. LLM хорошо работает с flat-markdown, плохо — с вложенными HTML/CSS/JS и cross-reference конструкциями. «Hallucination» AI часто происходит от плохого контекста, а не от незнания.
- **Ход (решение):** При создании документации для AI-потребления: (1) plain markdown без HTML; (2) self-contained chunks (каждый раздел понятен без предыдущего); (3) явные определения (не «как описано выше»); (4) метаданные в frontmatter (дата, версия, owner); (5) структура ≤ 3 уровня вложенности.
- **Последствия:** AI-агент корректно интерпретирует документы; меньше галлюцинаций из-за контекстного шума; документация служит и людям, и AI.
- **Trade-offs:** Self-contained chunks дублируют информацию; избыточность vs сжатость — компромисс.
- **Пример:** Enterprise Knowledge (2025): «AI-Ready Content» — ключевой тренд; KM-специалисты отвечают за то, чтобы контент был accurate, current, и trustworthy для AI.
- **Пограничные случаи:** Интерактивная документация (Stripe Markdoc) — хороша для людей, но requires preprocessing для LLM.
- **Антипаттерны:** «Заливаем PDF в AI» — теряем структуру; «ссылаемся на внешние URL без snapshot» — контент меняется; «один гигантский документ вместо структурированного набора» — контекстное окно переполнено нерелевантным.
- **Связи:** стыкуется с [P-03] (llms.txt), [P-04] (prompt compression); Enterprise Knowledge KM Trends 2025.
- **Когда НЕ применять:** Документация исключительно для людей (например, design mockups).
- **SoTA-источник:** Enterprise Knowledge, «AI-Ready Content», 2025 (enterprise-knowledge.com); llmstxt.org standard spec, 2024; Jeremy Howard, «introducing llms.txt», Answer.AI, September 2024.
- **Проверка качества (E.21):** AI-агент, получив только один раздел документа (без соседних), отвечает на вопрос о нём корректно — без «not mentioned above» ошибок.

| **Documentation debt spiral** | Документация откладывается «пока» → накапливается → слишком много → снова откладывается | Экспоненциальный рост технического долга знания | [P-08] governance + [P-06] шаблоны |

| **Терминологический дрейф** | Одно понятие называется по-разному в коде, документации и разговорах | Семантические баги; AI путается; new team members переспрашивают | [P-10] ubiquitous language |
| **Stale llms.txt** | llms.txt не обновлялся 2+ месяца; ссылки ведут на удалённые/переименованные файлы | AI-агент тратит токены на dead links или получает устаревший контекст | [P-03] + [P-07] freshness |
| **Кодификация всего подряд** | Документируются очевидные вещи, нет документации для сложных | Шум перекрывает сигнал; AI выдаёт тривиальное | [P-09] осознанный codification/personalization blend |
| **«Спроси Ивана»** | Критическое знание в голове одного человека, нигде не записано | Bus factor = 1; AI-агент ничего не знает | [P-06] tacit capture шаблоны |
| **Глоссарий без «не является»** | Определения позитивные, без антидефиниции | Ложный консенсус: каждый читает своё | [P-01] антидефиниция обязательна |

## 4. Карта противоречий (между школами)

| Вопрос | Школа A | Школа B | Как выбирать в нашем контексте |
|--------|---------|---------|--------------------------------|
| **Codification vs Personalization** | KM должно создавать документы и базы знаний (McKinsey, consulting-heavy); знание = актив который можно переиспользовать | KM = connecting people; документы убивают нюанс; знание живёт в сетях людей (Nonaka & Takeuchi, японская школа) | Для AI-assisted команды — обязательна кодификация (AI читает документы, не людей). Personalization — для сложных решений через живые Decider Protocol сессии. Blend, не выбор. |
| **Top-down taxonomy vs folksonomy** | Таксономия спроектирована экспертами сверху; строгая иерархия; единая для организации (IA-school) | Теги и категории возникают органически из использования (web 2.0 / folksonomy, Del.icio.us era) | Гибридный подход [P-05]: minimal top-level structure + позволять органическому росту до 7 элементов, затем структурировать |
| **Объём документации: maximum vs minimum** | Больше документации = меньше зависимость от людей; документируй всё | Lean documentation: документируй только уникальное и нестабильное (Agile Manifesto: working software over comprehensive documentation) | В AI-контексте: документируй всё, что читает AI-агент. Остальное — по бизнес-ценности |
| **Structured templates vs free-form** | Шаблоны обеспечивают completeness и findability (ISO 30401, APQC) | Шаблоны душат живое знание; людей раздражают; не заполняются | Mandatory templates для критических артефактов (decisions, role-context, handoff) + free-form для рабочих заметок |
| **Centralized vs distributed ownership** | Один knowledge manager / keeper = единая точка качества (Stripe head of docs) | Ownership распределён: каждый отвечает за своё (Stripe engineers own their docs) | Ответ один — не противоречие: central keeper = quality standards и governance; distributed engineering ownership = content production |
| **Markdown vs rich content** | Plain markdown + git = versionable, AI-readable, diff-friendly | Rich content (Confluence, Notion) = user-friendly, visual, collaborative | Для AI-assisted context: markdown приоритетен. Rich tools для human-facing документов |

## 5. Guide применения — что / когда / как

| Сигнал ситуации | Применить паттерн(ы) | Не применять |
|-----------------|----------------------|--------------|
| Новый проект, домен не изучен | [P-05] минимальная плоская структура + [P-01] начать glossary | [P-08] сложный governance пока структура нестабильна |
| Новый термин введён командой | [P-01] + [P-10] → сначала в glossary.md, потом в код | Использовать в коде без добавления в glossary |
