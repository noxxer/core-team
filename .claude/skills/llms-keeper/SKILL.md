# llms-keeper — Project Context for AI Agents

Генерация и поддержка `llms.txt` и `llms-full.txt` по стандарту [llmstxt.org](https://llmstxt.org). Цель — дать AI-агентам быстрое понимание проекта при onboarding.

## Принцип

> **"Find the smallest set of high-signal tokens that maximize the likelihood of desired outcome."**

Каждое предложение должно отвечать на вопрос: «Без этой информации AI выдаст результат хуже?» Если нет — удаляем.

## Два файла, две цели

| Файл | Назначение | Аудитория | Длина |
|------|------------|-----------|-------|
| `llms.txt` | Навигационный индекс | AI, нужна быстрая ориентация | 50-100 строк |
| `llms-full.txt` | Полный контекст | AI, нужно глубокое понимание | 200-600 строк |

## Signal vs Noise

### Включать (high-signal)

- Цель проекта + tech-стек с версиями
- Архитектура: иерархия модулей, направление зависимостей
- Data flow (ASCII-диаграммы для сложных flow)
- Паттерны, используемые в 3+ файлах
- Configuration approach + entry points
- Common pitfalls с решениями
- Essential development commands
- Testing approach + conventions

### НЕ включать (noise)

- Step-by-step туториалы / implementation guides
- Temporary workarounds, TODOs, FIXMEs
- Паттерны, используемые в 1 файле
- Verbose объяснения («Этот модуль отвечает за...»)
- Очевидное для опытных разработчиков
- Credentials, API keys, secrets

## Workflow

Skill делегирует основную работу keeper-subagent-у. Тот выполняет:

1. **Scan** — структура проекта + tech-стек по manifest-файлам (`pyproject.toml`, `package.json`, `Cargo.toml`, etc.)
2. **Read** — entry points, core модули, configs, tests, docs (README, CLAUDE.md)
3. **Extract** — high-signal information (architecture, patterns, pitfalls)
4. **Generate** `llms-full.txt` первым (self-contained content)
5. **Derive** `llms.txt` из него (navigation index с ссылками на секции)
6. **Verify** quality checklist (см. `references/llmstxt-spec.md`)

## Update vs Generate

- **Нет `llms.txt`** → полная генерация с нуля
- **Есть `llms.txt`** → инкрементальное обновление:
  - Парсим footer `*Generated by ... on YYYY-MM-DD, based on commit \`HASH\` (...)*`
  - `git log --oneline HASH..HEAD` → видим изменения с момента последней генерации
  - Обновляем только затронутые секции, не переписываем неизменное
  - Обновляем footer с актуальным hash

## Footer (обязательный, anchor для инкрементальных апдейтов)

В конце `llms-full.txt`:

```
---
*Generated by keeper on YYYY-MM-DD, based on commit `SHORT_HASH` (COMMIT_MESSAGE)*
```

Без footer-а инкрементальные обновления невозможны — это compass для следующего запуска.

## PII / secret stripping

Перед записью `llms.txt`/`llms-full.txt` маскируй:
- Emails: `<email>`
- Длинные токены (40+ chars `[A-Za-z0-9_\-]`): `<token>`
- User-home paths: `/Users/<user>/...`
