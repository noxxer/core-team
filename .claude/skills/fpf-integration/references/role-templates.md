# FPF Role Templates — Checklist by Archetype

> Шаблоны FPF-чеклистов по архетипам ролей.
> Адаптировать под конкретный проект: заменить примеры на доменные.

## Принцип построения чеклиста

1. Определить домен роли (что решает, что пишет)
2. Выбрать топ 3-5 FPF-принципов (не все, а критичные для домена)
3. Определить топ 2-3 когнитивных искажения (специфичных для роли)
4. Составить 5-7 пунктов: конкретных, не абстрактных
5. Записать в context.md роли

## Универсальные принципы (для всех ролей)

- **A.7 Strict Distinction** — описание != способность != факт
- **A.10 Evidence Graph** — утверждение без доказательства = мнение
- **NQD** — >= 3 альтернативы (для Complicated/Complex)

---

## Архетип: Аналитик / Исследователь

**Домен:** данные, метрики, выводы, рекомендации.
**Критичные принципы:** A.10, A.7, A.11, NQD, MVPK
**Опасные искажения:** Overconfidence, Illusion of Explanatory Depth, False Consensus

**Чеклист (перед публикацией вывода или метрики):**
- [ ] Каждое утверждение размечено: факт (источник + метод + дата) / гипотеза / мнение? (A.10)
- [ ] Метрика реально измерима в текущих условиях или требует инфраструктуры? (A.7)
- [ ] "Слепая зона" — точно отсутствие данных, а не данные в неформальном виде? (A.10)
- [ ] Порог метрики обоснован ссылкой на стандарт, а не интуитивно? (A.10)
- [ ] Рекомендация предложена с >= 3 вариантами реализации? (NQD)
- [ ] Не дублирует существующий инструмент? Если дополняет — разница описана? (A.11)

---

## Архетип: Проектировщик / Архитектор

**Домен:** структура, процессы, форматы, стандарты.
**Критичные принципы:** A.7, A.10, A.11, NQD
**Опасные искажения:** Confirmation Bias, Survivorship Bias, Planning Fallacy

**Чеклист (перед публикацией процесса или рекомендации):**
- [ ] Рекомендация улучшает реализацию или меняет идею? Если идею — обсуждено с Owner? (A.7)
- [ ] "Процесс работает так" — из документа или из наблюдения? (A.7)
- [ ] Gap назван критическим — есть данные о вреде (метрики, жалобы)? (A.10)
- [ ] >= 3 альтернативных подходов рассмотрены? (NQD)
- [ ] Каждая новая сущность — нельзя ли выразить через существующее? (A.11)
- [ ] Допущения отделены от выводов? (A.10)
- [ ] Изменения подкреплены расчётом, а не ощущением? (A.7 + A.10)

---

## Архетип: Критик / Аудитор

**Домен:** проверка, fairness, риски, bias detection.
**Критичные принципы:** A.10, A.7, A.2
**Опасные искажения:** Bias Blind Spot, Overconfidence, Anchoring

**Чеклист (перед публикацией аудитного вывода):**
- [ ] Каждый риск с обоснованием: почему HIGH, а не MEDIUM? Данные или экспертная оценка? (A.10)
- [ ] Аудит проведён независимо от выводов других ролей? (A.2)
- [ ] Обнаруженный bias — артефакт процесса или теоретическая возможность? (A.7)
- [ ] Рекомендации реалистичны? Изменения чужого домена обсуждены с владельцем? (A.2)
- [ ] Реестр рисков дополняет, а не дублирует другие артефакты? (A.11)
- [ ] Правовые рекомендации ссылаются на конкретный закон/стандарт? (A.10)

---

## Архетип: Голос стейкхолдера

**Домен:** потребности группы (кандидатов, команд, клиентов).
**Критичные принципы:** A.7, A.10, A.1.1
**Опасные искажения:** Framing Effect, Availability Heuristic, False Consensus

**Чеклист (перед публикацией pain point или рекомендации):**
- [ ] Pain point на реальных данных или логически выведен? Если выведен — пометить как гипотезу (A.10)
- [ ] "Группа страдает от X" — справедливо для всех или для сегмента? Сегмент указан? (A.1.1)
- [ ] Рекомендация проверена на стоимость для процесса? (A.11)
- [ ] Критика основана на наблюдении или на умозаключении из отсутствия артефакта? (A.7)
- [ ] Граничные случаи описаны как отдельные контексты? (A.1.1)

---

## Архетип: Координатор / Фасилитатор

**Домен:** повестка, синтез, решения, handoff.
**Критичные принципы:** A.1.1, A.2, A.7, A.10, NQD
**Опасные искажения:** Groupthink, Authority Bias, Hindsight Bias

**Чеклист (перед публикацией ADR, handoff, повестки):**
- [ ] Каждый голос роли зафиксирован? Есть место, где роль НЕ согласилась? (A.2)
- [ ] Owner получает только нерешаемое ролями? Не подменяю работу ролей? (A.2)
- [ ] Все утверждения в handoff размечены: факт / мнение / гипотеза? (A.10)
- [ ] Для каждого решения >= 2 отклонённые альтернативы с причинами? (NQD)
- [ ] Термины совпадают с глоссарием? (A.1.1)
- [ ] Ledger отражает текущее, а не желаемое состояние? (A.7)

---

## Архетип: Автоматизатор / Builder

**Домен:** инструменты, автоматизация, инфраструктура.
**Критичные принципы:** A.11, NQD, A.7
**Опасные искажения:** Availability Heuristic, Planning Fallacy, Inverted Status Quo Bias

**Чеклист (перед публикацией решения по автоматизации):**
- [ ] Процесс стабилен или ещё меняется? Автоматизировать нестабильное = tech debt (A.7)
- [ ] Автоматизация устраняет проблему или переносит? (A.11)
- [ ] >= 3 варианта с разной сложностью (ручной, полуавтомат, полный)? (NQD)
- [ ] Оценка разделяет: разработка, внедрение, поддержка? (A.7)
- [ ] Существующие инструменты точно не покрывают потребность? (A.11)
- [ ] За предложением — измеримая боль, а не гипотеза? (A.10)

---

## Архетип: Эксперт / Tech Lead

**Домен:** стандарты, грейды, технические требования.
**Критичные принципы:** A.7, A.10, A.11, NQD
**Опасные искажения:** Dunning-Kruger (экспертное направление), Training Data Bias, Overconfidence

**Чеклист (перед публикацией стандарта или оценки):**
- [ ] "Хороший ответ" — ДОЛЖЕН ответить (описание), МОЖЕТ (способность), или ОТВЕТИЛИ реальные люди (факт)? (A.7)
- [ ] Якорные примеры из реальных данных или сконструированы? Если сконструированы — гипотеза? (A.10)
- [ ] Критерии одинаковы для разных контекстов или обобщение из одного? (A.1.1)
- [ ] Рекомендация на данных о различающей способности или экспертном суждении? (A.10)
- [ ] >= 3 варианта замены перед объявлением "X плохой"? (NQD)
- [ ] Новые требования — нельзя ли выразить через существующее? (A.11)
