# FPF Audit Patterns — Retrofit Guide

> Паттерны ретроспективного ревью решений через FPF-линзу.
> Основано на реальном опыте интеграции FPF в Interview Team Framework (2026-03-31).

## Методика экспресс-ревью решения

Для каждого решения ответить на 6 вопросов. Время: 10-15 мин на решение.

### 1. Evidence Chain (A.10)

**Как проверять:**
- Прочитать каждое утверждение в обосновании
- Для каждого определить: факт (есть источник + метод + дата) / гипотеза (логически выведено) / мнение (экспертное суждение)
- Если утверждение подаётся как факт без источника → пометить как gap

**Оценка:**
- **Strong:** >= 80% утверждений подкреплены источником, гипотезы явно помечены
- **Partial:** 40-80% подкреплены, некоторые гипотезы выдаются за факты
- **Weak:** < 40% подкреплены, систематическое смешение фактов и мнений

**Типичные нарушения:**
- "30-60 X в месяц" — без указания источника и даты
- "Все роли согласились" — AI-агенты = одна модель, не независимые мнения
- "Уже работает" — design-time description выдаётся за run-time fact

### 2. Alternatives (NQD)

**Как проверять:**
- Найти секцию "Альтернативы" / "Варианты"
- Посчитать количество
- Проверить diversity: реально различаются или "то же, но чуть иначе"?
- Проверить trade-offs: каждая альтернатива с анализом плюсов/минусов?

**Cynefin-адаптация:**
- Clear domain → 0-1 допустимо
- Complicated → >= 3 обязательно
- Complex → >= 3 + safe-to-fail эксперименты

**Типичные нарушения:**
- 0 альтернатив: решение принято без рассмотрения вариантов
- 1-2 альтернативы отклонены одним аргументом без trade-off анализа
- "Альтернатива" = параллельный процесс, а не реальный вариант

### 3. Cognitive Biases

**Как проверять:**
- Кто предложил решение? (Authority bias если Owner)
- Были ли разногласия? (Groupthink если нет)
- Решение реактивное (на ошибку) или проактивное? (Availability bias)
- Первый вариант принят? (Einstellung)
- Консенсус AI-агентов? (Echo chamber)

**Частые биасы в мультиагентных системах:**

| Bias | Как проявляется | Маркер в тексте |
|------|-----------------|-----------------|
| AI Echo Chamber | "Все 5 ролей согласились" | Консенсус агентов на одной модели |
| Solution Salience | Конкретный артефакт > абстрактное улучшение | "Scorecard" побеждает "культуру fairness" |
| Anchoring на метрике | Улучшение с 0.087 → 0.241 = "успех" | Относительное улучшение маскирует абсолютно слабый результат |
| Precision Illusion | 3 знака после запятой при N=38 | Видимость точности без доверительных интервалов |
| Not Invented Here | "Собственная шкала, независимая от..." | Отказ от существующих систем без анализа |
| Authority Override | "Owner + Expert единодушны" → переопределяют консенсус | 2 человека vs медиана 9 |

### 4. Decay / Validity

**Как определять evidence_valid_until:**

| Тип evidence | Срок | Пример |
|-------------|------|--------|
| Метрика на данных | 3-6 мес после сбора | Kappa = 0.241 на 38 ответах |
| Экспертный консенсус | 6 мес или при ротации > 30% | 11 оценщиков калибровали шкалу |
| Технический факт | Бессрочно или до смены формата | Правило парсинга Excel |
| Стратегическое решение | 3 мес или до первых результатов | Фокус на scorecard |

**Как определять review_date:**
- = evidence_valid_until для большинства решений
- Раньше если есть зависимый эксперимент (blind test → ревью через 2 недели после)

### 5. Kill Criteria

**Хороший kill criteria:**
- Конкретный (метрика, порог, выборка)
- Имеет протокол проверки (кто, когда, каким методом)
- Outcome, не output ("Kappa >= 0.4" а не "есть прототип")

**Плохой kill criteria:**
- "Если не работает" — что значит "не работает"?
- Только delivery-метрика ("есть прототип через 6 недель") без outcome
- Без протокола проверки (декоративный)

### 6. Description vs Fact (A.7)

**Как проверять:**
- Найти утверждения о поведении системы/процесса
- Для каждого: это из документа (description), из наблюдения (fact), или из головы (hypothesis)?
- "L-профиль уточняется между этапами" — описание желаемого или наблюдённый факт?

**Маркер:** если есть открытый вопрос по тому же утверждению → это гипотеза, не факт.

## Case Study: Interview Team Framework (5 решений)

### Системные паттерны

| Паттерн | Частота | Серьёзность |
|---------|---------|-------------|
| NQD-нарушение (< 3 альтернатив) | 4 из 5 | Высокая |
| Отсутствие decay (review_date) | 5 из 5 | Средняя |
| AI-консенсус как evidence | 1 из 5 | Высокая |
| Description = fact (A.7) | 2 из 5 | Высокая |
| Позитивный тренд quality | от DEC-001 к DEC-005 | Позитив |

### Позитивный тренд

Evidence quality системно росла от первого решения к последнему:
- DEC-001 (partial): мнения + нецитированные данные
- DEC-005 (strong): измеримые метрики, конкретная выборка, цитаты

Это означает: команда учится строить обоснования. Закрепить как норму — требовать DEC-005 уровень quality для всех новых решений.

### Kill criteria: delivery vs outcome

Ранние решения (DEC-001) имели delivery kill criteria ("есть прототип"). Поздние (DEC-005) — outcome ("Kappa >= threshold"). Рекомендация: всегда дополнять delivery-критерии outcome-метрикой.
