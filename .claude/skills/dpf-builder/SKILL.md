# DPF Builder — постройка предметных фреймворков принципов

Метод превращения разрозненного материала о домене/ремесле в **DPF**: набор FPF-паттернов
(ситуация → силы → ход → последствия → антипаттерны → связи → проверка), на которые AI-агент
может опираться как на внешнее знание.

## Где DPF в экосистеме (FPPS → FPF → DPF → LPF)

| Слой | Что | В Core Team |
|------|-----|-------------|
| **FPPS** | онтологика/семиотика/эпистемология | ядро FPF (внешняя спека) |
| **FPF** | общие принципы работы в проектах | `.claude/knowledge/fpf/` (lite-индекс) + skill `fpf-integration` |
| **DPF** | предметные SoTA-ходы, термины, ошибки, границы | **этот skill** → `knowledge/dpf/<role>.md` и `project/dpf/<domain>.md` |
| **LPF** | локальные регламенты конкретного проекта | `project/values.md`, `infrastructure-principles.md`, role-context |

DPF — **недостающий средний слой**: между общим FPF и локальными артефактами проекта.

## Два вида DPF (разные bounded contexts)

- **role-DPF** — ремесло роли, переиспользуемо между проектами. Живёт в `.claude/knowledge/dpf/<role>.md`,
  едет с фреймворком, коммитится в репозиторий фреймворка.
- **domain-DPF** — предметная область конкретного проекта. Живёт в `project/dpf/<domain>.md`,
  генерится у потребителя при setup, коммитится в **его** репо.

## Метод: Collect → FPF-process → Loop-improve

### Стадия 1 — Collect (сбор)

Полный обход источников по **Collection Protocol** (11 каналов): концепции · топ-3 книги ·
лучшие статьи · SoTA-инструменты · **плейбуки топ-5 компаний** · стандарты/сертификации ·
постмортемы · именованные эксперты · сообщества · конкурирующие школы · метрики качества.

> Детали и спец-директивы (явный источник = выжать досуха; узкая область; провенанс): `references/collection-protocol.md`.

Выход — структурированный дайджест с источниками, расхождениями и замеченными ошибками.

### Стадия 2 — FPF-process (упаковка в паттерны)

Превратить дайджест в DPF по шаблону `.claude/templates/dpf-template.md`:

1. **Глоссарий** (§1) — термины домена (A.1.1: RU/EN/определение/«не является»).
2. **Паттерны** (§2, формат E.8) — на каждую повторяющуюся проблемную ситуацию: силы, ход,
   последствия, trade-offs, пример, пограничные случаи, **антипаттерны**, связи, «когда НЕ», источник, проверка.
3. **Каталог антипаттернов** (§3) — сквозные «красивые промахи» из постмортемов (канал 7).
4. **Карта противоречий** (§4) — где школы расходятся (каналы 9-10) и как выбирать.
5. **Guide применения** (§5) — что/когда/как; что НЕ применять.

**Провенанс-гейт (v0.2 — критично).** Каждое число/процент/«статистика X%» в теле паттерна обязано иметь
**воспроизводимый первичный источник**. Правило приоритета (выучено на refine-проходе v5.0):

1. Нашёл первоисточник реально → ставь его (+ добавь в frontmatter `sources`).
2. Не нашёл → пометь прямо в строке `[hypothesis]` (эвристика практиков) или `[vendor-claim]` (цифра вендора).
3. Если число не несёт нагрузки → **убери его**.

> **Запрещено выдумывать цитату.** «Добавь источник» легко вырождается в «придумай правдоподобный источник»
> (фабрикация blog/года/URL) — это **хуже** честной метки `[hypothesis]`. Лучше тег или удаление, чем ложная атрибуция.

Тезис без источника и без метки в паттерн **не входит** — идёт в §6 «открытые пробелы».

### Стадия 3 — Loop-improve (доведение, E.21)

Первый DPF — **всегда черновик** (seed/draft). Поднимаем maturity циклами:

1. Оценить паттерны по характеристикам (полнота полей, наличие антипаттернов, провенанс, узость scope).
2. Улучшить слабые: добрать источники (по провенанс-гейту), добавить границы, контрпримеры, связи.
3. Переоценить **независимым проверяющим**.
4. Зафиксировать открытые пробелы в §6 Improvement Log.

**Verify ставит maturity, не автор (v0.2).** На refine-проходе v5.0 авторы-фиксеры завышали `maturity: refined`
при незакрытом провенансе. Поэтому:

- **Проверяющий — другая/сильнее модель + адверсариальная стойка** (refuter): его цель — *найти* пропущенное,
  по умолчанию считать claim голословным. Однотиповый self-verify не годится (см. P-13 facilitation-DPF:
  единогласие одной модели ≠ правота). В workflow verify-стадия идёт на более сильной модели.
- **maturity выставляет verify**, по итогу проверки — автор лишь *предлагает*. Завышение `refined` при открытых
  провенанс-дырах = откат в `draft`.

Уровни: `seed` (знания модели) → `draft` (web-research пройден) → `refined` (≥8 каналов + независимый verify, провенанс-дыр нет) → `reliable` (противоречия разрешены, у всех паттернов метрика проверки, ключевые числа с первоисточником).

## Контракт вывода

- role-DPF → `.claude/knowledge/dpf/<role>.md` (frontmatter `type: role`).
- domain-DPF → `project/dpf/<domain>.md` (frontmatter `type: domain`).
- Структура — строго по `.claude/templates/dpf-template.md`.
- Frontmatter заполнен: `maturity`, `sources_count`, `based_on_fpf`, `sources`.

## Как роль потребляет свой DPF

Роль в активационном ритуале читает `.claude/knowledge/dpf/<role>.md` (если есть) — **после**
project-context и **до** работы. DPF не заменяет промпт роли: промпт = контракт роли,
DPF = предметная опора (A.7 Strict Distinction).

## Anti-patterns метода

| Anti-pattern | Симптом | Лечение |
|--------------|---------|---------|
| «Книга рецептов» | только проблема→решение, нет антипаттернов/связей | заполняй §3 и §4, это половина ценности |
| «Учебник обо всём» | широкий scope, общие слова | сузить §0 до повторяющейся задачи |
| «Один источник» | весь DPF из одной книги/сайта | минимум 8 каналов для refined |
| «Seed как готовый» | maturity не помечен, выдан за reliable | честный `maturity:` + §6 пробелы |
| **«Выдуманная цитата»** | под давлением «добавь источник» — фабрикация blog/года/URL или ложная атрибуция | провенанс-гейт: тег `[hypothesis]`/`[vendor-claim]` или удаление, НЕ выдумка |
| **«Самозваный refined»** | автор сам поднял maturity при открытых провенанс-дырах | maturity ставит независимый verify, не автор |
| «Self-verify» | проверка той же моделью одним проходом | адверсариальный verify сильнее/другой моделью (P-13) |
| «DPF вместо промпта» | паттерны диктуют пошаговые действия | DPF декларативен, промпт роли — контракт |

## Изменения v0.2 (уроки сборки v5.0)

Выучено на генерации + refine 11 role-DPF:

1. **Провенанс-гейт против фабрикации** — «добавь источник» провоцирует выдуманную цитату; приоритет тегу/удалению.
2. **maturity ставит независимый verify, не автор** — авторы-фиксеры завышали `refined` при открытом провенансе.
3. **Verify — сильнее/другая модель + адверсариально** — однотиповый self-verify пропускает завышение и галлюцинации (P-13).
4. **AI-specific канал** обязателен для ролей внутри AI-фреймворка (пробел guardian).
5. **Топ-источники постранично**, не только вторичные выжимки.
6. Шаблон: `channels_covered` (каналы) ≠ `sources_count` (записи sources).
