# Collection Protocol — сбор материала для DPF

> Вход стадии **Collect** метода `dpf-builder`. Цель — не «нагуглить статью», а собрать
> достаточно разнообразного материала, чтобы из него можно было сложить паттерны,
> **антипаттерны**, связи и противоречия (это требует FPF-process, стадия 2).

## Принцип охвата

DPF силён ровно настолько, насколько разнообразны источники. Один тип источника →
плоский «учебник из общих слов». Поэтому собираем по **11 каналам** (минимум 8 для maturity≥refined).

| # | Канал | Что вытаскиваем | Зачем для DPF |
|---|-------|-----------------|---------------|
| 1 | **Ключевые концепции домена** | базовые различения, словарь | §1 глоссарий, каркас паттернов |
| 2 | **Топ-3 канонические книги** | принципы и основы, выжимка | ядро паттернов P-* |
| 3 | **Лучшие методологические статьи** | методы, нюансы, edge cases | поля «силы», «пограничные случаи» |
| 4 | **Современные инструменты + SoTA** | что реально используют практики сегодня | «ход», актуальность |
| 5 | **Плейбуки топ-5 компаний мира** | как ремесло поставлено у лидеров (handbooks, eng/PM/design-гайды, open culture-доки) | проверенные на масштабе ходы |
| 6 | **Стандарты и сертификации** (IAF, PMI, IIBA, ISTQB, ISO, OWASP…) | кодифицированная рамка компетенций | полнота охвата, авторитет |
| 7 | **Литература провалов / постмортемы** | что и как ломается | §3 каталог антипаттернов |
| 8 | **Именованные эксперты-практики** | сигнатурные фреймворки конкретных людей | глубина сверх «учебника» |
| 9 | **Сообщества практики + конф-доклады** | живые споры, что оспаривается | §4 карта противоречий |
| 10 | **Смежные / конкурирующие школы** | альтернативные подходы к той же задаче | §4 противоречия, NQD-разнообразие |
| 11 | **Метрики качества ремесла** | как домен измеряет «хорошо» | поле «проверка качества» (E.21) |

## Спец-директивы

- **Явный источник = выжать досуха.** Если Founder указал конкретный сайт/книгу/репозиторий
  как источник — он приоритетен: обойти **целиком**, не ограничиваться первой страницей выдачи.
  Для сайта: пройти карту страниц (sitemap / навигация / все разделы), а не одну landing.
- **Узкая область сильнее широкой.** «Учебник обо всём» даёт общий результат. Если запрос широкий —
  сузить до повторяющейся задачи и зафиксировать сужение в §0.
- **Всегда полный web-research.** Дефолт проекта — реальный обход сети (каналы 1-11), не seed.
  Degraded-режим (только знания модели → maturity:seed) — лишь когда web физически недоступен;
  явно пометить `maturity: seed` и перечислить недобранные каналы в §6.
- **Провенанс обязателен.** Каждый собранный тезис несёт источник + дату (FPF A.10 Evidence Graph).
  Claim без источника на стадии 2 не становится паттерном — идёт в «открытые пробелы».
- **Без фабрикации (v0.2).** Не нашёл первоисточник для цифры — честная метка `[hypothesis]`/`[vendor-claim]`
  или удаление. Выдуманная цитата хуже отсутствия (см. провенанс-гейт в SKILL.md).
- **AI-specific канал.** Если DPF строится для роли/домена внутри AI-агентного фреймворка — обязателен
  отдельный сбор по AI-рискам и SoTA: prompt injection, data poisoning, agent misuse, eval/guardrails,
  многоагентные режимы отказа (напр. MAST). Без него получаются слепые зоны (урок: пробел guardian-DPF v5.0).
- **Топ-источники постранично.** Для 2-3 ключевых книг/спек — проход по содержанию, не только вторичные
  выжимки: нюанс паттерна часто живёт в главе, а пересказ его теряет (урок: Kaner/McCarthy в facilitation-DPF).

## Выход стадии Collect

Структурированный дайджест (не финальный DPF): по каждому каналу — ключевые тезисы с источниками,
явные расхождения между источниками, замеченные типовые ошибки. Этот дайджест — вход FPF-process.
