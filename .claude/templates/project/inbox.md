# Входящие (Intake) — сырой поток от Founder

> **Capture-first.** Founder вкидывает мысли, идеи, решения, вопросы и задачи вперемешку.
> Фасилитатор сначала **фиксирует сырой вход как есть** (ничего не теряется, не интерпретируется),
> потом классифицирует и направляет по протоколам. Разбор без фиксации запрещён.

## Правила триажа (тип → куда направить)

| Тип | Сигнал | Направление |
|-----|--------|-------------|
| **task** | «надо сделать X», конкретное действие | `TaskCreate` |
| **tension** | «что-то не так», конфликт, проблема | 3-стадийный протокол (TaskCreate → роли → Founder) |
| **decision** | стратегический выбор, развилка | Decider Protocol → `decisions/DEC-NNN.md` |
| **question** | неясность, нужен ответ | Clarity Protocol / `AskUserQuestion` |
| **idea** | гипотеза, «а что если», на потом | `backlog.md` |
| **fact** | утверждение о реальности | `domain.md` (с источником, A.10) |
| **term** | новое понятие | `glossary.md` (RU/EN/определение/«не является») |

Один сырой вход может расщепиться на несколько строк разных типов — это норма.

## Очередь (необработанное)

| # | Дата | Сырой вход (дословно) | Тип(ы) | Статус |
|---|------|------------------------|--------|--------|
|   |      |                        |        | new    |

Статус: `new` → `triaged` → `routed` → `done`.

## Обработанное (архив триажа)

| # | Дата | Краткое | Тип | Куда ушло (Task/DEC/backlog/domain…) |
|---|------|---------|-----|--------------------------------------|
|   |      |         |     |                                      |

## Инвариант

В конце сессии очередь «необработанного» должна быть пуста или каждый элемент имеет явное
«осознанно отложено» с причиной. Потерянный вход = процессная ошибка фасилитатора.
