# End Session — Закрытие сессии

Аргумент `$ARGUMENTS` — тема сессии (если пусто, определи по контексту).

## Шаг 1: Сбор контекста
1. `project/ledger.md` → последний handoff из `project/sessions/` → `TaskList` → определи тему

## Шаг 2: Обобщение выводов (Navigator)

### 2a. Iceberg Model — для каждого значимого результата пройди 4 уровня:
Событие (что произошло) → Паттерн (повторяется?) → Структура (что порождает) → Ментальная модель (убеждение за этим)
В session.md выноси **паттерны и структуры** — они полезны для следующих сессий.

### 2b. Three "Why?" — для каждого DEC-NNN:
Почему это решение? Почему важно для проекта? Почему сейчас? Нет чёткого ответа → OQ-NNN.

### 2c. System Operator — для ключевых решений:
Матрица 3x3: (Подсистема/Система/Надсистема) x (Было/Стало/Будет). Пустые ячейки = слепые зоны.

### 2d. Metacognition Checklist
- Выводы на правильном уровне? Альтернативы (>=3) для сложных решений?
- Проверка вверх (видение) и вниз (реализуемость)? Нет tunneling/premature closure/goal displacement?

### 2e. Итоговая фиксация
Решения (DEC-NNN + "почему"), обновлённые артефакты, ключевые выводы (паттерны/структуры), слепые зоны, новые OQ-NNN.

### 2f. Конвертация находок в задачи — ОБЯЗАТЕЛЬНО
**Каждая** слепая зона, неотвеченный "почему", паттерн без решения → `TaskCreate` (указать источник, контекст, роли). Если после 2e остались находки без задач — протокол НЕ ЗАВЕРШЁН.

## Шаг 3: Ревизия задач
Пройди `TaskList`: завершённые → `TaskUpdate` completed | давно неактуальные → предложи удалить | можно объединить → предложи | блокированные → уточни блокер. Изменения через `AskUserQuestion`.

## Шаг 4: Trap Scan + опыт

### 4a. Ловушки сессии:
| Ловушка | Признак |
|---------|---------|
| Einstellung | Первый вариант без альтернатив |
| Tunneling | Долго без прогресса |
| Bike-shedding | Час на мелочь, минута на ключевое |
| Goal displacement | Процесс стал целью |
| Premature closure | Закрыли слишком быстро |
| Dead-level abstracting | Застряли на одном уровне |

Если были — зафиксируй: что, как заметили, как избежать.

### 4b. Паттерны работы, поведение ролей, проблемы процесса, новые конвенции → обнови auto-memory.

## Шаг 5: Глоссарий/домен (опционально)
Новые термины → `project/glossary.md`. Новые факты → `project/domain.md`. Устаревшие тезисы → обновить.

## Шаг 5a: Code-vs-Ledger Reconciliation (git-verified)

> **Лёгкая сверка — ВСЕГДА.** Перед записью любого статуса «done» в ledger сверь его с git: `git log --since="<session-start>" --name-only` и (если есть remote) `git branch -r --merged`. Ledger пишется **по факту кода/коммитов, а не по памяти** — иначе «done»-заявки расходятся с реальностью и переживают сессии.
>
> **Полная reconciliation (architect discovery) — по триггеру `[STUB]`:** в сессии обнаружен маркер `[STUB]` / `# STUB:` / `# TODO:` / «вернёмся позже» в коде, в `roles/<role>/context.md`, или в session.md / handoff. Нет триггера — ограничься лёгкой сверкой выше.

Запуск `Agent(subagent_type=architect)` в discovery-режиме:

1. Пройди по последним коммитам сессии (`git log --since="<session-start>" --name-only`)
2. Grep по `STUB|TODO|FIXME|placeholder` в изменённых файлах
3. Сравни с `project/ledger.md` — есть ли расхождение между «что записано» и «что в коде»?
4. Обнови `ledger.md` **по факту кода**, не по плану
5. Для каждого `[STUB]` маркера → создать `TaskCreate` с приоритетом для следующей сессии (что именно недостроено + где)

**Reason:** после stub-фаз ledger расходится с реальностью (G.2 staleness post-stub). Без явной reconciliation в следующих сессиях architect/dev исходят из устаревшей картины и плодят расхождения.

## Шаг 5b: Обновление контекстов ролей

Для каждой роли, участвовавшей в сессии:
1. Прочитай `project/roles/<role>/context.md`
2. Обнови: текущий фокус, рабочие знания, заметки для будущих сессий
3. Добавь новые решения (DEC-NNN), артефакты, открытые вопросы по домену роли
4. Обнови `last_updated` в YAML-заголовке
Не трогай контексты ролей, которые НЕ участвовали в сессии.

## Шаг 6: Ревью промптов
Проверь `.claude/CLAUDE.md` и `.claude/commands/` на актуальность. SAFE-изменения — сразу, остальные — через `AskUserQuestion`.

## Шаг 6a: Гейт решений (pending-DEC) — ОБЯЗАТЕЛЬНО

> Проблема: решения «зависают» в статусе pending или живут только сжатой строкой в ledger без файла — обоснование теряется, заблокированная работа копится.

1. **DEC-NNN ⟹ файл.** Любой `DEC-NNN`, упомянутый в `ledger.md`, **обязан** иметь `project/decisions/DEC-NNN.md`. Нет файла → создать сейчас (по `adr-template.md`) или убрать упоминание. Сжатая строка ledger ≠ решение.
2. **Pending-решения.** Каждое решение в статусе `Proposed`/pending в этой сессии → либо ратифицировать в файл, либо явный **deferral с owner + deadline** (`TaskCreate`). Без owner+deadline pending-решение оставлять запрещено.
3. **Same-session DEC-propagation.** Если ратифицированный DEC меняет рабочий процесс (правила ролей, инварианты, конвенции) — в **этой же сессии** обнови: (1) `output-styles/` или `CLAUDE.md`, (2) TOP-строку `ledger.md`, (3) напоминание в `hooks/session-start.sh`, (4) чеклист затронутой роли. Иначе DEC «протухает» за 2-3 сессии.
4. **publish-with-task.** Если сессия обновила публичный артефакт с будущими обязательствами (обещания, награды, дедлайны вовне) → авто-`TaskCreate` с owner+deadline. Обещание без owner = забытый долг.

## Шаг 7: Запись артефактов
Директория: `project/sessions/YYYY-MM-DD_kebab-topic/`
1. `session.md` — по формату `.claude/templates/project/session-template.md`
2. `handoff.md` — по формату `.claude/templates/project/handoff-template.md`
3. Обнови `project/sessions/index.yaml`
4. Обнови `project/ledger.md` (last_updated, фаза, задачи, артефакты, DEC)

## Шаг 8: Git sync
1. `git add -A` → `git status` → проверь `git diff --cached` (нет секретов, бинарных файлов)
2. `git commit -m "session: {session_id}"`
3. Если remote настроен: `git pull --rebase origin main` → конфликт → СТОП, показать файлы. Нет конфликтов → `git push`
