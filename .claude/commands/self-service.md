# Self-Service — Самообслуживание системы

Мета-команда для поддержания здоровья и развития системы промптов.

## НЕПРИКОСНОВЕННЫЕ ФАЙЛЫ ФРЕЙМВОРКА

**НИКОГДА не модифицировать, не удалять, не перемещать:**
- `.claude/templates/**` — шаблоны фреймворка
- `.claude/skills/**` — навыки и их references
- `.claude/knowledge/**` — база знаний (FPF, протоколы, биасы, безопасность). **Исключение:** `.claude/knowledge/dpf/**` — генерируемый DPF-слой (имплант добавляет role-DPF, setup чистит неактивные)
- `.claude/output-styles/**` — стили вывода
- `.claude/hooks/**` — quality gates

**Только добавление новых файлов (implant):**
- `.claude/commands/<new-role>.md` — создание новых ролей

**Только Founder может изменить:**
- `project/values.md` — конституционный файл
- `project/decisions/**` — решения (append-only, не редактировать принятые)

## Режимы работы

Спроси основателя какой режим нужен:

1. **audit** — аудит связности системы
2. **implant** — имплантация новой роли/навыка
3. **compress** — архивация проектных данных
4. **optimize** — оптимизация рабочих процессов
5. **full** — полный цикл (audit → рекомендации)

## 1. Audit — Аудит связности

### Построение графа

Построй граф системы:

**Узлы:**
- Команды (`.claude/commands/*.md`)
- Роли (активные в CLAUDE.md)
- Skills (`.claude/skills/*/SKILL.md`)
- Артефакты (`project/artifacts/`, `project/decisions/`, etc.)
- Knowledge-файлы (`.claude/knowledge/`)

**Рёбра:**
- `вызывает` — команда → роль, команда → команда
- `читает` — роль → артефакт, роль → knowledge
- `пишет` — роль → артефакт (File Ownership)
- `зависит от` — skill → reference file
- `рекомендует` — facilitator → команда

### Проверки связности

| Проверка | Что ищем | Критичность |
|----------|----------|-------------|
| Достижимость | Каждый узел достижим из `/facilitator` | Высокая |
| Тупики | Роли без вызывающих сценариев | Высокая |
| Контекстная поверхность | Базовые файлы содержат достаточные саммари внешних документов | Высокая |
| Осиротевшие артефакты | Файлы без владельца в File Ownership | Средняя |
| Разрывы потока | Данные генерируются, но никем не читаются | Средняя |
| Дублирование | Одинаковые инструкции в разных промптах | Низкая |
| Противоречия | Конфликтующие инструкции между ролями | Высокая |
| Размер промптов | Файлы > 100 строк (команды) или > 300 строк (skills) | Низкая |
| Индексы | Новые артефакты доступны через индексы для facilitator | Средняя |

### Проверка контекстной поверхности

> **Принцип:** Claude Code автоматически загружает только CLAUDE.md и команды при вызове.
> Ссылки на файлы (`> Полная версия: ...`) — это текст, а не автозагрузка.
> Если важный аспект представлен только ссылкой без саммари, он будет проигнорирован
> в большинстве сессий. Лучше чаще загружать в контекст, чем пропустить когда важно.

**Базовые файлы** (загружаются всегда):
- `.claude/CLAUDE.md`
- Глобальный `~/.claude/CLAUDE.md`

**Файлы по вызову** (загружаются при активации команды/роли):
- `.claude/commands/*.md`

**Внешние документы** (НЕ загружаются автоматически):
- `.claude/skills/*/SKILL.md` и их `references/`
- `.claude/knowledge/*.md`

#### Алгоритм проверки

Для каждого внешнего документа, на который ссылается базовый файл:

1. **Найди ссылку** в CLAUDE.md или команде (паттерн `> ... .claude/...`)
2. **Оцени саммари** рядом со ссылкой:
   - Есть выжимка ключевых правил (3-7 пунктов)? → OK
   - Есть только ссылка без контента? → **ПРОБЛЕМА**
   - Саммари устарело (не соответствует оригиналу)? → **ПРОБЛЕМА**
3. **Определи критичность** документа:
   - Влияет на каждую сессию (принципы, безопасность, искажения) → саммари обязательно
   - Влияет ситуативно (FPF, протоколы) → ссылка + 1-2 строки контекста достаточно
   - Справочный (шаблоны, guide) → только ссылка OK

#### Матрица покрытия (заполняется при аудите)

| Внешний документ | Ссылается из | Критичность | Саммари | Статус |
|-----------------|--------------|-------------|---------|--------|
| functional-clarity/SKILL.md | CLAUDE.md | Каждая сессия | Выжимка 7 принципов | ? |
| security-rules.md | CLAUDE.md | Каждая сессия | 5 правил | ? |
| cognitive-biases.md | CLAUDE.md | Каждая сессия | Антибайас-протокол | ? |
| core-protocols.md | CLAUDE.md | Ситуативно | 3 протокола (названия) | ? |
| navigator/SKILL.md | CLAUDE.md | Ситуативно | 3 триггера активации | ? |
| tdd-master/SKILL.md | CLAUDE.md | Ситуативно | 2 правила | ? |
| fpf/glossary.md | CLAUDE.md | Справочный | Глоссарий FPF-терминов | ? |
| guide-multi-agent-teams.md | setup-project | Справочный | Только ссылка | ? |

#### Правила саммари

- **Каждая сессия**: 3-7 пунктов ключевых правил, достаточных чтобы применять без чтения оригинала
- **Ситуативно**: 1-3 строки контекста — когда активировать и зачем, достаточных чтобы принять решение о загрузке
- **Справочный**: ссылка + одна строка назначения
- Саммари не заменяет оригинал — это триггер для загрузки и минимальная защита от игнорирования

### Формат отчёта

```markdown
## Аудит системы: YYYY-MM-DD

### Граф
- Узлов: N (команд: X, ролей: Y, skills: Z, артефактов: W)
- Рёбер: N
- Связность: полная / есть разрывы

### Проблемы
| # | Тип | Описание | Рекомендация |
|---|-----|----------|-------------|

### Рекомендации
- [конкретные действия]

### Метрики здоровья
- Покрытие ролями: X% доменов
- Средний размер промпта: N строк
- Дублирование: N% инструкций
```

## 2. Implant — Имплантация роли/навыка

### Процесс создания роли

#### Фаза 1: Исследование (глубокое) — через `dpf-builder`

Имплант роли = постройка её **role-DPF**. Активируй skill `dpf-builder` и пройди стадию Collect
по **Collection Protocol** (11 каналов): концепции · топ-3 книги · лучшие статьи · SoTA-инструменты ·
**плейбуки топ-5 компаний** · стандарты/сертификации · постмортемы (антипаттерны) · именованные
эксперты · сообщества · конкурирующие школы · метрики качества. Дефолт — полный web-research.

Опорные вопросы при сборе:
1. **Лучшие промпты/гайды** для этой роли в открытых источниках
2. **Статьи лидеров индустрии** с рекомендациями по этой работе
3. **Канонические книги** — ключевые идеи каждой
4. **Что отличает отличного специалиста от среднего** (SoTA-ходы)
5. **Антипаттерны** — типичные ошибки роли (из постмортемов)

#### Фаза 2: Синтез

1. Из собранного материала выделить:
   - Миссия (1-2 предложения)
   - Домены (3-5 областей)
   - Ключевые функции (5-8)
   - Фреймворки и инструменты
   - Принципы поведения
   - Антипаттерны (что НЕ делать)
2. Сверить с принципами Функциональной Ясности
3. Определить tensions с существующими ролями

#### Фаза 3: Имплантация (с защитой целостности)

**Обязательные проверки перед добавлением:**

1. **Непротиворечивость** — новая роль не конфликтует с существующими
2. **Полнота потоков** — входы роли обеспечены, выходы востребованы
3. **File Ownership** — файлы роли не пересекаются с другими
4. **Достижимость** — роль вызывается из конкретных сценариев
5. **Необходимость** — без роли есть слепая зона (tensions test)

**Артефакты имплантации:**
- `.claude/agents/<role>.md` (subagent с frontmatter: name/model/color/description/tools, ≤ 200 строк)
- `.claude/knowledge/dpf/<role>.md` (role-DPF — стадия 2-3 `dpf-builder`: FPF-process + Loop-improve)
- `project/roles/<role>/context.md` (начальный контекст — память между сессиями)
- Обновлённая таблица ролей в CLAUDE.md + строка в `.claude/knowledge/dpf/README.md`
- Обновлённый File Ownership
- Обновлённый `.claude/planner-context.md` §1 (каталог subagent-ов)
- Активационный ритуал роли читает `.claude/knowledge/dpf/<role>.md`

#### Фаза 4: Верификация

Запусти audit после имплантации — убедись, что система целостна.

### Имплантация навыка (skill)

Аналогично, но создаётся:
- `.claude/skills/<skill>/SKILL.md`
- `.claude/skills/<skill>/references/` (справочные материалы)
- Ссылка в CLAUDE.md

## 3. Compress — Архивация проектных данных

> **ВНИМАНИЕ**: Операции сжатия необратимы. Выполнять с максимальной осторожностью.
> **ОГРАНИЧЕНИЕ**: Compress работает ТОЛЬКО с `project/` файлами. Файлы `.claude/` неприкосновенны.

### Принципы

1. **Нулевая потеря смысла** — сжимать синтаксис, не семантику
2. **Только стабильное** — сжимать паттерны, используемые 3+ раз без изменений
3. **Предварительная проверка** — показать diff основателю ДО применения
4. **Версионирование** — git commit перед каждым сжатием

### Что сжимать (только `project/`)

- `project/roles/<role>/context.md` — устаревшие заметки, дублирование
- `project/ledger.md` — завершённые задачи (архивировать в sessions/)

### Что НЕ сжимать

- Конституционные файлы (`project/values.md`)
- Решения (`project/decisions/`)
- Активные планы и задачи
- **Всё в `.claude/`** — шаблоны, skills, knowledge, commands, hooks

### Лимиты размеров (ориентиры)

| Файл | Максимум |
|------|----------|
| CLAUDE.md | 200 строк |
| Command-файл роли | 100 строк |
| SKILL.md | 300 строк |
| ledger.md | 150 строк |
| context.md роли | 200 строк |

## 4. Optimize — Оптимизация процессов

### Автоматизация рутины

1. **Выявление паттернов**: какие операции повторяются и требуют подтверждения?
2. **Предложение allowlist**: добавить частые безопасные операции в `settings.local.json`
3. **Запоминание решений**: если основатель 3+ раз подтверждал одно и то же — предложить автоматизацию

### Рекомендации команд

На основе анализа сессий:
- Какие команды фреймворка не используются, но могли бы помочь?
- Какие ручные действия можно заменить командой?
- Где навигатор мог бы помочь, но не был активирован?

### Мониторинг здоровья промптов

- Промпты не разрастаются за пределы лимитов?
- Нет ли "мёртвого кода" в инструкциях?
- Все ссылки на файлы актуальны?

## Общие принципы самообслуживания

1. **Целостность системы** — каждое изменение проверяется на совместимость
2. **Минимальное вмешательство** — не чини то, что работает
3. **Прозрачность** — все изменения объясняются и показываются основателю
4. **Обратимость** — git commit перед любыми изменениями
5. **Инкрементальность** — малые шаги, проверка после каждого
6. **Защита фреймворка** — `.claude/` файлы неприкосновенны (кроме implant новых)
